# MoleOption 项目总纲

## 1. 项目一句话定义

MoleOption 是部署在 Solana 上、使用 Rust/Anchor 开发的链上衍生品协议，目标是提供“像永续合约一样简单、像期权一样有限亏损、无传统强平、无资金费率、基于全局配对清算”的杠杆交易产品。

## 2. 项目愿景

MoleOption 试图重构链上杠杆交易的基本范式：

- 用户体验保持永续合约式的直观：选择标的、方向、杠杆和保证金即可开仓。
- 风险暴露保持期权式的上限：单个仓位最大亏损永远不超过其开仓本金。
- 清算逻辑去中心化且透明：盈利方收益来自同场亏损方可兑现损失，协议不承诺无限流动性。
- 资金流动由智能合约托管和规则执行，避免传统做市商、订单簿滑点、资金费率和强平插针。

## 3. 核心产品主张

### 3.1 永不爆仓

协议不触发传统意义上的强平，不会因为价格穿越某个爆仓线而删除用户仓位。用户仓位可以进入权益归零或低权益状态，但仓位记录仍然存在。

### 3.2 本金风险封顶

每个仓位的最大可损失金额为开仓本金。合约层必须强制满足：

```text
0 <= locked_loss <= principal
withdrawable_amount >= 0
user_total_loss <= principal
```

### 3.3 盈利受真实对手盘约束

盈利并非由协议凭空铸造，而是来自同一隔离市场中亏损仓位可兑现损失。若亏损方不足，盈利方按各自账面盈利金额占总账面盈利的比例获得可兑现收益。

在最小两人场景中，实际转移金额严格等于：

```text
min(亏损方可兑现亏损, 盈利方账面盈利)
```

在多人场景中，系统先计算全局实际转移金额，再让盈利方按各自账面盈利金额占比分享，让亏损方按各自可兑现亏损金额占比承担。

该分配没有开仓时间优先级，也没有历史亏损优先级。只要仓位在当前价格区间产生正向盈利或恢复 claim，就按 claim 权重参与本轮分配。

### 3.4 杠杆交易场隔离

不同标的、不同抵押资产、不同杠杆倍数是不同市场。5x、10x、20x、50x、100x 不共享清算池，以避免低杠杆用户为高杠杆用户承担错配风险。

## 4. 项目边界

### 4.1 首个真实资金版本范围

首个真实资金版本可以限制市场数量和资金规模，但智能合约架构必须一步到位支持亿级仓位的清算模型：

- 单标的：例如 BTC/USD。
- 单抵押资产：USDC SPL Token。
- 单杠杆倍数：建议 10x 或 20x。
- 单预言机来源：Pyth。
- 指令：初始化市场、开仓、触发 epoch、分片结算、最终化 epoch、申请平仓、领取资金、暂停市场。
- 前端：基础交易页、仓位页、风险提示、当前权益和参考盈亏展示。
- 合约：从第一版就包含 SettlementEpoch、SettlementShard、Position lazy settlement、CloseRequest 和可验证聚合账户。

### 4.2 首版可延后范围

以下功能可以不进入首个真实资金版本，但不能影响核心合约未来扩展：

- 多抵押资产。
- 跨保证金。
- 用户间仓位转让。
- 复杂治理代币经济。
- 自动做市商。
- 高级组合期权。
- 移动端原生 App。
- 多预言机仲裁与社区申诉。

这些能力应在核心数学模型被链上验证后逐步加入。

## 5. 设计原则

### 5.1 资金守恒优先

任何时候协议可支付权益不得超过资金池真实余额。相比“显示盈利”，系统应优先维护“可兑现盈利”的真实性。

### 5.2 明确区分显示层与结算层

前端必须同时展示：

- 参考盈亏：类似永续合约，便于用户理解方向性表现。
- 当前权益：链上真实可提取金额或估算值。
- 盈利兑现率：当前市场亏损池对盈利池的覆盖程度。

不得把参考盈亏包装成确定可提取收益。

### 5.3 参数保守启动

初期应限制市场规模、杠杆上限、单仓规模、总未平仓名义价值、预言机延迟和价格跳变阈值。

### 5.4 全链上关键状态

影响用户权益的变量必须在链上可验证，包括市场重置价格、仓位本金、锁定损失、权益积分或累计分配索引。

### 5.5 可测试性优先于抽象优雅

清算引擎必须能被离线仿真复现。每次链上状态变更应有明确的数学公式、事件日志和测试样例对应。

## 6. 目标用户

### 6.1 加密高杠杆交易者

需要高资本效率，但厌恶插针爆仓、资金费率和强平机制。

### 6.2 中低杠杆趋势交易者

希望使用 5x-20x 方向性杠杆，同时把最大亏损锁定为本金。

### 6.3 DeFi 原生用户

偏好透明、可验证、非托管的链上交易体验。

### 6.4 策略开发者和量化团队

关注可预测规则、低交易成本、零滑点预言机成交和可索引链上状态。

## 7. 核心成功指标

### 7.1 协议安全指标

- 无资金池资不抵债。
- 无用户亏损超过本金。
- 无非法提款或重复领取。
- 重置算法在极端价格下仍保持不变量。

### 7.2 产品指标

- 首次开仓流程小于 3 步。
- 用户能清楚区分参考盈亏与当前权益。
- 用户能在开仓前看到最大亏损和盈利兑现风险。

### 7.3 市场指标

- 目标市场中多空资金分布相对均衡。
- 盈利兑现率在高流动性市场中长期接近 100%。
- 仓位开平和重置成本可被普通用户接受。

## 8. 风险总览

### 8.1 数学模型风险

MoleOption 的核心模型必须明确区分两个概念：

- `locked_loss` 不可逆：已经支付给其他盈利退出者的资金，不能因为后续价格反转而追回或冲回。
- 权益可恢复：仓位保留方向性敞口，后续价格反转且出现新的亏损方时，可以通过新增 `realized_profit_balance` 恢复部分或全部历史亏损。

因此，首个真实资金版本必须实现“锁定亏损不可逆 + 后续流动性恢复权益”的完整机制。工程上不能把权益恢复放到后续版本，也不能在 `locked_loss == principal` 时删除仓位的方向性经济权利。

同时，当仓位累计锁定亏损达到 100% 本金时，必须记录归零价格锚点。价格继续向不利方向移动时，该仓位价值保持 0，不再为市场提供亏损赔付，也不参与当前统计；只有价格回到归零锚点后，仓位才重新进入统计范围并按权重参与可分配利润。

### 8.2 预言机风险

高杠杆对价格错误极其敏感。首版应先使用 Pyth，并设置价格新鲜度、置信区间、价格跳变和市场暂停逻辑。

### 8.3 流动性风险

协议不保证任何时刻账面盈利都能全额兑现。该风险必须成为产品展示的一等公民。

### 8.4 监管风险

MoleOption 是衍生品协议，可能触及不同司法辖区的杠杆、差价合约、期权或赌博监管边界。上线前需法律评估。

### 8.5 高并发与可扩展性风险

白皮书中的“全局重置”是数学语义，不代表工程上每次开平仓都可以遍历所有用户。若市场达到百万级、亿级仓位，单笔 Solana 交易无法读取和写入所有仓位账户。

正式架构必须采用：

- settlement epoch。
- 市场分片。
- 分片批处理统计。
- epoch 最终化。
- 仓位懒结算。
- 链下索引辅助展示，链上校验最终权益。

没有这套可扩展清算架构，协议只能作为小规模实验，不能支撑 1 亿用户高并发。

## 9. 推荐研发路线

1. 数学模型定稿与仿真。
2. Solana/Anchor 生产级可扩展合约。
3. 本地测试和属性测试。
4. Devnet 前端闭环。
5. 经济攻击仿真和外部审计。
6. 限额主网上线。
7. 多市场和治理扩展。

## 10. 已锁定关键决策

- 是否采用 Anchor 作为合约框架。
- 重置是否由开平仓强制触发，还是允许 keeper 定期触发。
- 如何设计 settlement epoch、Shard 和仓位懒结算，确保亿级仓位下仍保持全局比例清算语义。
- 当前区块即时确定平仓金额是链上产品硬要求，因此正式采用 `方向权益池 + shares + recovery/dormant bucket` 作为大规模链上实时结算主模型。
- 白皮书中的逐仓比例清算是基础原则表达；shares 模型是该原则在亿级用户、当前区块即时提款约束下的可扩展实现。
- 首个市场选择 BTC/USD、ETH/USD 还是 SOL/USD。
