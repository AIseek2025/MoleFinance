# MoleOption 智能合约设计

## 1. 合约目标

MoleOption 智能合约负责托管用户资金并执行清算规则。链上程序必须保证：

- 用户不能损失超过仓位本金。
- 协议不能支付超过 Vault 余额的资金。
- 开仓、重置、平仓的状态转换确定且可验证。
- 管理员无法挪用用户资金。

## 2. 推荐框架

首个真实资金版本推荐：

- Solana Program。
- Rust。
- Anchor。
- SPL Token。
- Pyth 价格账户。

说明：可以通过离线原型和测试网验证，但真实资金智能合约不能设计成临时简化版本。账户结构从第一版就必须包含 epoch、shard、close request 和懒结算能力。

### 4.2 Market

生产 Market 字段以 shares 模型为主，逐仓字段（`reset_price / total_notional_long/short / total_principal`）只在事件中保留，Market 账户本身不再持有这些字段。详见 `18-shares模型实现细则与边界条件.md` §2。

### 4.3 Position

生产合约的 Position 以 shares 模型为唯一资金路径来源。`locked_loss` 和 `realized_profit_balance` 不进入链上 Position 字段，仅作为索引服务事件回放后的展示与审计字段。详见 `18-shares模型实现细则与边界条件.md`。

字段：

```rust
pub struct Position {
    pub market: Pubkey,
    pub sub_pool: Pubkey,
    pub owner: Pubkey,
    pub position_id: u64,

    pub direction: Direction,
    pub status: PositionStatus,

    pub principal: u64,
    pub leverage_bps: u32,
    pub notional: u128,

    // shares 模型核心字段
    pub active_shares: u128,
    pub recovery_shares: u128,
    pub recovery_bucket: Option<Pubkey>,
    pub zero_price: u64,
    pub zero_price_tick: i64,
    pub dormant: bool,

    pub entry_price: u64,
    pub last_sync_slot: u64,

    pub opened_at: i64,
    pub updated_at: i64,
    pub closed_at: i64,
    pub schema_version: u16,
    pub bump: u8,
}
```

枚举：

```rust
pub enum Direction {
    Long,
    Short,
}

pub enum PositionStatus {
    Open,
    CloseRequested,
    Closed,
}
```

说明：

- 不设置 `Depleted` 状态。即使 `pool_equity * active_shares / total_active_shares + recovery_value == 0`，仓位仍保持 `Open`，因为它通过 `recovery_shares` 和 dormant_bucket 保留方向性恢复资格。
- 仓位是否当前可提款为 0，由 shares 与池估值推导，而不是通过终止状态表达。
- `dormant = true` 表示仓位的 active_shares 已迁移至 recovery_shares 并注册到 dormant_bucket。
- `last_sync_slot` 用于资金敏感判断；`opened_at / updated_at` 为 unix_timestamp 仅展示。

### 4.4 DormantBucket

生产合约采用 shares 模型，必须设置零值锚点账户：

```rust
pub struct DormantBucket {
    pub sub_pool: Pubkey,
    pub direction: Direction,
    pub zero_price_tick: i64,
    pub total_recovery_shares: u128,
    pub total_recovery_notional: u128,
    pub position_count: u64,
    pub bump: u8,
}
```

每个 SubPool 各方向维护一棵 bucket 树（Fenwick / Segment tree），支持 O(log N) 的 `prefix_sum_up_to_tick` 与 `suffix_sum_from_tick`，用于价格回到锚点后聚合激活。

仓位归零时记录在 `Position`：

```rust
pub zero_price: u64,
pub zero_price_tick: i64,
pub dormant: bool,
```

当价格未回到 `zero_price_tick` 时，dormant 仓位不参与当前可赔付亏损或盈利 claim；当价格回到锚点后，仓位的 `recovery_shares` 在新 transfer 出现时按 claim weight 比例分享分配。

详见 `18-shares模型实现细则与边界条件.md` §6。

## 5. 指令设计

生产合约指令以 shares 模型为核心。`open_epoch / settle_shard_batch / finalize_epoch / apply_position_epochs` 仅作为离线 oracle 重建/审计指令，资金路径上不依赖它们。

### 5.0 sync_pool

任何会改变状态的指令在主体逻辑前必须先调用 `sync_pool`：

```text
sync_pool(market, sub_pool, oracle_account)
```

此外可作为公开指令被任何账户调用，让长期空闲的子池保持新鲜度。Keeper 必须保证每个子池在 `max_idle_slots` 内被同步至少一次。

实现见 `18-shares模型实现细则与边界条件.md` §4。

### 5.1 initialize_global_config

用途：

- 初始化全局配置。

权限：

- 部署者或指定 authority。

输入：

```text
authority
emergency_authority
fee_authority
fee_vault
```

### 5.2 initialize_market

用途：

- 创建独立杠杆交易场。
- 创建或绑定 Vault。
- 设置预言机和风控参数。

权限：

- GlobalConfig authority。

关键校验：

- 杠杆在允许范围内。
- 手续费不超过上限。
- 预言机 feed 与 symbol 匹配。
- 初始 reset_price 来自有效预言机。

### 5.3 open_position

用途：

- 用户开新仓。

账户：

- owner signer。
- market。
- sub_pool（按 PDA 路由 `hash(market || owner) % sub_pool_count`）。
- position PDA（`seeds = ["position", market, owner, owner_nonce]`）。
- user collateral ATA。
- market vault。
- fee vault。
- token program。
- oracle account（必须为最新 Pyth update）。

输入：

```text
amount: u64
expected_price_min: u64
expected_price_max: u64
expected_price_max_age_slots: u64
owner_nonce: u64
```

流程：

1. 校验市场未暂停且未冻结新开仓。
2. 校验金额、市场容量和预言机有效性。
3. 调用 `sync_pool(sub_pool, oracle_price)`，并校验价格保护。
4. 计算手续费 (`mul_div_ceil`) 与净本金。
5. 从用户 ATA 转入 Vault 和 fee_vault。
6. 计算 `shares_minted`，处理 `dilution_safety` 与 `pool_equity == 0` 边界。
7. 校验 `shares_minted > 0`，否则失败 `SharesMintedTooSmall`。
8. 更新 sub_pool（`<direction>_pool_equity`、`<direction>_active_shares`、`<direction>_notional`、dust）。
9. 创建 Position 持有 `active_shares = shares_minted`。
10. 检查不变量。
11. emit `PositionOpened`。

详见 `18-shares模型实现细则与边界条件.md` §9。

### 5.4 close_position

用途：

- 用户在当前区块即时平仓并领取资金。

账户：

- owner signer。
- market、sub_pool、position。
- vault、fee_vault（可选，若收平仓费）。
- user collateral ATA。
- oracle account。
- 用户对应 dormant_bucket（如果 position.recovery_shares > 0）。

流程：

1. 校验 owner、状态、市场未暂停。
2. 调用 `sync_pool` 并校验价格保护。
3. 计算 active 部分价值 `pool_equity * active_shares / total_active_shares`。
4. 取回 recovery 部分（如果 dormant 已激活并积累了可领取价值）。
5. 校验 `vault_balance >= withdrawable`。
6. 更新 sub_pool / dormant_bucket / position 状态（先写状态后转账）。
7. Vault 转账给用户。
8. 检查不变量。
9. emit `PositionClosed`。

如果 `withdrawable == 0`，必须拒绝执行普通 close，要求用户调用 `force_close_zero_value_position` 并显式 `acknowledge_forfeit_recovery = true`。

### 5.5 force_close_zero_value_position

用途：

- 关闭当前权益为 0 的 dormant 或 active 0 仓位，用户主动放弃恢复权。

输入：

```text
acknowledge_forfeit_recovery: bool   // 必须为 true
```

流程：

1. 校验 owner。
2. 校验 `withdrawable == 0`。
3. 减去 dormant_bucket 中相应 recovery_shares 与 notional。
4. 标记 position closed。
5. emit `PositionForfeitClosed`。

### 5.6 claim_dormant_recovery

用途：

- dormant bucket 被激活并产生新 transfer 后，用户主动领取所累积的可恢复价值，但不平仓。

流程：

1. 调用 `sync_pool` 并校验价格保护。
2. 计算当前 bucket 中分配给本仓位的可领取价值。
3. 减去对应 recovery_shares 与 bucket 累计金额。
4. 转账给用户。
5. emit `RecoveryClaimed`。

仓位继续持有剩余 recovery_shares。

### 5.7 sync_pool（公开）

任何账户可调用，强制让目标子池同步到最新 oracle 价格。Keeper 网络必须周期性调用。

### 5.8 harvest_dust

用途：

- 由治理或 fee_authority 调用，把 sub_pool 中累积的 dust 转入 fee_vault 或保险基金账户。

约束：

- 不允许转入用户钱包。
- 不允许转入 vault。

### 5.9 离线 oracle 辅助指令（非资金路径）

以下指令仅在审计或离线 oracle 模式下被填充，资金路径不依赖：

```text
open_epoch
settle_shard_batch
finalize_epoch
apply_position_epochs
```

它们用于让链上能够保留逐仓比例语义的可验证审计轨迹，便于与离线模拟器对账。生产合约可以选择把这些指令合并到独立的 audit program，避免与主合约共用账户。

### 5.10 pause_market / resume_market

用途：

- 应急风控（Level 0）。

权限：

- pause: emergency_authority 或 authority。
- resume: emergency_authority 与 global_config_authority 双签。

约束：

- pause 可立即执行；最大持续 `max_emergency_pause_slots`，超时自动解除。
- resume 必须先校验预言机连续 `recovery_check_window` slot 健康，且不变量自动检查通过。
- pause 不允许直接转移用户资金。
- 详见 `16-合约升级与治理紧急响应.md`。

### 5.11 update_market_params

用途：

- 调整限额、手续费、预言机阈值（Level 1 / Level 2 时间锁）。

权限：

- authority（必须为多签）。
- 经过 `propose -> queue -> wait timelock -> execute` 流程。

约束：

- 不允许修改会破坏现有仓位权益的参数。
- 杠杆、抵押资产、symbol 不可变。
- 不允许直接修改 `pool_equity`、`active_shares`、`recovery_shares` 数值。

## 6. 内部数学函数

### 6.1 calculate_delta_pnl

```rust
pub fn calculate_delta_pnl(
    direction: Direction,
    notional: u128,
    old_price: u64,
    new_price: u64,
) -> Result<i128>
```

公式：

```text
direction * notional * (new_price - old_price) / old_price
```

### 6.2 mul_div_floor

```rust
pub fn mul_div_floor(a: u128, b: u128, denominator: u128) -> Result<u128>
```

必须使用 checked math。

### 6.3 compute_withdrawable

```rust
pub fn compute_withdrawable(position: &Position) -> Result<u64>
```

公式：

```text
principal - locked_loss + realized_profit_balance
```

### 6.4 proportional_settlement

```rust
pub fn proportional_settlement(
    profit_increment: u128,
    total_profit: u128,
    actual_transfer: u128,
) -> Result<u128>
```

盈利方实际收益必须按账面盈利比例计算：

```text
realized_profit = actual_transfer * profit_increment / total_profit
```

亏损方实际锁定损失必须按可兑现亏损比例计算：

```text
realized_loss = actual_transfer * realizable_loss / total_realizable_loss
```

仓位后续再次成为亏损方时，应先消耗未提取盈利余额，再增加不可逆锁定亏损：

```text
profit_credit_spent = min(realized_profit_balance, realized_loss)
locked_loss_increase = realized_loss - profit_credit_spent

realized_profit_balance -= profit_credit_spent
locked_loss += locked_loss_increase
```

两人场景下，该函数结果必须等价于：

```text
min(single_profit_increment, single_realizable_loss)
```

## 8. 权限与安全约束

### 8.1 PDA 控制 Vault 与 fee_vault

Vault authority 必须是 Market PDA。fee_vault 是独立 PDA，独立 authority。任何指令都不能把 Vault 余额转入 fee_vault，反之亦然。

### 8.2 仓位所有权

平仓必须要求：

```text
position.owner == signer
position.market == market.key()
position.sub_pool == sub_pool.key()
```

### 8.3 防重复提款

平仓前：

```text
status != Closed
```

平仓必须先更新状态再 token 转账。任何 CPI 调用必须遵循 Solana 程序无重入原则；如果未来引入回调（hook）必须独立审计。

### 8.4 市场账户绑定

所有 Position 必须通过 seeds 绑定 Market、owner、owner_nonce，防止伪造仓位账户。SubPool seeds 包含 market 和 sub_pool_id。

### 8.5 PDA 抢占防护

Position PDA seeds 使用 `["position", market, owner, owner_nonce]`，由用户提交 owner_nonce。owner_nonce 不存在重叠时拒绝。这避免基于全局 `next_position_id` 递增的抢占型 DoS。

### 8.6 Oracle 程序 ID 校验

任何接收 oracle account 的指令必须验证 `oracle_account.owner == PYTH_PROGRAM_ID`，防止伪造 oracle program 注入恶意价格。

## 11. 生产级限制声明

真实资金合约不得使用“传入全部仓位账户进行重置”的设计。任何需要全市场遍历才能完成的链上指令都不允许进入生产版本。直接遍历只能存在于离线模拟器、测试 oracle 或审计复现工具中。

## 12. 合约验收标准

- 开仓正确扣款并创建仓位。
- 平仓不能提取超过可得权益。
- 仓位 locked_loss 不超过 principal。
- Vault 余额永远不小于所有可提款权益总和。
- 预言机异常时拒绝结算。
- 非 owner 不能关闭仓位。
- 管理员不能直接提取用户资金。

## 14. 当前区块即时清算主模型

平仓交易必须在当前区块内确定精确可领取金额，因此生产合约采用方向权益池 shares 模型作为实时结算主模型。epoch 分片和逐仓遍历只能作为审计、仿真或非实时统计辅助。

核心状态：

```text
long_pool_equity
short_pool_equity
long_total_shares
short_total_shares
long_recovery_shares
short_recovery_shares
long_dormant_bucket_root
short_dormant_bucket_root
long_notional
short_notional
last_price
```

用户仓位持有 shares，平仓金额为：

```text
withdrawable = pool_equity * user_shares / total_shares
```

该模型是白皮书配对清算原则在大规模链上环境中的生产实现：同方向用户通过 shares 表达权重，方向池之间按价格变化和对手方可赔付权益完成 O(1) 资金转移。

链上指令顺序应为：

```text
sync_pool_to_current_price()
then open_position / close_position
```

如果某方向池权益为 0 且仍存在历史恢复 shares，合约必须防止新开仓稀释历史恢复权，同时必须继续允许任何方向新开仓。

生产方案应实现 `active_shares / recovery_shares` 双 shares 结构：

- 历史零价值仓位进入 `recovery_shares`。
- 新用户本金进入 `active_shares`。
- 未来对手方亏损形成的 transfer 不得按开仓时间或 recovery/active 身份设置优先级。
- `active_shares` 与 `recovery_shares` 只用于账务分层和防稀释；分配时所有当前有盈利/恢复 claim 的仓位按 claim 权重比例分享。
- 任何时候不得因为某方向池权益为 0 而禁止该方向新开仓。
