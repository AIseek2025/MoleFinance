# MoleOption 合规与地理屏蔽

## 1. 背景

MoleOption 提供高杠杆链上衍生品产品。在大多数司法辖区，向公众提供杠杆衍生品需要金融牌照、KYC、AML 流程、风险揭示书、广告限制等合规要求。即使协议本身高度去中心化，前端、域名、API、运营公司、关键开发者团队仍可能被追究责任。

本文件不构成法律建议，但确立 MoleOption 在合规边界设计上的工程级要求。

## 2. 合规分层

将合规约束按“链上协议层 / 协议托管前端层 / 受监管运营公司层”分开：

```text
Layer 0 链上程序        无 KYC, 全开放, 仅受智能合约规则约束
Layer 1 官方前端站点    对受限地区做 IP / 钱包 / 设备屏蔽, 强制风险揭示与 ToS
Layer 2 官方移动 App    高强度合规, 可能要求 KYC/KYB
Layer 3 受监管前端     针对特定司法辖区, 可能集成机构 KYC 与 OTC 通道
```

链上程序无法单独实施 KYC，但官方运营前端必须严格执行屏蔽。这与去中心化精神不冲突：协议开放，但官方运营受合规约束。

## 3. 受限地区策略

### 3.1 地理屏蔽

官方前端必须维护“受限地区列表”：

```text
restricted_jurisdictions = [
  "United States and U.S. territories",
  "Canada (subject to provincial review)",
  "European Union jurisdictions where derivatives require MiFID II authorisation",
  "United Kingdom",
  "China mainland",
  "Hong Kong SAR",
  "OFAC sanctioned jurisdictions",
  ...
]
```

具体清单必须由法律顾问审定。运营方应确保：

- IP 地理判断结合 GeoIP + ISP 数据库 + ASN 屏蔽。
- 拒绝来自已知 VPN 出口节点的访问，或对 VPN 用户额外提示。
- 网站 ToS 明确写明用户责任。
- 明显规避屏蔽的行为不得被视为合规。

### 3.2 钱包屏蔽

Solana 钱包地址本身不携带身份，但官方前端可以接入风控数据：

- 链下风控数据库，标记被制裁地址、混币器输出、已知诈骗资金来源。
- 出现命中时拒绝交易构造。
- 命中策略对所有用户一致，不允许人工豁免。

### 3.3 设备指纹

至少使用以下信号联合判断：

- IP / ASN / GeoIP。
- 浏览器指纹（user agent, canvas, font, timezone）。
- 钱包地址历史。
- 注册或访问时间。

不应在未告知用户的情况下上传非必要数据。隐私政策必须公开声明收集范围。

## 4. KYC / AML 策略

### 4.1 普通公开前端

公开前端不强制 KYC，但必须：

- 拒绝受限地区。
- 拒绝制裁地址。
- 不主动寻求美国零售用户。
- 不允许营销渠道针对受限地区投放。

### 4.2 机构与高额用户

如果协议愿意服务机构资金，可以单独提供：

- 机构 KYC 入口。
- 来源资金审查。
- 单仓上限提升。
- 法币出入金通道与受监管合作机构。

机构通道与公开前端使用相同链上协议，但身份与限额由 Layer 2/3 应用控制。

### 4.3 反洗钱监控

无论是否 KYC，运营方应：

- 对协议链上活动做 AML 风险评分。
- 与 Solana 链上分析服务合作（Chainalysis, TRM, Elliptic 等）。
- 对高风险地址访问公共前端时显示风险提示并阻止交易构造。
- 对监管查询保留必要的访问日志（在隐私政策中披露保留期限和范围）。

## 5. 法律披露要求

公开前端必须显著展示：

- 协议风险揭示书。
- 衍生品交易的本金风险。
- 永不爆仓不等于永不亏损。
- 流动性不足时盈利可能无法兑现。
- shares 模型的 LP 性质：进场即接受当前池子已含的隐含盈亏分布。
- 当前权益为 0 仓位的恢复机制和潜在不可恢复风险。
- 合约可升级与升级风险。
- 没有任何方面承诺收益、保本或固定回报。

签字流程：

- 首次访问需要点击同意 ToS 与风险揭示。
- 高杠杆开仓必须二次确认。
- 同意记录链下保存（hash + timestamp + wallet 签名），不强制上链以避免隐私问题。

## 6. 营销与对外沟通规范

### 6.1 禁止

- 在受限地区投放广告。
- 在任何渠道宣传“无亏损可能”、“保本”、“无风险”。
- 在任何渠道宣传具体收益预测。
- 暗示协议有牌照或保险。

### 6.2 允许

- 解释机制原理，强调本金封顶、零滑点、无资金费率。
- 强调 LP 风险与流动性兑现率的本质。
- 对特定受监管地区的合作伙伴营销，需获得当地法律意见。

## 7. 隐私与数据保护

### 7.1 数据最小化

收集仅完成业务所需的数据：

- IP（用于地理屏蔽）。
- 钱包地址（链上公开信息）。
- 浏览器指纹（用于风控）。
- 不收集姓名、身份证、手机、邮件等 PII，除非进入 KYC 流程。

### 7.2 法律要求

- 满足 GDPR 数据主体权利（如适用）。
- 满足 CCPA、PIPEDA 等其他地区数据法律。
- 提供数据删除与调取请求渠道。

### 7.3 数据留存

日志数据按司法辖区要求设定最短和最长留存期：

- 一般留存 6 个月，用于运营审计。
- 风控相关日志可延长至 24 个月。
- 用户主动注销时执行可删除字段的删除（链上数据无法删除，但运营层可以删除前端账户绑定）。

## 8. 制裁名单合规

运营方必须：

- 接入主流制裁名单（OFAC SDN、EU、UK、UN）。
- 链上分析数据库。
- 当命中时禁止任何前端操作。
- 不解封任何受制裁地址。

链上协议本身不具备制裁屏蔽能力，但官方前端必须强制实施。

## 9. 多前端策略

合规友好的多前端策略：

- 官方主前端：保守屏蔽，不提供 KYC。
- 机构前端：完整 KYC + 法定风险揭示书 + 出入金通道。
- 社区开源前端：允许任何第三方在自身合规框架下接入协议；官方不背书。

链上协议不区分前端来源，但官方应公开声明：

- 仅对官方前端做合规承诺。
- 第三方前端用户自行承担额外风险。

## 10. 财务与税务

### 10.1 协议金库

`fee_vault` 必须在受认可法人实体名下管理，账目对外披露：

- 收入来源。
- 支出项目（开发、审计、运维、市场）。
- 余额。

### 10.2 用户资金

- 用户资金始终由 Vault PDA 控制，运营方任何人都不得拥有 Vault 直接转账权限。
- 多签紧急权限只能触发暂停与限流，不能挪用资金。

### 10.3 税务

运营方在自己的法人实体所在司法辖区遵守税法。用户的税务责任由用户承担，前端展示中可以提供链上交易记录导出，但不提供税务建议。

## 11. 与 16 号文档的衔接

合规事件 (例如某地区监管下达通知) 可触发：

- 对该地区前端访问的立即屏蔽 (Level 0)。
- 治理提案要求修改协议参数 (Level 1 / 2)。
- 紧急暂停某市场 (Level 0 / 1)。

这必须遵守 `16-合约升级与治理紧急响应.md` 的权限边界，不能因合规借口绕开多签或时间锁的资金路径限制。

## 12. 应急合规事件响应

### 12.1 监管询问

收到监管机关询问时：

- 法律团队主导回应。
- 不在没有法律建议下擅自暂停协议或冻结特定钱包。
- 公开回应内容前与多签治理沟通。

### 12.2 法院命令

链上协议无法定向冻结某个钱包；如果监管要求，运营方至多可以：

- 屏蔽该钱包在官方前端的访问。
- 在合规要求范围内披露链上活动数据（链上数据本身公开）。
- 不得修改链上状态以协助冻结，因为这会破坏协议中立性，损害所有其他用户。

### 12.3 协议失误导致的合规风险

例如：发现协议向受制裁地址支付。

- 上链路径不可逆，资金无法追回。
- 运营方应主动披露事件、分析原因并修复链下风控漏洞。
- 受影响法人主体应配合监管调查。

## 13. 与 15 号文档审计漏洞清单的关系

合规失误同样是高危风险：

- 即使智能合约 100% 安全，受限地区被监管打击可能导致前端被强制下线。
- 没有 KYC 不等于没有 AML 责任。
- 受制裁地址使用协议可能引发协议团队个人责任。
- 升级后用户资金路径变化必须重新合规评估。

## 14. 合规验收门槛

主网上线前必须满足：

- 法律意见书覆盖目标司法辖区。
- 受限地区屏蔽 QA 通过。
- KYC（如启用）流程符合当地金融机构最低要求。
- 风险揭示书、ToS、隐私政策三件套上线并存档。
- 应急合规事件响应流程文档化。
- 关键签名密钥与法人实体合规结构匹配。
- 公开域名注册信息合规。

## 15. 结论

去中心化不是合规豁免。MoleOption 协议本身可以无需许可，但官方前端、运营公司、营销活动和资金账户必须在严格合规边界内运作。本文件给出最低工程级合规要求；每次进入新司法辖区前都必须重新进行法律评估。
