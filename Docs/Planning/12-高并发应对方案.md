# MoleOption 高并发应对方案

## 1. 结论先行

朴素“每次开平仓遍历全市场所有仓位”的清算算法，不能支撑 1 亿用户，也不能在 Solana 单笔交易中执行。它只能作为离线数学原型、仿真 oracle 和审计复现工具，不能进入真实资金智能合约。

首个真实资金智能合约必须采用：

- Settlement Epoch。
- Market Sharding。
- Shard Batch Settlement。
- Epoch Finalization。
- Position Lazy Settlement。
- Off-chain Index + On-chain Verification。
- 多级限流和受限模式。

目标是在不改变 MoleOption 核心清算语义的前提下，让系统在亿级仓位下仍能做到精确、可审计、可恢复。

## 2. 清算语义不可改变

无论系统规模多大，每个 epoch 的最终结果必须等价于朴素全量遍历：

```text
TOTAL_PROFIT = sum(profit_increment_i)
TOTAL_REALIZABLE_LOSS = sum(realizable_loss_increment_i)
ACTUAL_TRANSFER = min(TOTAL_PROFIT, TOTAL_REALIZABLE_LOSS)

realized_profit_i =
  ACTUAL_TRANSFER * profit_increment_i / TOTAL_PROFIT

realized_loss_i =
  ACTUAL_TRANSFER * realizable_loss_increment_i / TOTAL_REALIZABLE_LOSS
```

亏损仓位扣减顺序：

```text
profit_credit_spent_i = min(realized_profit_balance_i, realized_loss_i)
locked_loss_increase_i = realized_loss_i - profit_credit_spent_i

realized_profit_balance_i -= profit_credit_spent_i
locked_loss_i += locked_loss_increase_i
```

该语义必须在单用户、双用户、百万用户、亿级用户下保持一致。

## 3. 场景推演

### 3.1 只有一个用户

如果市场只有用户 A：

- A 开多或开空。
- 价格变化后，A 可能有账面盈利或账面亏损。
- 但没有对手方，`TOTAL_PROFIT` 或 `TOTAL_REALIZABLE_LOSS` 至少一侧为 0。

结果：

```text
ACTUAL_TRANSFER = 0
```

A 平仓时只能取回未被锁定的本金和已存在的未提取收益。由于没有对手盘，单用户不能凭空产生盈利，也不会因为无人盈利退出而被锁定新亏损。

### 3.2 两个用户

用户 A 和 B 方向相反或价格路径不同：

- A 亏，B 盈：B 平仓时最多拿 `min(B账面盈利, A可兑现亏损)`。
- A 盈，B 亏：反向同理。
- 两人都账面盈利：没有亏损来源，不能兑现新增盈利。
- 两人都账面亏损：没有盈利索取方，不发生转移。

### 3.3 一个用户退出后只剩另一个用户

如果 B 已经从 A 的亏损中拿走盈利并退出：

- A 的 `locked_loss` 增加，不可逆。
- 之后如果市场只剩 A，即使价格反转，也没有新的亏损方提供恢复资金。
- A 的账面表现可以改善，但不能凭空恢复可提取权益。

### 3.4 第三个用户进入

如果 C 进入后价格反转，C 成为亏损方，A 成为盈利方或恢复方：

- C 的可兑现亏损形成新流动性。
- A 可以从 C 的亏损中获得 `realized_profit_balance`。
- A 的 `locked_loss` 不减少，但可提取权益增加。
- 如果 A 以及其他历史亏损仓位需要恢复的金额大于 C 等新亏损方的可兑现亏损，则恢复总额仍然以新亏损方可兑现亏损为上限。

### 3.5 亿级用户

亿级用户只是上述规则的高维组合。复杂性来自：

- 同时有新开仓。
- 同时有平仓。
- 仓位经历多次盈利、亏损、恢复、再亏损。
- 用户交互发生在不同 epoch。
- 部分仓位长期不交互。
- 结算任务必须分批完成。

因此系统必须把“全局数学结果”拆成多个可验证阶段执行。

### 3.6 极端行情与区块边界

数字货币市场可能在极短时间内上涨数倍、下跌 90%，并伴随数百万到数千万用户同时开仓和平仓。

“每个区块精准清算”的想法在原则上抓住了关键：清算必须有明确的时间边界和价格边界。但在工程上，不能要求每个 Solana slot 内完成全市场所有仓位的最终写入。正确做法是：

- 以 slot 或价格更新为 epoch 边界，冻结该 epoch 的起止价格。
- 在后续多个交易中分片统计并最终化该 epoch。
- 未最终化 epoch 不允许确定性提款。
- 最终化后，任意仓位懒结算结果必须等价于该 epoch 边界下的全量遍历结果。

因此，协议追求的不是“每个区块内同步写完所有仓位”，而是“每个区块/epoch 的输入边界确定，最终化结果精确可验证，提款只基于 finalized 状态”。

## 4. 为什么直接遍历不可行

Solana 单笔交易限制包括：

- 账户数量限制。
- 计算单元限制。
- 交易大小限制。
- 写账户锁冲突。
- RPC 和索引吞吐瓶颈。

若每次开平仓都需要读取和写入所有仓位：

- 1 万仓位已经难以承受。
- 100 万仓位完全不可执行。
- 1 亿仓位不可能单交易结算。

因此，白皮书的“全局重置”应理解为数学语义，而不是单笔链上遍历实现。

## 5. 生产架构

### 5.1 Settlement Epoch

市场按时间或事件切分 epoch：

```text
epoch_id
start_price
end_price
start_slot
end_slot
status
```

状态：

```text
Open
CollectingShards
Finalized
Failed
Restricted
```

开仓和平仓不能跨越未最终化 epoch 直接确定提款，必须先应用已最终化结算。

### 5.2 Market Sharding

每个 Market 拆成多个 Shard：

```text
shard_id = hash(position_id) % shard_count
```

或按方向、价格区间、仓位规模进一步分片。

每个 Shard 维护：

```text
open_position_count
profit_sum
realizable_loss_sum
last_settled_epoch
```

Shard 之间可以并行结算，降低写账户冲突。

### 5.3 Shard Batch Settlement

每个 shard 的结算可以分多笔交易完成：

```text
settle_shard_batch(epoch_id, shard_id, positions[])
```

该指令只处理有限数量仓位，并累加：

```text
shard.profit_sum += batch_profit_sum
shard.realizable_loss_sum += batch_realizable_loss_sum
```

所有 batch 完成后，Shard 标记 finalized。

### 5.4 Epoch Finalization

当所有 Shard 完成统计：

```text
market.total_profit = sum(shard.profit_sum)
market.total_realizable_loss = sum(shard.realizable_loss_sum)
market.actual_transfer = min(total_profit, total_realizable_loss)
```

写入：

```text
profit_rate = actual_transfer / total_profit
loss_rate = actual_transfer / total_realizable_loss
```

该 epoch 进入 Finalized。

### 5.5 Position Lazy Settlement

仓位不需要在 epoch finalize 时立即写入。仓位在以下情况应用历史 epoch：

- 用户平仓。
- 用户刷新仓位。
- 用户修改仓位，如果未来支持。
- 后台批处理迁移。

应用公式：

```text
for epoch in position.last_settled_epoch+1..market.finalized_epoch:
  delta = calculate_delta(position, epoch.start_price, epoch.end_price)

  if delta > 0:
    realized_profit_balance += delta * epoch.profit_rate
  else:
    charged = min(abs(delta), current_equity_capacity) * epoch.loss_rate
    spend realized_profit_balance first
    then increase locked_loss
```

懒结算使长期不交互仓位不消耗链上计算。

## 6. 开仓和平仓并发规则

### 6.1 开仓

开仓时：

1. 读取最新可用价格。
2. 确认当前 market 不处于不可接受的结算中断状态。
3. 将新仓位加入当前 active epoch 之后。
4. 新仓位不参与入场前已发生的 epoch。

### 6.2 平仓

平仓时：

1. 应用该仓位所有未结算 finalized epoch。
2. 若当前 epoch 未 finalized，可选择：
   - 等待最终化后平仓。
   - 以受限模式提交平仓请求，最终化后领取。
3. 计算可提取权益。
4. 标记 closed。

### 6.3 平仓请求队列

大规模并发下，平仓可以拆成：

```text
request_close_position
claim_closed_position
```

用户先请求退出，系统在 epoch finalized 后确定金额，用户再领取。

这能避免未最终化状态下错误支付。

## 8. 精确性保障

### 8.1 等价性测试

任何分片清算结果必须等价于朴素遍历：

```text
naive_result == sharded_result +/- dust
```

### 8.2 舍入策略

- 分片统计尽量不提前舍入。
- 最终分配时统一 floor。
- dust 留在 Vault。
- 不允许 dust 被攻击者系统性提取。

### 8.3 可重放性

每个 epoch 的输入应可由链上事件和账户状态重放，方便审计。

## 9. 高并发吞吐策略

### 9.1 多市场隔离

不同标的和杠杆天然隔离，可以并行处理。

### 9.2 Shard 并行

同一 Market 内多个 Shard 并行结算。

### 9.3 Keeper 网络

多个 keeper 竞争或协同处理 shard batch。

### 9.4 索引服务

索引服务负责：

- 显示估算权益。
- 展示结算进度。
- 提供待处理 shard 列表。
- 帮助前端构造交易。

索引服务不能决定最终提款金额。

## 10. 受限模式

当 epoch 长时间未最终化：

- 禁止确定性平仓提款。
- 允许提交平仓请求。
- 可禁止新开仓。
- 显示市场处于结算延迟。
- keeper 或治理补齐 shard。

## 11. 风险与代价

该架构解决规模问题，但带来：

- 结算延迟。
- 实现复杂度提高。
- 前端需要解释 pending 状态。
- 审计难度提高。
- Keeper 和索引基础设施要求提高。

这些代价是亿级用户规模下保持精确清算必须付出的。

## 12. 推荐实施路线

1. 离线模拟器：直接遍历，只作为数学 oracle。
2. 生产级合约开发：直接实现 epoch、shard、close request 和懒结算账户结构。
3. 测试网压力测试：使用百万到亿级模拟仓位验证结果等价性。
4. 限额主网：限制 TVL、市场数量、单仓规模和开放节奏，但不简化核心合约架构。
5. 主网扩展：动态 shard、keeper 网络、批量懒结算和多市场扩展。

## 13. 上线门槛

进入大规模主网前必须证明：

- 分片结果与朴素遍历结果等价。
- 单笔交易永不需要全市场账户。
- epoch 失败可恢复。
- 平仓不会在未最终化 epoch 下超额支付。
- 亿级仓位仿真下核心不变量成立。

## 14. 最终判断

MoleOption 的清算机制在数学上可以支撑任意数量用户，但工程上不能用全局遍历实现。只有采用 epoch 分片清算和仓位懒结算，才能在保持精确性的同时支撑上亿用户和高并发交易。

由于产品要求“当前区块内立即给平仓用户确定可领取金额”，异步 epoch 分片最终化方案不作为生产实时清算主路径。生产主路径采用 `14-当前区块即时清算与简化模型评估.md` 中的方向权益池 shares 模型，用 O(1) 池级状态更新实现当前交易即时提款。

在该路线下，高并发目标不是“所有链下请求都进入同一个 0.4 秒区块”，而是“每笔被纳入区块的交易都能在本交易内 O(1) 同步池状态、计算提款并完成转账”。吞吐扩展依赖子池分片、账户写锁分散和交易入口限流。

采用方向权益池模型后，必须处理亿级零价值仓位。归零仓位不能被删除，但在价格继续向不利方向移动时也不能继续参与可赔付亏损或盈利 claim。生产实现应按 `zero_price_tick` 把 dormant 仓位聚合进链上 bucket，并用可验证的前缀/后缀聚合结构在价格回到锚点时重新纳入统计。
