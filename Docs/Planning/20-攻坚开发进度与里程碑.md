# 20 · 攻坚开发进度与里程碑

> 本文档跟踪 MoleOption 底层（数学 / 清结算 / 智能合约 / 预言机 / 索引）攻坚阶段的实现进度。每完成一波交付，会在此追加一个章节。规划文档（01–19）描述"应该做什么"；本文档描述"已经做了什么、用什么测试证明、下一步攻坚什么"。

## 第一波交付：数学与清结算引擎落地

完成时间：见仓库 `git log` 最早一批 `crates/` 提交。

新增 crate：

- `crates/molemath/`：定点货币级算术原语；`mul_div_floor / mul_div_ceil / signed_pnl_increment / price_move_bps`；所有算术 checked 不溢出。
- `crates/clearing-core/`：O(1) 方向权益池 shares 模型的 Rust 实现，包含 `MarketParams / SubPool / Position / DormantBucket / DormantStore`，引擎函数 `sync_pool / open_position / close_position / force_close_zero_value_position / claim_dormant_recovery / harvest_dust`，惰性迁移 `lazy_migrate_position`，运行期不变量 `check_subpool_invariants`。
- `crates/simulation/`：白皮书 §3 逐仓比例清结算预言机，作为 `clearing-core` 的离线 ground truth。
- `programs/mole-option/`：Solana / Anchor 合约骨架。每个指令处理函数都是 `clearing-core` 的薄封装，自身只做账户校验、PDA 推导和 SPL 转账。

测试覆盖：

- 31 个测试全过：molemath 单元/属性 17 项 + clearing-core 单元/属性 12 项（含随机漫游守恒，32 例 × 50–150 步） + simulation 等价 2 项。
- 修复的关键 bug：`dilution_safety_bps` 公式在 §5.5 与 `engine.rs::open_position` 中互相不一致（不等号反向，稳态健康池被误判为反向稀释风险）。两边已对齐为 `pool_equity * 10_000 < shares * dilution_safety_bps`，并新增 `dilution_safety_passes_at_steady_state` 单测。

锁定的工程决策：

1. **Rust workspace 多 crate**：核心数学/清结算放在 host 可测的纯 Rust crate；Solana 程序作为 thin wrapper。
2. **资金永远 floor 向协议**：所有 `mul_div` 都是 `floor`，残差进 `<dir>_dust` 而不是返回给用户。
3. **零权益不等于关仓**：`force_close_zero_value_position` 必须显式 `acknowledge_forfeit=true`，否则保留 `recovery_shares` 等待对手方亏损流动性。
4. **active_generation + RotateRecord 惰性迁移**：池级别 active→recovery rotation 不立即扫描每个仓位；用户下次触达自己仓位时，引擎对比 `position.active_generation` 与池当前 generation，必要时把 active_shares 折叠进对应 dormant bucket。

由本波交付直接回答的产品问题：

- 用户在前端看到的"我现在浮盈/浮亏多少"如何精准生成？  
  答：链上只存 `active_shares / recovery_shares`；indexer 重放事件流得到逐仓 `(principal, locked_loss, realized_profit_balance)`，对称场景下与白皮书 oracle 逐字节相等，异质场景下与链上 `withdrawable` 误差有界并被属性测试持续监控。
- 价格快速大幅波动时，链上是否还能在当前区块完成精确清算？  
  答：`clearing-core` 的 `sync_pool` 是 O(1) 操作；`pyth-adapter` 在进入 `sync_pool` 之前已经把 oracle 攻击面（错主、错魔数、过期、置信度过宽、负价等）逐项拒绝；多周期 dormant 测试证明在多次池清零 + 恢复 + 再清零的极端场景下仍然守恒。
- 链下重建是否可以独立验证链上结算？  
  答：indexer 只消费链上事件，不读取链上状态；它的 `(locked_loss, realized_profit_balance)` 投影与白皮书 oracle 逐字节匹配的事实，意味着任何独立的第三方监控者都可以从节点 RPC 拉取事件流、复现同一份逐仓视图、与链上 `withdrawable` 自动比对。

## 第三波交付：Dormant 可扩展性 + 索引聚合 + CU 预算

新增文档：

- `Docs/Planning/21-Dormant存储与CU预算.md`：把 `clearing-core::DormantStore` 的 **Eager** 与 **Lazy** 两种模式翻译成 Solana 真实 CU 数字，给出 K（活跃 bucket 数）的可承载上限矩阵：默认 200k CU 预算下 Eager ≤ 80 / Lazy ≤ 13 000；单笔放开到 1.4M CU 后分别为 ≤ 700 / ≤ 92 000。同时列出 `Market` 中所有 dormant 相关治理参数与紧急上限。

`clearing-core` 增量：

- 修复 `redeem_split_in_time_matches_redeem_at_end` 测试（更名为 `partial_redeem_then_sync_forfesits_share_of_subsequent_distribution`）：旧版断言 `drift ≤ 8`（rounding only），与白皮书"耐心持有者奖励"语义直接冲突。新版断言三条结构性不变式：
  1. `order_a_total ≥ order_b_total`：等到最后再赎回的用户从不被中途分批赎回的用户跑赢——这是协议的**反 grief** 保证。
  2. `drift ≤ analytic_upper_bound = (notional/2) · (p_2 − anchor) / anchor + 16`：可量化的损失上限，等价于"半 burn 的 bucket 在第二次 sync 上能拿到的 intrinsic 上界"。一旦突破说明引擎在 distribute 阶段误算了。
  3. `drift > 8`：证明路径相关性是真实经济效应而非 rounding 噪声——如果模型被错误改成"notional 不随 redeem 缩小"，drift 会塌缩到 rounding，被这条断言抓住。
- `engine.rs::pre_sync_dormant_bucket` 在 `Vec::with_capacity(1) + push` clippy 警告下改用 `vec![]`（不影响语义，仅构造效率）。

由本波交付回答的产品/工程问题：

- **百万级 dormant bucket 是否拖死 sync_pool？** 默认预算 K ≤ 80 拖不死；K > 80 时 lazy mode + ledger 把单 sync 的 K 上限放大到 13 000，超过这个再上 1.4M CU 预算。整个上限矩阵在 §21 量化。
- **lazy mode 切换是否破坏白皮书语义？** 不会。`tests/dormant_lazy_equivalence.rs` 4 个 prop 测试（含 400 步随机游走 + 激进 compaction 周期）证明同一操作序列在 eager/lazy 实现上**逐字节同状态**。lazy 模式只是把"何时写 bucket 账户"从 sync 时延后到用户 / keeper 触碰时，对外可观察的金额、shares、accrued 完全一致。
- **为什么提前赎回的用户拿到的钱更少？是 bug 吗？** 不是 bug，是设计。白皮书第 6 节"后续流动性可恢复历史亏损"明确说仍在 bucket 中的仓位才能瓜分对手方后续亏损。`partial_redeem_then_sync_forfesits_share_of_subsequent_distribution` 测试的 3 条新断言把这个语义从"模型 implicit"提升为"测试 explicit"。
- **前端怎么实现"该市场目前共有 X 个多仓 / Y 个空仓 / 未平仓本金 Z USDC"？** 新增的 `IndexerState::sub_pool_stats` 和 `dormant_inventory` 直接返回这些聚合，复杂度 O(positions) 单次构建，可以在每次 `PoolSync` / open / close 事件后增量刷新。

## 第四波交付：协议级仿真器 + 原子事务语义收尾

新增 crate（已在第三波末尾出现，第四波收口）：

- `crates/protocol-harness/`：把 `clearing-core` + `indexer` + 模拟 SPL/费用金库缝合成"主机端可自由仿真的协议状态机"，每一步操作都跑一遍守恒检查 + bucket-by-bucket 平价检查。详情见 `Docs/Planning/22-wave3-protocol-harness.md`。

第四波单独完成的关键修复（**核心架构性 bug，影响所有失败 close/sync**）：

- 通过 harness 的 1000-op 随机走子捕获 chain↔indexer 漂移，最初观察到 ~0.7%–1.5% 的聚合层漂移（链上从未少付，但前端账面会"惊喜地多收"）。Wave 4 把根因定位到 **harness 没有模拟 Solana tx-revert 语义**：`close_position` 在内部依次 mutate `sub_pool`（sync）→ `position`（lazy_migrate）→ 烧 active 份额（mutate sub_pool）→ 从 dormant bucket redeem（mutate sub_pool，**可能直接删除桶**）→ 最后才检查 `withdrawable == 0`。链上 Solana runtime 在 Err 时会把所有账户写回滚、丢弃事件；host 端 harness 没做这件事，导致链上 mutate 出去了、indexer 永远收不到 burn 事件，逐步累积成"indexer 维护了一个链上已不存在的 ghost bucket"。
- **修复**：harness 每个变更性入口（`open / close / force_close / claim_recovery / sync / harvest_dust`）现在都在调用 engine 前 `clone()` 一份 `sub_pool`（必要时还有 position）作为快照，engine 返回 `Err` 时把快照写回，丢弃事件。这与 Solana runtime 的 atomic commit-or-rollback 行为一一对应。
- **效果**：`aggregate_chain_payouts_match_indexer_with_rotations` 的聚合漂移从 ~0.7%–1.5% 直接降到 < 1 ppm（170 raw units / ~2×10¹⁰ deposits，纯属 floor-rounding 余项），测试上限也从 wave 3 临时放宽的 2% 收紧回 **1 ppm（带 1024 单位绝对地板）**。
- **回归保护**：新增 `crates/protocol-harness/tests/atomic_revert.rs`（2 用例），构造长仓深度 crash 后 close 必然 `WithdrawableZero` 的场景，断言 sub_pool/position/buckets/indexer 在失败 close 前后**字节相等**。
- **链端契约**：`clearing_core::close_position` 的 doc-comment 现在显式声明该函数"非内部原子"，调用方负责 Err-on-failure 的 snapshot/restore。链上由 Solana tx runtime 提供，链下由 harness 提供。

测试覆盖（累计 **105/105 通过**，clippy `-D warnings` 干净）：

- 第三波末新增的 27 个 harness 用例（smoke 7 + random_workload 1 + indexer_parity 3 + rotation_focused 7 + adversarial 9）。
- 第四波 +2 atomic_revert 回归。

## 第五波交付：链端 dormant 桥接器 (chain-mirror) + harness↔chain byte-equal 平价

第五波回头审视 `programs/mole-option/src/instructions/sync.rs`，发现一个被 wave 1–4 全部测试链路绕过的 production blocker：

```rust
// programs/mole-option/src/instructions/sync.rs (wave 4 状态)
// Dormant bucket aggregates are persisted in DormantBucket accounts; this
// skeleton does not bridge them. The follow-up phase wires Fenwick trees.
```

`clearing_view(sp_acc)` 每条指令都构造一个**全新空 `DormantStore`**（host 侧 BTreeMap 引用实现），跑完 engine 后写回时**不持久化任何 dormant 状态**。这意味着——尽管 wave 1–4 的 105/105 测试全过——只要主网上线后第一次发生 rotation：

1. rotation 创建的 bucket 在内存里，tx 结束就丢失。
2. 任何 dormant 仓位的 close/force_close 看到的是空 store，要么 panic，要么静默归零。
3. `claim_dormant_recovery` 完全失效。

第五波填上这个 dormant 数据通路。新增 crate：

- `crates/chain-mirror/`：在 host 上完整复刻 Anchor 程序的**账户级运行时模型**——不是 BPF 解释器、也不走 Borsh 真实 wire format，而是把"每个 Anchor `#[account]` 当独立宿主结构 + 按 `remaining_accounts` 模型读写 + 每条指令前后 snapshot/restore 模拟 Solana tx revert"这套语义忠实搬到 host。每条 instruction handler 在指令入口用 `clearing_core::unpack_dormant_store` 把所有 `DormantBucket` PDA 与 `DistributionLedger` PDA 解包成内存中的 `DormantStore`，运行 engine，再 `pack_dormant_store` 写回每条账户。

`clearing-core` 增量：

- `OnChainBucketRecord` (96 字节)、`OnChainLedger`、`OnChainLedgerEntry` (64 字节)、`pack_dormant_store`、`unpack_dormant_store` 已在 wave 3-4 准备好；wave 5 第一次让它们端到端跑通所有 instruction（不只是 dormant 单元测试）。

测试覆盖（累计 **125/125 通过**，clippy `-D warnings` 干净）：

| crate | 新增 | 累计 |
| --- | --- | --- |
| `chain-mirror` smoke | open/close round-trip · rotation 后 bucket 持久化 · 失败 close 全量 revert · harvest dust 守恒 = **4** | 4 |
| `chain-mirror` parity | eager (3 seeds × 400 ops × 2 sub-pools) + lazy (3 seeds × 400 ops × 2 sub-pools, 含 drain) + stress (3 seeds × 800 ops × 3 sub-pools, 周期 ±50 % 价格冲击) = **3** | 3 |

**核心 wave 5 平价契约**：把同一组随机操作流（open/close/force_close/claim_recovery/sync/harvest_dust）同时驱动 `protocol_harness::Harness` 与 `chain_mirror::ChainRuntime`，每一步后断言：

1. 两边 Ok/Err 与错误分类（WithdrawableZero / PositionNotOpen / Other）一致。
2. 全量状态观测**字节级相等**——sub_pool 标量聚合、按 tick 排序的 DormantBucket 记录、按 `(sub_pool, direction)` 索引的 DistributionLedger header + entries、Position 状态、vault/fee_vault/total_deposits/total_withdrawals。
3. 每个 runtime 各自的守恒不变量（vault 分解）保持。

因为 chain-mirror 的 dormant 数据通路是**纯**通过 `pack_dormant_store ↔ unpack_dormant_store` 走的，平价测试通过即等价于：

> "把 wave 1–4 的全部业务语义在 Anchor 的多账户模型下重放一遍后，仍与 host engine byte-equal。"

这是在不安装 Solana toolchain 前能给出的**最强 BPF parity 保证**：后面真正写 Anchor instruction handler 时，每一行只是在做 `AccountInfo<T> ↔ POD bytes` 的搬运，可以靠 chain-mirror 的差分测试反推正确性。

第五波单独完成的关键发现（**真实 engine 留账，影响 lazy mode 的链上守恒**）：

### 6.1 Wave 5.5：`pending_distribution_total` 真正落地（含 engine root cause）

平价测试在 wave 5 末尾的现象——"lazy mode 的 vault 决算缺一项"——在 wave 6 攻坚时挖出了**比文档原本判断更深一层的 engine 缺陷**。

```text
seed=0 step=50 op=Compact:
  pre   pending=3517647 live=10850349
  post  pending=3517647 live=1743318
  ← compact 丢了 9107031 单位的 allocated_sum，
    但 pending 一分没减
```

上面这条 trace 本来不应该出现：`compact_ledger` 只 drop "所有 bucket 都已经 walk 过的" 条目，按 invariant 这些 entries 的 pending 应当已被 `apply_pending_to_bucket` 减到 0。但**在混合 eager+lazy 操作流下不是这样**：

```text
1. distribute_lazy(p1): pending += 10, ledger.push(e0). bucket A.last_applied = 0.
2. distribute(p2)（eager）：
   ▷ 直接读 bucket.accrued = 0 算 outstanding（bug：没 drain pending）。
   ▷ 给 A 加 share=20, A.accrued=20。
   ▷ ★ 把 **所有 bucket（含 A）** 的 last_applied 推到 event_index+1=2。
3. apply_pending(A)：last_A=2=next_event=2 → 立刻返回 0。
   ← e0 的 share=10 永远跳过，pending 卡死 10 个单位。
```

修复（`crates/clearing-core/src/dormant.rs::DormantStore::distribute`）：

- **add**：eager 路径开头先 `apply_pending_to_bucket(*key)?` 把每个 activated bucket drain 干净，然后再读 `bucket.accrued` 算 outstanding。生产环境 fixed-mode 市场下这一段是 no-op（pending 始终为 0）。
- **delete**：原本"为所有 bucket（含 non-activated）一律 bump `last_applied = event_index+1`"的优化代码删除——纯 eager 模式下它是免费优化，混合模式下却把 pending lazy entries 跳过去。

接配套：

```rust
// crates/clearing-core/src/dormant.rs
+ pub fn pending_for_bucket(&self, tick: i64) -> ClearingResult<u128>;
//   ↑ 纯只读，与 apply_pending_to_bucket 字节相同的 replay 公式。

// crates/protocol-harness/src/lib.rs
- expected = pool_equity + dormant_accrued + dust
+ expected = pool_equity + dormant_accrued + dormant_pending + dust
+ idx.accrued == chain.accrued + chain.pending_for_bucket(tick)

// crates/chain-mirror/src/lib.rs
- vault == pool_equity + bucket.accrued + dust
+ vault == pool_equity + bucket.accrued + ledger.pending + dust

// crates/chain-mirror/tests/harness_parity.rs

+ h.check_invariants().unwrap();
+ rt.check_vault_decomposition().unwrap();

// crates/clearing-core/tests/onchain_layout.rs

```

### 7.2 关键 bug：`unpack_direction` 不跳过 fully-dead PDA

在 keeper 的真实工作流下：

1. 引擎首次产出新 tick 的 bucket，`pack_direction` 找不到空槽 → `DormantBridgeBucketSlotExhausted`，tx 回滚。
2. keeper 调 `init_dormant_bucket(tick)` 创建空 PDA。
3. 下次 sync_pool 把这个空 PDA 加进 `remaining_accounts`。
4. **`unpack_direction` 把它装进 `DormantStore`**（直接 push record，不论是否纯零）。
5. 引擎 `insert_or_merge(tick, anchor_price, ...)` 看见 `contains_key(tick)` 为真 → 走"已存在"分支 → **`anchor_price` 永远不会被更新**，bucket 永久 inactivable，`compute_intrinsic` 始终为 0，份额不收。

修复：

```rust
// programs/mole-option/src/instructions/dormant_bridge.rs
+ pub fn record_is_dead(b: &DormantBucket) -> bool {
+     b.total_recovery_shares == 0 && b.total_recovery_notional == 0
+         && b.accrued_value == 0 && b.position_count == 0
+ }
+
  pub fn unpack_direction(...) -> Result<DormantStore> {
      ...
      for b in bucket_accounts {
          ...
+         if record_is_dead(b) { continue; }   // ← 关键 fix
          bucket_records.push(...);
      }
  }
```

`pack_direction` 的 Pass 1（match-by-tick）/ Pass 2（dead-slot fill）/ Pass 3（zero-out engine-removed）三个分支也统一切到 `record_is_dead`，桥两端对"空槽"语义保持字节级对齐。

为什么 chain-mirror parity 测试没抓到这个：chain-mirror 不模拟"keeper 提前 init dormant_bucket"工作流——所有 bucket 都是引擎在 `commit_core_sub_pool` 中带着正确 `anchor_price` 写出来的，从来不是预 init 的空槽。这个 bug 只在生产 keeper 路径上触发；wave 7 通过桥层语义修复 + 后续 7.4 中的 keeper drain 等价测试一起覆盖。

### 7.3 `close_dormant_bucket` 指令 —— PDA 池长期可治理

lazy 模式 keeper 网络运行半年后，被引擎 redeem-to-zero 后清零的 `DormantBucket` PDA 会持续累积，每个长期占着 ~120 字节 + rent。新增：

```rust
// programs/mole-option/src/instructions/init.rs
#[derive(Accounts)]
pub struct CloseDormantBucket<'info> {
    pub market: Account<'info, Market>,
    #[account(has_one = market)] pub sub_pool: Account<'info, SubPool>,
    #[account(has_one = sub_pool, /* direction must align */)]
    pub ledger: Account<'info, DistributionLedger>,
    #[account(
        mut, has_one = sub_pool,
        constraint = bucket.total_recovery_shares == 0
                  && bucket.total_recovery_notional == 0
                  && bucket.accrued_value == 0
                  && bucket.position_count == 0
                  @ ProgramError::DormantBucketStillLive,
        constraint = bucket.last_applied_index >= ledger.next_event_index
                  @ ProgramError::DormantBucketHasPendingApply,
        close = receiver,
    )] pub bucket: Account<'info, DormantBucket>,
    /// CHECK: lamport receiver. Permissionless. */
    #[account(mut)] pub receiver: AccountInfo<'info>,
    pub keeper: Signer<'info>,
}
```

两条不变量保证 close 时不会污染状态：

1. **`record_is_dead`** —— bucket 在引擎层无 live 资源。
2. **`last_applied_index >= next_event_index`** —— 该 bucket 在 ledger 上没有 pending lazy 待领（pack Pass 3 把 zero 时已经把 `last_applied_index = next_event_index`，所以工程师只要"等 sync_pool 走过 redeem-to-zero 那一步"就总成立）。

错误码：`DormantBucketStillLive`、`DormantBucketHasPendingApply`。permissionless 设计——任何 keeper 都能触发 close，最坏情况是浪费自己的 CU（拿不到 rent）。

### 7.4 wave 7 真正的攻坚证据：`keeper_drain_equivalence`

`crates/protocol-harness/tests/keeper_drain_equivalence.rs`：双 Harness 并行（一个 eager，一个 lazy + 周期 drain），同一组随机操作流（open/close/force_close/claim/sync/harvest_dust），3 个 seed × 400 步 = **1200 个操作**。每一步：

1. 两边同时执行操作，`Ok` / `Err` 分类一致。
2. lazy 端 `drain_all_buckets(sub_pool, slot)` 走遍每个 live bucket 调 `pre_sync_dormant_bucket`。

### 8.1 chain-mirror 严格 PDA 生命周期模式（防 wave 7.2 bug 回归）

`ChainRuntime::with_strict_pda_lifecycle(true)` 模拟 Anchor 桥层的 PDA 生命周期约束：

- 新 API：`ChainRuntime::close_dormant_bucket(...)` 复制 anchor 的两条不变量（`record_is_dead` + `last_applied_index >= ledger.next_event_index`），失败回 `BucketLifecycleViolation`。
- `clearing_core::OnChainBucketRecord::is_dead()` / `OnChainBucketRecord::dead(direction, tick)` 提升为**dead PDA 单一权威谓词**。`unpack_dormant_store` 直接跳过 dead records，桥层 `record_is_dead` 退化为对同一谓词的 thin wrapper，永远不会再有"两份独立判定"漂移。

新增 4 个 chain-mirror 单测——其中 `strict_mode_rejects_rotate_without_preinit_then_keeper_init_unblocks` 是 wave 7.2 fix 的**永久回归守护**：rotate 无预 init 必报 `BucketSlotExhausted`，keeper init 后再 rotate 必让 `anchor_price` 被正确 promote（不是 0）。

### 8.2 `schema_version` 端到端拒绝矩阵

`clearing_core::SCHEMA_VERSION_CURRENT: u16 = 1` 是单一权威。`assert_schema_version` helper 在每个 funds-touching engine entrypoint 入口先于一切业务逻辑执行：`sync_pool` / `open_position` / `close_position` / `force_close_zero_value_position` / `claim_dormant_recovery` / `pre_sync_dormant_bucket` / `harvest_dust` 全部检查 `market.schema_version`；带仓位的入口额外检查 `position.schema_version == market.schema_version`。任何不一致返回 `ClearingError::SchemaVersionMismatch` + sub_pool 原子回滚。Anchor handler 透过 `map_err` 把错误传到 `ProgramError::SchemaVersionMismatch`，用户得到清晰可操作的报错而不是状态损坏。

1. 加新逻辑落 feature gate（旧字段 frozen），保留常量旧值；

3. Squads 多签 tx 同步 bump `market.schema_version` 与编译时常量；
4. 旧 migration 指令保留一个治理周期作 fallback。

### 8.4 `crates/keeper/` 软件骨架

新 crate（145 行核心 + 14 测试），lazy 模式 keeper 的纯 Rust 调度内核：

- `KeeperChainView` trait 抽象 chain-side 视图（生产用 RPC client、host 测试用 chain-mirror newtype wrapper 实现）。
- `Scheduler::plan(&dyn KeeperChainView) -> Vec<KeeperAction>`：按规则一次性枚举三类 action 并按优先级 sort：
  - **`PreSyncDormantBucket`**：`bucket.last_applied_index < ledger.next_event_index` 时触发；优先级 = `1e7 + pending`（落后越多越紧急）。
  - **`CloseDormantBucket`**：`bucket.is_dead() && last_applied >= head` 时触发；优先级 1（best-effort 租金回收）。
  - **`InitDormantBucket`**：仅在 caller 通过 `Scheduler::record_init_hint` 显式 hint 后产出；优先级 1e9（最高，缺 PDA 会让用户 tx revert）。

- 集成 `tests/chain_mirror_integration.rs`：用 chain-mirror 跑端到端 lazy 闭环——open + crash → plan 输出 PreSync → keeper apply → plan 空 → claim → bucket dead → plan 不会对已 drained 的 bucket 再发 PreSync。

### 9.3 keeper crate `RotateRiskPredictor` —— 自动产 init hint

wave 8 keeper 只在 caller 显式 `record_init_hint` 后才发 `InitDormantBucket` action。wave 9 给 `crates/keeper/` 加：

```text
pub trait KeeperChainView {
    ...
    fn sub_pool_health(&self, _id) -> Option<SubPoolHealth> { None }   // ← 新方法，default None 兼容
}

pub struct SubPoolHealth { last_price, long_anchor_price, short_anchor_price,
                           long_pool_equity, short_pool_equity,
                           long_active_notional, short_active_notional,
                           long_active_generation, short_active_generation }

pub struct RotateRiskPredictor { config: PredictorConfig }
pub struct PredictorConfig { annual_vol, horizon_slots, slots_per_second,
                             min_probability, tick_aggregation_factor, price_tick }
pub struct RotatePrediction { sub_pool_id, direction, zero_price, zero_tick, probability }

impl RotateRiskPredictor {
    pub fn predict_one(&self, h: &SubPoolHealth) -> Vec<RotatePrediction>;
    pub fn populate_scheduler(&self, view, sched) -> Vec<RotatePrediction>;
}
```

数学模型：几何布朗运动假设 + 线性 P&L 反推 zero-price：

- `S_zero_long = long_anchor_price · (1 − long_pool_equity / long_active_notional)`
- `S_zero_short = short_anchor_price · (1 + short_pool_equity / short_active_notional)`
- 一接触概率（terminal-hit 上界）：`P_long ≈ Φ(log(S_zero/S_now) / σ√T)`，反方向取 `1 − Φ`。

实现 detail：自带 Abramowitz-Stegun 标准正态 CDF rational approximation（max abs 误差 ≈ 7.5e-8），不引入 `libm` / `statrs` 依赖以保持 host vs BPF 编译确定性。Predictor **保守跳过** over-collateralised 长仓（`equity ≥ notional` → 仅靠负向漂移无法清零）以及空池（`notional == 0` → 没有 rotation 可能）。

`populate_scheduler(view, sched)` 为每条 prediction 调 `Scheduler::record_init_hint(sub_pool, direction, zero_tick, InitRationale::RotateRiskHorizon)`，下一次 `Scheduler::plan` 调用就会以最高优先级（`1e9`）emit 对应的 `InitDormantBucket` action。

测试覆盖（keeper 单测从 12 → 19 项，integration 从 2 → 3 项）：

- 5 个 predictor 单测：CDF 锚点（`Φ(0)=0.5`、`Φ(1.96)≈0.975`、`Φ(-2.0)≈0.0228`、tail 饱和）；price→tick 网格化；over-collateralised 长仓必须被过滤；空池必须被过滤；倾斜池必须 surface。
- `populate_scheduler_drives_init_hints_into_plan` 单测：预测 → record_init_hint → plan 出 `InitDormantBucket` 闭环。
- `rotate_risk_predictor_drives_init_hint_and_pre_init_pda` 集成测试：在 chain-mirror lazy 模式上做 1% 价格漂移，predictor 检出长仓 zero-tick，scheduler emit init action，keeper `pre_init_dormant_bucket` 应用，dead PDA 落地。

### 10.1 `keeper::RealizedVolatilityEstimator` —— σ̂ 自调谐闭环

wave 9 的 `PredictorConfig::annual_vol` 是手动常量；wave 10 把它接到滚动 realized-vol 估计器：

```text
pub struct PriceSample { price: u64, slot: u64 }
pub struct RealizedVolatilityEstimator { config, samples: VecDeque<PriceSample> }
pub struct RealizedVolatilityEstimatorConfig {
    max_samples, max_age_slots, min_samples, slots_per_second, min_clamp, max_clamp
}

impl RealizedVolatilityEstimator {
    pub fn record(&mut self, sample: PriceSample);          // 顺序保护 + zero-price reject
    pub fn current_estimate(&self) -> Option<f64>;          // None until min_samples 满
    pub fn apply_to_predictor_config(&self, &mut PredictorConfig) -> bool;
    pub fn reset(&mut self);                                 // 暂停-恢复时调
}
```

数学：时间加权 σ̂² = Σ r²ᵢ / Σ Δtᵢ，其中 rᵢ = ln(pᵢ/pᵢ₋₁)、Δtᵢ = (slotᵢ−slotᵢ₋₁) / slots_per_second / SECONDS_PER_YEAR。**不**用简单 stddev(r) × √(N/T) 形式——Solana slot 间隔有 ±20% 抖动，scaled-by-time 公式才能在长跑中保持正确性。窗口同时受 `max_samples`（峰值常量）和 `max_age_slots`（slot 加速时丢旧）双约束。`min_clamp/max_clamp` 给 σ̂ 加 [0.05, 5.0] 默认带，防止短窗 outlier 把 predictor 喂到 σ ≈ 0（少出 hint）或 σ → ∞（出过多 hint）。

8 个 estimator 单测：未 warm 期返 None；常量价 → 落到 floor；合成 GBM 行走 → 量级正确；out-of-order/duplicate slot 静默丢弃；count + age 双驱逐路径；`apply_to_predictor_config` 仅在 warm 后覆写 caller 的手设值；`reset` 清史。

### 10.2 `keeper::KeeperLoop` —— 端到端状态机

封装 wave 9/10 全套构件成一个可由 bot harness 按 cadence 驱动的循环：

```text
pub trait KeeperBotEnvironment {
    fn chain_view(&self) -> &dyn KeeperChainView;
    fn fetch_price_sample(&mut self) -> Option<PriceSample>;
    fn executor(&mut self) -> &mut dyn ActionExecutor;
}

pub struct KeeperLoopConfig { run_predictor: bool, auto_tune_vol: bool }
pub struct KeeperLoop {
    config, scheduler, predictor, vol_estimator
}

impl KeeperLoop {
    pub fn tick<E: KeeperBotEnvironment + ?Sized>(&mut self, env: &mut E)
        -> Result<KeeperLoopOutcome, KeeperError>;
}
```

tick 生命周期（每次同步执行）：

1. `env.fetch_price_sample()` → `vol_estimator.record(...)`
2. `auto_tune_vol` 时 `vol_estimator.apply_to_predictor_config(predictor.config)`
3. `run_predictor` 时 `predictor.populate_scheduler(view, scheduler)` 产 init hints
4. `scheduler.plan(view)` 出 actions
5. 逐条 `executor.execute(action)`，把 `Submitted`/`Failed`/`Skipped` 滚到 `KeeperLoopMetrics`
6. 返回 `KeeperLoopOutcome { metrics, dispatched: Vec<(Action, Result)> }`

显式**不**引 tokio——整个 loop 是同步函数，部署 harness 自己拥有 cadence。这样同一份代码就能驱动：CI 集成测试（`StubBotEnv`）、回测 replay（事件流注入）、生产 bot（tokio interval 包一层）。失败的单条 dispatch 不 abort tick，metrics 字段拆 `actions_submitted/failed/skipped` 让 dashboard 直接出图。

6 个 KeeperLoop 单测：empty pool → 0 metrics；预灌的 init hint → 1 InitDormantBucket dispatch；vol 估计器 warm 后 predictor.annual_vol 被覆写；`auto_tune_vol=false` 时即使 estimator warm 也不动 predictor；flaky executor（隔条失败）的 metrics 拆字段正确；`KeeperLoopMetrics::merge` 字段加和正确（dashboard 滚窗口聚合用）。

```rust

    [0x8f, 0xcb, 0x7b, 0xd0, 0xc2, 0x40, 0x6e, 0x35];
```

热路径写死成 `pub const`（不付每次 SHA-256 成本），但 `discriminator_constants_match_sha256_of_anchor_namespace` 单测每次 CI 都用 `sha2::Sha256` 重算 `sha256("global:<name>")[..8]` 比对常量——任何指令重命名（如把 `pre_sync_dormant_bucket` 改名）会立即让该单测失败，绝对不会出现"常量与 program 不一致还跑过 CI"的 bit-rot 情形。**初次提交时常量错位被这个 self-test 捕获**——三个常量都是错的，由 `print_canonical_discriminators -- --ignored --nocapture` 输出正确字节后修复。

```text
impl KeeperBot {

        &mut self, fetcher: &F, ctx: &MarketContext, builder: B,
        keeper_pk, clock_sysvar, system_program,
    ) -> Result<(TickReport, B), BotError>;
}
```

### 11.1 `keeper-rpc::solana` —— production RPC binding 接通

依赖：把 wave 10 在 Cargo.toml 占位的 `[features.solana-rpc]` 真正落地。三个关键挑战 + 解法：

**(a) 4.0 transition 的版本歪斜**：`solana-sdk = 4.0.1` 重新导出 `solana-transaction = 4.0.0`，但 `solana-rpc-client = 4.0.0` 内部消费 `solana-transaction = 3.1.0`——两个 sibling `Transaction` 类型在 trait 层互不兼容（`SerializableTransaction` 只对 3.1 类型有 impl）。**解法**：完全绕开 `solana-sdk` 伞包，依赖 `solana-rpc-client 4.0` 实际所拉的细粒度 crate（`solana-pubkey 4.1` / `solana-transaction 3.1` / `solana-instruction 3.3` / `solana-commitment-config 3.1` / `solana-keypair 3.1` / `solana-signer 3.0` / `solana-hash 4.2`）。这种"按 transitive 实际版本钉死"的做法把上面那条踩坑日志从下游 wave 12 提前消化掉了。

**(b) 同步 `RpcClient` 不暴露 `get_program_accounts_with_config`**：sync 端只有 `get_program_accounts(prog) -> Vec<(Pubkey, Account)>`，没法附带 `RpcAccountInfoConfig + Memcmp` 过滤——nonblocking 端有但要拖 tokio 进 `AccountFetcher` trait 的同步形状。**解法**：直接走 `client.send::<OptionalContext<Vec<RpcKeyedAccount>>>(RpcRequest::GetProgramAccounts, json!([prog.to_string(), cfg]))`，**完全复刻 nonblocking 高级 API 的 wire shape**，绕开缺失的高级方法。`RpcKeyedAccount.account.data` 是 `UiAccountData`，再 round-trip 一次 `serde_json::Value` 拿到 `[base64, "base64"]` 数组并由 `decode_account_data_blob` 还原。这个 helper 显式拒 `base64+zstd` / `base58` 等编码——keeper 主动只请求 base64，任何其他编码即视为后端配置漂移，立即 fail-fast。

**(c) `Pubkey32 ↔ solana_pubkey::Pubkey` 字节级正确性**：协议侧每个 PDA 都靠这个 32-byte newtype 在 chain mirror、scheduler、keeper-bot、tx builder 里穿——一次方向错位即可让生产 bot 把 ix 发到错的账户。**解法**：`pubkey_round_trip_is_byte_equal` 单测对 8 个 seed（覆盖最低位/最高位 byte 两端）做 `Pubkey32 → Pubkey → Pubkey32` 比对，并以 `dummy_pk32(seed)` 把 byte\[31\] 设为 `0xab` 阻止 truncation/zero-padding 类 bug 蒙混过关。

模块产物（`crates/keeper-rpc/src/solana.rs`，~430 行包注释 + 测试）：

| API | 职责 |
|-----|------|
| `pubkey32_to_pubkey` / `pubkey_to_pubkey32` | 32-byte 字节级身份转换 |
| `account_meta_to_sdk` | 把 wave 10 portable `AccountMeta` → `solana_instruction::AccountMeta`，保 signer/writable 比特 |
| `dispatched_to_instruction` | `DispatchedAction { program_id, data, accounts } → Instruction`，data 字节相等保留（Anchor disc + borsh 在 wave 10 已 pin） |
| `SolanaRpcAccountFetcher` | impl `AccountFetcher`：`fetch_account` 走 `get_account_with_commitment`、`fetch_program_accounts_filter` 走 raw `client.send`，`classify_client_error` 把 `ClientError` 二分到 `RpcError::{Code, Transport}` |
| `SolanaTxBuilder` | impl `TxBuilder`：`get_latest_blockhash` → `Transaction::new_signed_with_payer` → `send_and_confirm_transaction`；payer 按值持有，`rotate_payer` 返回旧 keypair 让 ops 审计 |
| `decode_account_data_blob` | helper：`UiAccountData` JSON → `Vec<u8>` 经过 base64，拒 zstd/base58 |

11 个 solana 模块单测：

1. `pubkey_round_trip_is_byte_equal` —— PDA byte-level 锁
2. `account_meta_conversion_preserves_flags` —— 4 种 (signer × writable) 组合，治理 ix 签名校验前置守门
3. `dispatched_to_instruction_preserves_bytes` —— Anchor disc + borsh 字节相等（与 wave 10 self-test 联动）
4. `decode_account_data_blob_round_trip_base64` —— 32-byte fixture 编/解码相等
5. `decode_account_data_blob_rejects_unsupported_encoding` —— `base64+zstd` / `base58` 都拒
6. `decode_account_data_blob_rejects_malformed_shape` —— 非数组、单元素数组、首元素非 string 都拒
7. `classify_client_error_distinguishes_transport_vs_code` —— "RPC response error" 字符串契约（heuristic 二分逻辑文档锁）
8. `fetch_account_against_unreachable_url_yields_transport_error` —— 端到端打 `127.0.0.1:1`，断言 `RpcError::Transport`
9. `fetch_program_accounts_against_unreachable_url_yields_transport_error` —— 同上、`getProgramAccounts` 路径
10. `solana_tx_builder_submit_against_unreachable_url_returns_failed` —— `submit` 端到端拿 `ActionDispatchResult::Failed`，never panic（KeeperLoop 的容错前提）

### 12.1 Keeper bot 生产化（`crates/keeper-bot/src/{metrics,serve,run}.rs`）

把 wave 10 / 11 的 tick 状态机外面套了一层 production wrapper：

- **`KeeperMetrics`**（`metrics.rs`，~ 380 LOC）：`AtomicU64` / `AtomicI64` / `AtomicU8` 全 lock-free，13 个 metric（6 counter + 7 gauge）。`observe_tick(&TickReport, duration_ms)` 把 wave 10 的 `dispatched: Vec<(action, result)>` 拍扁成 submitted/failed/skipped 三个累计计数器；`render_prometheus()` 一次性产出 OpenMetrics 0.0.4 文本。`applied_vol = None` 编码为 `NaN`，scrape 端 PromQL 用 `< bool +Inf` 判 warming up。
- **`serve.rs`**（~ 220 LOC）：手写 HTTP/1.1 listener，**不引入** `axum / hyper / tokio` —— `set_nonblocking(true)` + 200ms accept poll 间隔 + `shutdown: Arc<AtomicBool>` 协作退出。两个路由：`GET /metrics`（200 OK + `text/plain; version=0.0.4; charset=utf-8`）/ `GET /healthz`（boot 前 503 / boot 后 200 JSON）。`render_response(&request_line, &metrics)` 是纯函数，可以直接喂给单元测试。
- **`run.rs`**（~ 240 LOC）：`run_loop` + `run_loop_with_factory` 两个变体，前者老式 `B: TxBuilder` by-value 复用，后者 `factory: FnMut() -> B` 适合每 tick 重新铸 builder（生产推荐路径，因为它能在 transient RPC 错误后用新 builder 重试）。核心是 `is_transient(&BotError)` 二分：`SnapshotError::{MarketPaused, SchemaVersionMismatch, MarketNotFound, SubPoolNotFound, Decode, Scheduler}` 永久 → 退出，`SnapshotError::Rpc(_)` 暂时 → backoff 续跑。这把 wave 9 的 governance 拒绝语义 1:1 镜像到守护进程层。

## 16 Wave 16 —— `tick_with_snap` + RPC reconcile + 真 leader heartbeat 链路 + 前端 leader banner + program-test 骨架

> **目标**：把 wave 15 留下的「双 RPC / 内存 mirror only」缺口闭起来；让 keeper-bot 真正在每个 tick 用一份 snapshot 完成 leader 判断 + 协议 tick；让 leader-lock 状态在前端面板上可见；为 SBF CI runner 准备好可即刻跑的 reject 矩阵。

### 16.1 单 snapshot tick：`KeeperBot::tick_with_snap`

**问题**：wave 15 `run_loop_with_leader` 先 prefetch 一次 snapshot 仅为了拿 `current_slot` 喂给 `LeaderPolicy::should_submit`，然后 `KeeperBot::tick` 内部又自己 refresh 一次。10² 副本 × 800ms tick = 250 RPS 浪费；按生产 4 副本 + 200ms tick 算也是 80 RPS 直接出血。

**做法**：

- 新增 `KeeperBot::tick_with_snap(&snap, ctx, builder, …)`，把 wave-9 / wave-10 tick 的「post-refresh pipeline」（vol 记录 → predictor 喂数据 → `scheduler.plan` → `RpcExecutor::dispatch`）抽成一个不依赖 fetcher 的纯函数。
- 旧 `tick(fetcher, …)` 改成 `tick_with_snap` 的 thin wrapper：先 `snap.refresh(fetcher, ctx, snapshot_cfg)` 再 forward。所有现有调用方（直接 `bot.tick`）一行代码不动；新调用方（leader-gated 主循环）可以按需自带 snapshot。
- `run_loop_with_leader` 改成单次 refresh + `tick_with_snap`，从此一个 tick 一次 RPC。
- 新加单测 `tick_with_snap_on_empty_snapshot_returns_zero_actions` 锁住空 snapshot 路径。e2e 既有的 `end_to_end_leader_gate_*` 走的就是新路径，全数仍 pass。

### 16.2 keeper-rpc 扩展：`leader_tx` 模块

**问题**：wave 15 把 4 条 keeper-leader ix 的 byte-exact 编码搬到 `keeper-decoder::ix`，但 keeper-bot 没有把它们接到现有的 `TxBuilder` 链路 —— `TxBuilder::submit(&DispatchedAction)` 是 `KeeperAction`-keyed 的（清算意义上的协议动作），而 keeper-leader ix 是「能否运行清算队列」的 gate。

**做法**：

- `crates/keeper-rpc/src/leader_tx.rs`（新文件）：
  - `LeaderInstruction { program_id, data, accounts }` 把 wave-15 编码器 + 账户布局打成可签结构。
  - `build_keeper_leader_heartbeat / acquire / release(program_id, market, lock_pda, keeper, args) -> LeaderInstruction`，三条 ix 的账户顺序与 Anchor handler `KeeperLeaderHeartbeat / KeeperLeaderRelease` 1:1 对应。
  - `KeeperLeaderTxBuilder` trait —— 与 `TxBuilder` 解耦的 submission seam；`MockKeeperLeaderTxBuilder` 给 host 测试用。
  - `fetch_keeper_leader_lock(fetcher, lock_pda) -> Result<KeeperLeaderLock, LeaderReconcileError>`：通过任意 `AccountFetcher` 抓 PDA 数据 → 验 8 字节 Anchor account discriminator (`account:KeeperLeaderLock`) → Borsh 解 49 字节 body → 返回 host state-machine 类型。失败码区分 `NotFound`（PDA 还没初始化）/ `Decode`（discriminator/Borsh 失败）/ `Rpc`（传输错误）。

### 16.3 `HostMirrorLeaderPolicy` 上链 reconcile + `run_loop_with_leader_and_rpc_reconcile`

**问题**：wave 15 的 `HostMirrorLeaderPolicy` 内部缓存了一份 `KeeperLeaderLock`，但只用 `current_slot` 推算 freshness；如果链上 holder 已经被另一个副本抢走，本地 mirror 看不到，副本会自以为还是 leader。

**做法**：

- `HostMirrorLeaderPolicy::reconcile(chain_lock)` 已经是 wave-15 既有 API；新增一个完整的 run-loop 把它真正用起来：
- `run_loop_with_leader_and_rpc_reconcile(bot, fetcher, ctx, factory, keeper_pk, …, policy: &HostMirrorLeaderPolicy, rpc_cfg: LeaderRpcReconcileConfig, leader_builder)`：
  - 单次 snapshot refresh / tick（继承 16.1 的优化）。
  - **Reconcile cadence**：每 `reconcile_every` ticks 调用 `fetch_keeper_leader_lock(fetcher, lock_pda)` → `policy.reconcile(chain_lock)`。`NotFound` 在日志里嗷一声（提示 ops 跑 `initialize_keeper_leader_lock`）；`Rpc / Decode` 错误**不致命**，只回退到上一份缓存 mirror。
  - **Heartbeat publish cadence**：每 `heartbeat_every` ticks（且**首次成为 leader 时立刻**）通过 `leader_builder.submit_leader_ix(build_keeper_leader_heartbeat(…, observed_slot))` 推一发链上 heartbeat，让 chain `current_leader / last_heartbeat_slot` 与 host mirror 对齐。失败也**不致命**，下一个 cadence 会重试。
  - Standby tick 不发 heartbeat（不是 leader 就不应该写自己进 lock）。
  - 默认 `reconcile_every = 20 ticks ≈ 16s @ 800ms`，`heartbeat_every = 5 ticks ≈ 4s`，远小于 wave-15 默认 `takeover_threshold_slots = 75 ≈ 30s`。
- 新增 2 条 `keeper-bot` e2e 测试：
  - `end_to_end_leader_rpc_reconcile_publishes_heartbeat_on_first_leader_tick` —— 链上 lock 已是自己持有，reconcile 拉回 → 第一拍 leader 立即发 heartbeat ix + 后续 tick 不超频发。
  - `end_to_end_leader_rpc_reconcile_yields_when_chain_says_other_holder` —— 本地 mirror 自我乐观地认为是 leader，reconcile 看到链上是别人 → 立刻翻 Standby + 不发 heartbeat。这是 wave-15「at-most-one holder」端到端拉链。

## 19 Wave 19 —— 多市场用户产品 + prober 守护进程 + env-var 化配置

### 19.3 关键设计

- **面板零侵入扩展**：原本想给 `TraderPanel` / `IndexerPanel` / `KeeperPanel` 都加 `activeSymbol` 参数，最终选择**保持面板 API 不变**——`selectActiveMarketSnapshot(feed, symbol)` 在 `App.tsx` 层把 `feed.indexer / feed.keeper` 改写成活跃市场的解码视图。这样新功能的代码集中在两个文件（`feed/selectMarket.ts` + `App.tsx`），三个面板一行不动。
- **市场选择持久化**：URL `?market=` 优先 → `localStorage` 兜底 → 第一个市场作默认。`useActiveMarket` 在 React 之外暴露 `resolveActiveMarket` 纯函数，单元测试覆盖所有优先级和"配置漂移后旧值不再有效"的边界。
- **共享 program-account 订阅**：wave-18 每市场 2 个 `accountSubscribe`（market + lock）。wave-19 加了**一个共享** `onProgramAccountChange(programId)` 流（仅当 caller 提供 `discriminators` 才订阅），所有市场共用 1 条流，避免 N 倍订阅放大 RPC 压力。子池路由通过 `OnchainSubPool.market.hex → marketHexToSymbol` O(1) 查表；干涸桶路由通过 `OnchainDormantBucket.sub_pool.hex → subPoolToMarket` 反查父子池所有市场。
- **prober 守护严格 fail-closed**：fetcher 返回 `Err` 时**整个周期不发布任何文件**，让 AlertManager 的 `for: 30s` 规则因 textfile gap 而触发，而不是误报"all-Pass"。生产 fetcher 必须在内部降级（缓存上次良好 ctx、把 `rpc_p95_ms` 拉高），让标准 21-check 电池能正确告警。
- **prober I/O-free trait 化**：`MarketFetcher` / `ProberClock` / `ProberSink` 三个 trait 让测试路径完全同步、零依赖（无 `tokio` / `solana-client` / 文件系统）。生产 binary 在 `main.rs` 实现 `StdClock` (`std::thread::sleep`) + `FileSink` (`std::fs::write`) + `DemoFetcher`（暂时跑 `healthy_demo_context()`，留待 SBF runner 上线后接 `solana-client`）。
- **Prom textfile 重打标签**：每市场的 `render_prometheus_textfile` 输出在合并到统一 textfile 时被 `relabel_with_market` 重写——所有 `mole_health_*` 指标自动加上 `market="<symbol>"` 标签，确保 node_exporter 的 textfile collector 识别为不同时间序列。
- **`${VAR}` 语法 Rust ↔ TS 字节级一致**：Rust 实现 `substitute_env_vars(input, &lookup)`，TS 实现 `substituteEnvVars(input, lookup)`。两边对相同输入产生相同输出（包括 `$$` 转义、变量名校验 `[A-Za-z_][A-Za-z0-9_]*`、未设置 / 空值 / 未闭合 / 空大括号 / 非法字符的错误信息）。这样 SOPS 解密产物可以直接喂给 Rust 或 TS 工具，不会因为细微差异导致 keeper-bot 与前端读到不同的市场配置。

### 20.2 核心交付

- **`PositionSummary.marketPdaHex?` + `selectActiveMarketSnapshot` 仓位过滤** —— wave-19 的 snapshot 改写器现在多了一步：用纯函数 `filterPositionsByMarket(positions, target)` 过滤 `feed.positions`。tagged 仓位若不属于当前市场则丢弃；untagged 仓位（wave-9..18 历史 mock）保留以维持向下兼容。`mockGenerator.buildPositions` 在 wave-20 把 `FEED_FAKE_MARKET_PDA.hex` 也写入每个仓位，所以 mock 路径与未来 live 路径行为一致。

- **`ops-toolkit::rpc_fetcher::RpcMarketFetcher`** —— 真 RPC fetcher 的 trait 抽象 + 默认实现。新 `RpcAccountSource` trait 包了 `getMultipleAccounts` + `getSlot`；`RpcMarketFetcher` 用一次 `getMultipleAccounts` 同时拉 `Market` + `KeeperLeaderLock` 两个 PDA，再用 `keeper_decoder` 把字节转成 `MarketFacts` / `LeaderLockFacts`。`getSlot` 用 `Instant::now()` 做单点延迟样本写到 `RpcFacts.primary_get_slot_p95_ms`。10 条主机测试覆盖 happy path / pause / freeze / schema mismatch / 缺失 PDA / 双向传输失败 / takeover threshold fallback / config defaults。
- **`ops-toolkit::cli_loader` —— `--markets-stdin` + `--env-from-file=PATH`** —— 新模块，把 `markets.toml` 来源（`File` / `Stdin`）+ env overlay（`Process` / `File` / `Inline`）做成 trait-free 纯函数。`parse_env_file` 接受 `KEY=VALUE`，支持 `export ` 前缀、可选双引号包裹、`#` 注释、空行；`extract_sources` 解析 CLI flag；`load_registry` 串起 markets 读取 + env overlay + `MarketRegistry::from_toml_str_with_env` 解析。所有 I/O 通过传入闭包注入，所以 16 条单元测试零文件系统依赖。`prober` 与 `scan` 两个子命令都接入新 flag，wave-19 的位置参数形态 100% 向后兼容。

### 20.3 关键设计决策

- **untagged-position fallback** —— 仓位 `marketPdaHex?` 是 *optional*，过滤器只在 tag 存在时校验。这让多市场过滤路径可以分阶段上线：先在 mock 与多市场 adapter 写 tag，再让 wave-21 单市场 websocketAdapter 解码 `OnchainPosition` 时把 `market` 字段写进来。期间已有的非多市场 fixture 与单元测试零修改。
- **`keeperState` 默认回落** —— wave-20 没有强行让 keeper-bot 发布 per-market metrics（那是 wave-21）。前端 wave-20 只把"如果有就用"的开关装好；当 `entry.keeperState === undefined`，沿用全局 `feed.keeper`，行为与 wave-19 完全一致。这避免了"前端先依赖未发布字段 → 报错 → 二次 ship"的 flaky deploy 周期。
- **`RpcAccountSource` trait** —— production 接 `solana_client::nonblocking::RpcClient`（沙箱外、需要 `solana-rpc` feature），unit test 接 `StubRpc`，两条路径都是同一个 `RpcMarketFetcher`。`Vec<Option<FetchedAccount>>` 直接对齐 `solana-client` 的 `getMultipleAccounts` 返回形态，避免再做一层适配。
- **批量 RTT** —— `RpcMarketFetcher::fetch` 一次拉两个 PDA，比 wave-19 假设的两次 `getAccountInfo` 节省 50% 网络往返。`getSlot` 之后再发，且只发一次（每次 `fetch` = 一次 `getSlot` + 一次 `getMultipleAccounts`）。10 个市场一次 cycle = 11 次 RTT。
- **env overlay 退化到进程环境** —— `cli_loader::load_registry` 当 overlay 文件 / inline 表里没有 key 时，回落到 `std::env::var`。这让"非密钥配置走 shell env，密钥走 SOPS overlay"的常见姿势可以无缝混用。
- **CLI 兼容** —— `extract_sources` 把 flag 收掉，剩下的位置参数原样返回；wave-19 的 `ops-toolkit prober ./markets.toml /a/b /c/d 10 0` 形态零变化。这避免破坏 systemd unit / k8s manifest。

## Wave 24 — 链上 ↔ indexer 本金/名义对账（prober 漂移检查 + 前端徽章）

> wave 23 把 open-interest 聚合前后端两半都交付了；wave 24 把它升级成完整性信号：第 22 个 prober 健康检查把 indexer 报告的 pool notional 与链上独立聚合（解码 `Position.notional` 求和）对账，前端在 open-interest 卡片加对账徽章。两者出现非平凡差距 = indexer 与链上对"live 敞口"的认知分歧，money-grade 告警。

### 24.1 问题陈述

* **open-interest 是单源观测**：wave-23 的聚合只在链上仓位*内部*自洽，没有与 indexer 侧报告的 pool 账目交叉验证；indexer bug / 漏 sync 会静默偏离链上真值。
* **`position_principal_drift` 缺位**：wave-23 changelog 与 wave-24 优先级都点名"把 `OpenInterestFacts` 接成 prober 漂移检查"，但 wave-23 只交付了聚合本体。

### 24.2 核心交付

1. **第 22 个健康检查 `position_principal_drift`（后端）**（`crates/ops-toolkit/src/checks.rs`）
   * 对比 `PoolFacts.onchain_position_notional_micro_usdc`（由 `apply_open_interest_to_pool` 从 `OpenInterestFacts::total_notional` 喂入）vs indexer 报告的 `total_notional_micro_usdc`。
   * drift = `|onchain − reported| / max(reported, 1)`：Pass < 0.5%、Warn(P2) < 2%、Critical(P1) ≥ 2%。
   * `onchain == 0`（本周期未跑 open-interest 扫描）→ **跳过**（Pass，`drift_enabled = 0`），与 wave-17 leader-lock 检查"源未启用即不误报"契约一致。
   * check battery 21 → **22**；`CHECK_NAMES` / `run_all_checks` / 计数断言全部同步。
   * `PoolFacts` 新增 `onchain_position_notional_micro_usdc`（默认 0）；6 处字面量同步，demo / healthy ctx 设为等于报告值以保证新检查干净通过。

2. **`apply_open_interest_to_pool` 折叠 helper（后端）**（`position_interest.rs`）—— 把 `OpenInterestFacts` 折进 `PoolFacts`，只写 on-chain 字段，reported 侧仍归 indexer fetch。

3. **前端 `reconcilePrincipal` + 对账徽章**（`frontend/src/feed/openInterest.ts` + `TraderPanel`）
   * `reconcilePrincipal(onchainCollateral, reportedCollateral)` → `ok / warn / critical / disabled`，阈值与后端一致。
   * TraderPanel open-interest 卡片新增 "Indexer reconciliation" 徽章：live 仓位抵押和 vs sub-pool 报告抵押和。
   * 前端对账走 collateral（principal）、后端走 notional —— 各自 apples-to-apples，意图相同（检测 indexer↔链上分歧）。

### 24.3 关键设计

* **跳过语义复用**：drift_enabled=0 的跳过路径与 leader-lock 的 disabled-source 模式统一，零误报。
* **对称漂移**：用 `abs_diff` / 绝对值，over-report 与 under-report 同幅落同一档。
* **新增字段默认 0 安全**：`PoolFacts` 派生 `Default`，未跑探针的部署天然跳过。

### 24.4 验证结果

## Wave 25 — open-interest 接入 live prober loop（per-market 漂移检查上线真数据）

> wave 24 加了 `position_principal_drift` 检查但没有任何东西喂它真数据，所以它永远*跳过*（on-chain notional 卡在 0）。wave 25 闭合这个回路：prober 现在每周期扫描 live 仓位并**按市场**对账，让完整性信号跑生产数据而非 fixture。

### 25.1 问题陈述

* **漂移检查空转**：wave-24 的 `position_principal_drift` 在 `onchain_position_notional == 0` 时跳过；live 路径从未填充该字段，检查从未真正运行。
* **多市场扫描需按市场切分**：程序级 `getProgramAccounts` 一次返回*所有*市场的仓位，但每个市场的漂移检查只能对账自己那一片。

### 25.2 核心交付

1. **`OpenInterestAugmentingFetcher<F, S>`（prober 装饰器）**（`crates/ops-toolkit/src/prober.rs`）
   * 包一个 base `MarketFetcher`（构建带 indexer 报告 pool 的 ctx）+ 一个 `AccountFetcher` source。
   * 每次 `fetch`：跑 per-market open-interest 扫描，经 `apply_open_interest_to_pool` 折进 `ctx.pool`，让 wave-24 漂移检查对账真仓位。
   * source 泛型：测试用 `MockAccountFetcher`，生产用 `SolanaRpcAccountFetcher`（`solana-rpc` feature 后，已实现 `AccountFetcher`）。
   * **生产构造入口**：`OpenInterestAugmentingFetcher::with_solana_rpc(base, rpc_url, commitment)`（`#[cfg(feature = "solana-rpc")]`）一行用真 RPC source 包 base fetcher，生产 prober 二进制无需手搓 `getProgramAccounts` 装配；`--features solana-rpc` 下编译期校验。
   * **扫描失败非致命**：base ctx 原样透传（on-chain notional 留 0 ⇒ 漂移检查跳过而非误报），瞬时 `getProgramAccounts` 抖动不会中断整个周期发布。

2. **per-market open-interest 扫描（后端）**（`position_interest.rs`）
   * `fetch_open_interest_for_market(fetcher, program_id, market_pda)` + `aggregate_open_interest_for_market(positions, market_pda)`：程序级扫描一次，按 `Position.market` 过滤到单市场，一个 RPC round-trip 喂多市场周期里的每个漂移检查。

3. **前端 `reconcileByMarket` + MarketSelector per-market 提示**（`frontend/src/feed/openInterest.ts` + `MarketSelector.tsx`）
   * `reconcileByMarket(positions, reportedByMarket)`：按 `marketPdaHex` 分组 live feed，逐市场对账链上抵押 vs indexer 报告抵押 —— 后端 per-market 检查的前端镜像。单边出现的市场也会发出（缺侧记 0）。
   * `MarketSelector` 每个 pill 显示该市场 live 仓位数徽章，tooltip 给出对账结论 + drift%；`reportedCollateralByMarket(view)` 从 `view.entries` 的 sub-pool 抵押求和。

### 25.3 关键设计

* **一次扫描喂多市场**：程序级扫描 + Rust 内按 market 过滤，避免每市场一次 RPC。
* **失败降级而非中断**：OI 扫描 `Err` 走跳过路径，复用 wave-24 "源未启用即不误报"契约；持续 RPC 故障由专门 RPC 检查暴露。
* **泛型 source**：`OpenInterestAugmentingFetcher` 对 `AccountFetcher` 泛型，测试零依赖、生产真 RPC，编译期保证接线非空。

## Wave 27 — live reported notional 从 Market PDA 提取（漂移检查比较两个独立链上真源）

> wave 26 把 open-interest 接进 prober，*on-chain* 侧（Σ`Position.notional`）已 live，但*报告*侧 `total_notional_micro_usdc` 仍来自 `RpcMarketFetcher` 的静态 `default_pool` 常量。wave 27 让报告侧也走真数据：直接从链上 `Market.current_total_notional`（程序自己的运行聚合）提取，使 `position_principal_drift` 比较**两个独立的链上真源**——程序聚合计数器 vs 实际持仓求和。两者背离即程序聚合与真实仓位不一致（bug/损坏），是比 wave-24（indexer vs 仓位）更强的完整性信号。

### 27.1 问题陈述

* **报告侧仍是 fixture**：`RpcMarketFetcher::build_ctx_from_accounts` 解码 Market PDA 取 paused/frozen/schema，但 `pool.total_notional_micro_usdc` 直接 clone `cfg.default_pool`，从未读链上聚合。
* **前端无程序聚合对账**：wave-24 前端只对 Σ持仓抵押 vs 子池报告抵押，未把 `Market` 程序自己的聚合计数器纳入对账。

### 27.2 核心交付

1. **`RpcMarketFetcher` 提取链上聚合 notional（后端）**（`crates/ops-toolkit/src/rpc_fetcher.rs`）
   * 新增 `RpcMarketFetcherConfig.lift_reported_notional_from_market: bool`（默认 `true`）。
   * `build_ctx_from_accounts` 解码 Market 成功且 flag 开时，`pool.total_notional_micro_usdc = m.current_total_notional`，覆盖静态默认；market 缺失/解码失败则保留 indexer 默认。
   * 4 个新 Rust 测试：提取生效、flag 关闭保留默认、market 缺失保留默认、默认 config 启用提取。

2. **前端 `reconcileProgramAggregate` + Program aggregate 对账行**（`frontend/src/feed/openInterest.ts` + `types.ts` + `decode.ts` + `mockGenerator.ts` + `panels/TraderPanel.tsx`）
   * `MarketSummary` 增可选 `currentTotalPrincipal` / `currentTotalNotional`（bigint），由 `onchainMarketToSummary` 真解码路径填充；mock 给 demo 一个健康可对账值。
   * `reconcileProgramAggregate(positions, marketAggregatePrincipal)`：Σ持仓 collateral vs `Market.current_total_principal`，复用 `reconcilePrincipal` 阈值（ok<0.5% / warn<2% / critical≥2%），无聚合或无持仓即 `disabled`。前端用 principal（两侧都可得），后端用 notional，各自 apples-to-apples。
   * `TraderPanel` open-interest 卡片新增 "Program aggregate" 对账行，与既有 "Indexer reconciliation" 并列。
   * 4 个新 vitest：对账通过、critical 背离、聚合缺失 disabled、无持仓 disabled。

### 27.3 关键设计

* **两个独立链上真源**：报告侧 = `Market` 程序聚合，on-chain 侧 = `getProgramAccounts` 持仓求和，互不依赖；背离即程序内部账与仓位实际不符。
* **可选字段不破坏 fixtures**：`MarketSummary` 新字段为可选，多处 fixture/fallback 字面量无需改动。
