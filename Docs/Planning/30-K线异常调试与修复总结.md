# 30 · `molefinance.net/app` K 线异常调试与修复总结

> 本文总结本轮 `https://molefinance.net/app` 交易页 K 线异常问题的排查、根因、修复、上线与收尾状态。问题经历了两个阶段的用户可见症状:先是“所有 K 线异常,只剩一根红柱子”,随后在第一次修复后又出现“k 线都没有了”。本轮目标是在不回退前序功能的前提下,恢复真实 live price 驱动的连续 candles。

---

## 1. 问题概览

### 1.1 用户反馈

本轮与 K 线相关的用户反馈按时间顺序如下:

1. `https://molefinance.net/app` 交易界面异常。
2. K 线“不是真实 K 线”,要求修复并重新部署 ECS。
3. “所有k线异常,只有一根红柱子,请修复”。
4. 第一次修复后,用户反馈“k线都没有了”。
5. 用户要求“中止调试,总结一下输出一份文档”。

### 1.2 现象拆分

- **现象 A: 单根异常红柱**
  - 图表区域不是连续 candles,而是被极端价格样本压扁后只剩一根夸张红柱。
- **现象 B: 图表空白**
  - 第一次修复错误价格来源后,图表没有继续收到可用于生成 candles 的 live 价格,导致完全无 K 线显示。

---

## 2. 调试策略

本轮严格按 evidence-driven debugging 方式推进,先采证据,再改业务逻辑:

1. 建立独立调试会话 `single-red-candle`。
2. 在 `PriceChart.tsx` 和 `TradeView.tsx` 先加运行时埋点,采集“图表收到的真实价格序列”和“价格来自哪个 market/base”。
3. 通过本地 build + preview + 浏览器复现实验,验证图表层是否只是呈现异常,还是上游喂价本身就有问题。
4. 沿 feed 聚合链路回溯到 `decode.ts` / `multiMarketAdapter.ts` / `websocketAdapter.ts`。
5. 对照链上 Rust 定义与 keeper 脚本确认字段语义和价格精度。

这一步的核心结论是:问题不是 `lightweight-charts` 绘图算法本身,而是上游价格样本错误和历史脏数据共同污染了 candle 序列。

---

## 3. 关键证据

### 3.1 运行时证据

在图表层埋点中,出现了同一根 candle 内极不合理的价格组合:

- `open ≈ 63028`
- `low = 0.1`
- `close = 0.1`

这说明图表并不是“只会画一根柱子”,而是收到了**数量级错误**的价格样本。价格一旦从几万美元跳到 `0.1`,图形就会被压缩成肉眼看上去近似一根红柱。

`TradeView` 埋点进一步证明,异常样本来自 live price 路径:

- `selectedBase = BTC`
- `onchainSymbol = BTC-20X`
- `marketSymbol = BTC-5X` 或 `BTC-20X`
- `livePriceUsd = 0.1`

### 3.2 静态代码证据

前端原本在 [decode.ts](file:///Users/surferboy/MoleOption/frontend/src/feed/decode.ts) 中把 `Market.price_tick` 直接映射成 `midPriceMicro`,本质上把“价格量化步长”误读成了“当前价格”。

而链上 `Rust` 定义明确表明:

- `price_tick` 是 oracle price 的量化步长
- `SubPool.last_price` 才是最近一次同步得到的真实 oracle price

同时,`keeper-devnet.mjs` 明确了链上真实价格精度是 `1e8`,前端展示精度是 `1e6`,因此还需要做一次 `1e8 -> 1e6` 的换算。

### 3.3 历史污染证据

图表历史存储在 `sessionStorage` 中。只要异常价格样本曾被写入本地历史,即使后来切回正确价格源,旧的脏 candle 仍会继续影响视觉表现。

---

## 4. 根因分析

本轮最终确认有两个连续根因:

### 4.1 根因一:错误价格源

前端把 `Market.price_tick` 当成 live market price 使用,导致:

- `midPriceMicro` 被填入错误语义的数据
- 不同 market/base 组合下可能出现 `0.1` 这类伪实时价格
- 图表使用该价格构造 candles,进而出现单根异常红柱

### 4.2 根因二:修复时价格源收得过严

第一次修复时,为了防止“错 market 喂价”,曾把图表价格源收紧为:

- 只有当前精确 `market.symbol === onchainSymbol` 的 market live price 才能喂给图表

这一改动虽然挡住了错误 market 的污染,但在多杠杆结构下也挡住了本来可复用的“同基础资产真实价格”,使图表经常拿不到 live 样本,从而出现空白。

### 4.3 放大因素:本地历史脏数据

即使 live 价格源修正后,如果 `sessionStorage` 中仍保留着异常 candle,图表仍可能继续异常。也就是说,错误价格源是主因,历史脏数据是放大器。

## 5. 修复措施

### 5.1 修复一:真实价格改为 `SubPool.last_price`

核心修复落在以下文件:

- [decode.ts](file:///Users/surferboy/MoleOption/frontend/src/feed/decode.ts)
- [multiMarketAdapter.ts](file:///Users/surferboy/MoleOption/frontend/src/feed/multiMarketAdapter.ts)
- [websocketAdapter.ts](file:///Users/surferboy/MoleOption/frontend/src/feed/websocketAdapter.ts)

具体做法:

1. 在 `decode.ts` 中新增:
   - `oraclePriceToMicro(price)`
   - `latestSubPoolPrice(subPools)`
2. `onchainMarketToSummary()` 不再把 `price_tick` 当现价:
   - `midPriceMicro` 默认置为 `0n`
3. 在单市场与多市场 adapter 聚合阶段:
   - 从最新 `SubPool.last_price` 读取真实价格
   - 以 `last_sync_slot` 最大者作为 freshest sample
   - 将价格从 `1e8` 转为前端 `1e6`
   - 用该值覆盖 `marketSummary.midPriceMicro` 和 `lastOracleSlot`

结果:

- 错把 `price_tick` 当 live price 的链路被彻底切断
- 图表上游价格恢复为真实 oracle 同步值

### 5.2 修复二:图表层过滤异常绝对价格

在 [PriceChart.tsx](file:///Users/surferboy/MoleOption/frontend/src/trade/PriceChart.tsx) 中新增:

- `isPlausibleAbsolutePrice(price, expectedPriceUsd)`
- `sanitizeHistory(raw, expectedPriceUsd)`

过滤规则:

- 价格必须是有限正数
- 若已知品种参考价,则只接受 `expected / 100 ~ expected * 100` 范围内的样本

应用点:

- 图表恢复历史时过滤脏 candle
- 实时生成 candle 时忽略异常价格样本

结果:

- 即使再次出现极端坏样本,也不会再直接写进当前 candle 或持久化历史

### 5.3 修复三:图表改为使用“同基础资产的真实 live 价”

在 [TradeView.tsx](file:///Users/surferboy/MoleOption/frontend/src/trade/TradeView.tsx) 中,最终逻辑调整为:

- 从当前 live feed 中提取“与所选 base 相同”的真实 live price
- 通过 `liveOverrides` 注入到 `useTickers`
- 只有 `priceIsLive === true` 时,才把价格传给 `PriceChart`

这意味着:

- 图表不再依赖 synthetic fallback 来生成“伪 K 线”
- 也不再过度收紧到“必须精确匹配当前杠杆 market”
- 同一基础资产不同杠杆 market 可共享同一份真实 live 价

结果:

- 修复了第一次修复后引入的“图表空白”
- 图表重新能够持续 update 当前 candle 并在跨桶时 append 新 candle

### 5.4 修复四:测试预期同步更新

在 [multiMarketAdapter.test.ts](file:///Users/surferboy/MoleOption/frontend/src/feed/multiMarketAdapter.test.ts) 中同步调整预期:

- 当 market 数据已到,但 sub-pool 真实价格尚未到时:
  - `midPriceMicro` 现在应为 `0n`

这和新语义保持一致,防止测试继续依赖错误的 `price_tick -> live price` 假设。

## 8. 结论

本轮 K 线问题并非图表库本身故障,而是**价格源语义误判 + 历史脏数据污染 + 第一次修复时喂价范围过严**共同导致:

1. 前端误把 `Market.price_tick` 当成当前价格,生成了 `0.1` 级别的伪 live 样本。
2. 这些异常样本写入 candle 历史后,图形被压缩成“单根红柱”。
3. 第一次修复为了止血收得过严,又导致多杠杆场景下图表拿不到价格,变成空白。
4. 最终通过“真实价格切到 `SubPool.last_price` + 脏样本过滤 + 同 base live price 驱动图表”的组合修复,恢复了真实 candles 的连续生成。

## 11. 接手最终修复:接入 TradingView 真实 K 线（本轮决定性方案）

### 11.1 为什么前两轮无法实现“真实 K 线”

前两轮修复都在“用链上 `SubPool.last_price` 实时累积 candle”这条路上打转，但这条路在本产品的现实约束下**根本做不出真实 K 线**：

1. **链上价本身不是真实行情。** devnet keeper 推给 mock-oracle 的价格，是围绕 catalog `basePriceUsd` 的合成随机游走，不是真实市场价。即使图表正确读到 `SubPool.last_price`，画出来也只是“围绕基准价的近似直线”。
2. **只用 live tick 累积 → 必然空白或稀疏。** 重写后的 `PriceChart` 取消了任何历史回填，只靠 8s 一次的 tick 累积。首屏必然空白；分钟级要等很久，日/周/月/年级别实际上永远凑不齐 candles。
3. **多市场只跟随一个 active market。** feed 一次只跟随一个链上 market，未选中的 base 拿不到 live price，图表自然空。
4. **VOL 被删除。** 重写过程中成交量直方图也一并被去掉，不满足“K 线要接入 VOL”的要求。

结论:继续在“链上价驱动自绘图表”上修，无论怎么改都到不了“真实 K 线”。

### 11.2 决定性方案:TradingView 高级图表 widget

参考 Hyperliquid / Drift 等主流交易前端的通行做法,图表层改为接入 **TradingView Advanced Real-Time Chart widget**,一次性拿到:

- **真实蜡烛**(真实历史 + 实时),覆盖 crypto / 指数 / 商品 / 外汇 / 美股 全资产类别
- **真实成交量 VOL**(`studies: ["STD;Volume"]`,内置)
- **全周期**:1m/5m/15m/30m/1h/4h/1D/1W/1M/1Y → TradingView interval
- **多语言**:i18n 语言 → TradingView locale
- 无需 API key、无 CORS 自管、无需自建历史存储

新增文件 `frontend/src/trade/TradingViewChart.tsx`:
- `TV_SYMBOL`:catalog base → TradingView ticker(crypto 走 `BINANCE:*USDT`,HYPE 走 `MEXC:HYPEUSDT`,指数 `SP:SPX`/`NASDAQ:NDX`,商品 `OANDA:XAUUSD`/`OANDA:XAGUSD`/`TVC:USOIL`,外汇 `FX:*`,美股 `NASDAQ:*` / `KRX:000660`)
- `tradingViewSymbol(base)`:解析不到的 base(仅 `SPCX` SpaceX 无公开行情)返回 `null`,回退到原 `PriceChart`(链上/合成)。

`TradeView.tsx`:有 TV 映射 → `TradingViewChart`,否则回退 `PriceChart`。29/30 个品种走真实图表。

### 11.3 顶部行情与图表一致:接入真实现货报价

接了真实图表后出现新问题:**顶部价是 mock 合成价(如 BTC ~\$60,104),图表是真实价(~\$64,256),数字对不上**。

新增 `frontend/src/markets/realQuotes.ts`:
- `useRealQuotes()` 轮询 Binance 公共 24h ticker(CORS 友好、无需 key),返回 crypto base → `{price, change24hPct, volume24hUsd}`。
- `useTickers(liveOverrides, realQuotes)` 中真实报价**优先级最高**,使顶部价 / 24h 涨跌 / 24h 量 / 品种选择器与图表一致。
- 非加密(指数/商品/外汇/美股)暂无免 key 的 CORS 行情源(Yahoo 被 CORS 拦截),保持合成报价,但**图表仍是真实的**。

> **排查记录**:首次接入时 Binance 批量请求整批失败,定位为 `HYPEUSDT` 在币安现货未上市,导致整批返回错误对象。已从 Binance 列表移除 HYPE(其余 14 个加密品种正常),并加 `Array.isArray` 防御;HYPE 图表改用 `MEXC:HYPEUSDT`。

### 11.4 关键正确性修复:显示价与“结算/包络价”解耦

把顶部价改为真实币安价后,引入一个**会破坏真实交易的隐患**:下单价格包络原本用 `priceUsd`(现在=真实价)构造,而链上 oracle 仍是 mock 合成价,两者偏差 > ±0.5% 包络容差 → **真实 open/close 会校验失败**。

修复:`TradeView.tsx` 新增 `settlementPriceUsd = livePriceUsd ?? priceUsd`(`livePriceUsd` 来自链上 `market.midPriceMicro`,即真实 oracle 价)。`submitOpen` / `submitClose` 的 `buildEnvelopeFromUsd` 改用 `settlementPriceUsd`:

- 顶部价 / 图表 / Est. Entry → 真实市场价(可读、与图一致)
- 下单包络 / 链上结算 → 链上 oracle 价(交易可正常校验通过)

图表说明文案同步更新为 `Live market chart · TradingView · settled at on-chain oracle`(6 语言)。

### 11.7 遗留与建议

1. **非加密真实报价**:指数/商品/外汇/美股的顶部数字仍是合成(图表已真实)。如需顶部数字也真实,建议接一个支持这些资产类别的带 key 行情源(如 Twelve Data 免费档)或服务端代理 Yahoo 以绕开 CORS。
2. **devnet 结算价 vs 真实价**:devnet 上 mock-oracle 与真实价天然不同,因此“图表真实价”和“链上结算价”会有差。主网上 oracle 跟随真实价后两者收敛。若希望 devnet 下也一致,可让 keeper 直接推真实价(如 Binance/Pyth)到 mock-oracle。
3. **ECS 复发**:本地 clean working copy(含本轮 TradingView 真实 K 线)需重新 build 并发布一版到 ECS,使线上与工作区一致。
