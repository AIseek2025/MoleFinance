# MoleOption 底层系统架构

## 1. 架构目标

底层架构必须同时满足：

- 资金安全：用户最大亏损不超过本金，协议不出现超额支付。
- 规则透明：关键状态链上可验证。
- 成本可控：开仓、平仓和重置交易在 Solana 上可执行。
- 可扩展：首个真实资金版本即支持 epoch、分片、批处理和懒结算，再逐步扩展到多标的、多杠杆市场。
- 可观测：链下索引能还原市场历史和用户权益变化。

## 2. 总体分层

```text
用户钱包
  |
前端应用
  |
Anchor Client / 索引 API
  |
Solana Program: MoleOption
  |
SPL Token 金库 + Pyth 预言机账户
```

## 3. 链上核心组件

### 3.1 GlobalConfig

全局配置账户。

职责：

- 记录协议管理员或多签。
- 记录手续费接收地址。
- 记录紧急暂停权限。
- 记录支持的抵押资产白名单。
- 记录程序版本。

### 3.2 Market

每个标的、抵押资产、杠杆倍数对应一个独立 Market。

示例：

```text
BTC/USD + USDC + 10x = Market A
BTC/USD + USDC + 20x = Market B
ETH/USD + USDC + 10x = Market C
```

职责：

- 存储市场参数。
- 存储最近重置价格。
- 存储市场聚合状态。
- 管理市场金库。
- 维护市场暂停状态。

### 3.3 Vault

SPL Token 金库账户，由 Market PDA 控制。

职责：

- 接收用户开仓本金。
- 接收手续费分流前资金。
- 支付平仓可提取金额。

约束：

- 用户不能直接从 Vault 提款。
- 任何提款必须经过合约结算。

### 3.4 Position

用户仓位账户。

职责：

- 记录开仓参数。
- 记录本金、不可逆锁定损失、已实现盈利余额或分配索引。
- 记录仓位状态。
- 支持平仓和历史查询。
- 即使当前权益为 0，也保留方向性敞口，等待后续价格反转和新亏损流动性带来的权益恢复。

### 3.5 Settlement State

结算聚合状态，生产级版本必须拆分为 Market、SettlementEpoch、SettlementShard 和 Position checkpoint。

职责：

- 记录多空总本金。
- 记录多空总名义价值。
- 记录多空可损失余额。
- 记录累计分配指数。
- 支持懒结算。

## 4. 链下组件

### 4.1 前端应用

职责：

- 用户交易入口。
- 钱包连接。
- 风险解释。
- 交易构造。
- 链上账户读取。
- 展示仓位状态。

### 4.2 索引服务

职责：

- 监听合约事件。
- 聚合市场数据。
- 提供仓位历史。
- 提供图表和统计数据。

索引服务不得决定资金结算结果。

### 4.3 Keeper

职责：

- 定期触发 reset。
- 在价格大幅变化时触发 reset。
- 在预言机异常时触发 pause 或报告。

Keeper 只是自动化调用者，不拥有特殊资金权限。

### 4.4 风险监控服务

职责：

- 监控预言机新鲜度。
- 监控市场不平衡。
- 监控 Vault 余额与账面权益差异。
- 触发告警。

## 5. 账户模型建议

### 5.1 PDA 设计

```text
GlobalConfig:
  seeds = ["global-config"]

Market:
  seeds = ["market", symbol, collateral_mint, leverage_bps]

Vault:
  seeds = ["vault", market]

Position:
  seeds = ["position", market, owner, position_id]
```

### 5.2 Position ID

Position ID 可以来自 Market 内递增 nonce，也可以由用户侧 nonce 传入。首版推荐使用 Market `next_position_id` 递增，避免用户重复创建冲突。

## 6. 核心数据流

### 6.1 初始化市场

```text
管理员 -> initialize_market
  -> 创建 Market
  -> 创建 Vault
  -> 绑定 Pyth price feed
  -> 设置杠杆、手续费、限额、暂停参数
```

### 6.2 开仓

```text
用户 -> open_position
  -> 校验市场状态
  -> 读取预言机价格
  -> 确认当前 epoch 状态
  -> 收取本金和手续费
  -> 创建 Position
  -> 更新 Market 聚合状态
```

### 6.3 Epoch 结算

```text
Keeper/任意调用者 -> open_epoch
  -> 读取有效价格并冻结 epoch 边界
Keeper/任意调用者 -> settle_shard_batch
  -> 分批统计 shard 盈亏和可兑现亏损
Keeper/任意调用者 -> finalize_epoch
  -> 汇总 shard，计算 ACTUAL_TRANSFER 和结算率
```

### 6.4 平仓

```text
用户 -> close_position
  -> 校验仓位所有权
  -> 读取预言机价格
  -> request_close_position
  -> 等待相关 epoch finalized
  -> apply_position_epochs
  -> claim_closed_position
  -> Vault 转账给用户
```

## 7. 关键架构难点

### 7.1 Solana 不适合单笔遍历所有仓位

白皮书中的全局重置算法描述为“遍历所有仓位”。在 Solana 上，这种方式无法扩展，因为交易账户数、计算单元和交易大小都有限。

正式实现应改为 epoch 化、分片聚合、批处理最终化和懒结算组合：

- Market 将时间切为连续 settlement epoch。
- 仓位按 Market、方向、价格区间或哈希分片进入 Shard。
- 每个 Shard 独立计算本分片 `profit_sum` 和 `realizable_loss_sum`。
- Market 汇总所有 Shard 后得到全局 `ACTUAL_TRANSFER`。
- 最终化 epoch 后写入全局结算率。
- 仓位在开仓、平仓、迁移或刷新时根据其尚未应用的 epoch 结算率懒更新。

直接遍历只能作为离线参考实现和测试 oracle，不应进入真实资金合约。

### 7.2 盈利权益的表示方式

协议需要明确盈利方的权益如何记账：

- 禁止方案：每次重置写入全市场仓位权益，需要遍历仓位，不可扩展。
- 推荐方案：按 epoch/shard 维护累计收益和亏损结算率，仓位懒结算。

简单方向级指数不足以精准处理“剩余本金上限”和“未提取盈利余额先扣减”的仓位级状态。生产级版本需要引入 Shard 累计索引、仓位 checkpoint 和可验证批处理，确保不遍历全市场也能精确结算每个仓位。

### 7.3 亏损锁定的表示方式

亏损方也需要按各自可兑现亏损金额占总可兑现亏损的比例承担 `ACTUAL_TRANSFER`。推荐使用累计亏损指数或按方向、价格区间的分摊模型。

需要保证：

- 亏损不会超过剩余本金。
- 关闭仓位前必须结算其历史指数。
- 新仓位只参与入场后的指数变化。

## 8. 市场隔离

每个 Market 独立拥有：

- Vault。
- Reset price。
- 聚合多空状态。
- 费用参数。
- 风控参数。
- 暂停状态。

不同 Market 不共享资金和清算结果。

## 9. 权限模型

### 9.1 用户权限

用户可：

- 开仓。
- 平仓自己的仓位。
- 触发公开重置。
- 查询账户。

用户不可：

- 修改市场参数。
- 提取他人资金。
- 操作 Vault。
- 修改预言机。

### 9.2 管理员权限

管理员可：

- 初始化市场。
- 调整市场限额。
- 暂停或恢复市场。
- 更新手续费参数。

管理员不可：

- 提取用户本金。
- 修改用户仓位结果。
- 绕过结算提款。

生产环境管理员应为多签。

## 10. 事件设计

合约应 emit：

- `MarketInitialized`
- `MarketPaused`
- `MarketResumed`
- `MarketReset`
- `PositionOpened`
- `PositionClosed`
- `FeesCollected`
- `OracleRejected`

事件应包含足够字段，使索引服务能重建用户体验数据。

## 11. 生产架构

- 单程序或多程序边界清晰。
- Market、SettlementEpoch、SettlementShard、Position、CloseRequest 分离。
- Keeper 网络处理 epoch 和 shard batch。
- 索引服务从测试网开始就是必需组件。
- 支持多个 Market，但每个 Market 独立扩容和限流。
- 多签治理。
- 多预言机。
- 批处理结算。
- 风险监控。
- 保险基金。
- 版本化 Market 配置。

## 12. 架构结论

MoleOption 的工程核心不是“写一个开平仓合约”，而是把全局配对清算改造成 Solana 可执行的聚合状态机。所有后续开发都应围绕三个不变量展开：

- Vault 余额覆盖所有可提款权益。
- 单仓亏损不超过本金。
- 盈利分配不超过亏损方可兑现损失。
