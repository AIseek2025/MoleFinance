# MoleOption 总设计师审计与漏洞清单

## 1. 总体判断

模型决策已锁定：

```text
方向权益池 + shares + recovery/dormant bucket
```

这是白皮书“多空互为对手、盈利来自亏损、仓位归零不死亡”原则在亿级用户和当前区块即时提款约束下的正式链上实现。逐仓比例模型保留为解释性模型和离线仿真对照，不作为生产实时结算路径。

## 2. 高危未决问题

### 2.1 模型语义锁定

已定方案：

- 生产链上实时结算采用方向权益池 shares 模型。
- 逐仓比例模型仅用于解释白皮书原则、离线仿真、测试 oracle。
- 产品文案和前端展示必须围绕 shares 权益池解释，不得让用户以为链上逐仓遍历配对。

剩余风险：

- 白皮书原始逐仓表达与生产 shares 实现之间需要写清楚映射关系。
- 所有公式、测试、前端说明必须统一到 shares 主模型。

### 2.2 shares 模型公平性未完全证明

风险：

- 新用户开仓时 shares 铸造可能稀释老用户。
- 池权益为 0 时 active/recovery shares 的 claim weight 需要严格定义。
- 不同开仓价用户在同方向池中的收益分配可能偏离用户直觉。

必须证明：

- 同一方向新仓加入不会稀释旧仓已存在权益。
- 零值仓位恢复权不会被新仓稀释。
- 新仓也不会被历史恢复仓无理由剥夺本金权益。

### 2.3 归零锚点 bucket 操作风险

风险：

- 价格剧烈穿越大量 `zero_price_tick`。
- 同一区块可能激活大量 dormant buckets。
- 如果需要遍历 bucket，O(1) 目标失效。

必须设计：

- 固定 tick 粒度。
- bucket 数量上限。
- 前缀/后缀聚合数据结构。
- 激活权重查询的账户上限。

### 2.4 同一 Market 写锁瓶颈

风险：

- O(1) 数学不等于无限并发。
- Solana 同一 Market/SubPool 账户写锁会限制吞吐。

必须设计：

- 子池分片。
- 路由规则。
- 子池间流动性展示。
- 防止用户选择对自己最有利的子池套利。

### 2.5 子池套利

风险：

- 如果同一标的/杠杆拆成多个子池，不同子池权益率不同。
- 用户可能选择更有利子池开仓或平仓。
- 大户可能操纵小子池。

缓解方向：

- 随机或哈希路由。
- 大额订单按比例拆分到多个子池。
- 子池容量和偏离度限制。
- 前端展示聚合市场，但链上明确子池归属。

### 2.6 预言机极端波动

风险：

- 单个区块价格跳变 50%、数倍或 -90%。
- Pyth 价格更新和用户交易排序可能被利用。
- stale price 或高 confidence 区间会造成错误提款。

必须设计：

- price age 上限。
- confidence 上限。
- 单次价格跳变保护。
- 分段价格同步或市场保护模式。
- 极端价格下是否允许开仓/平仓的规则。

### 2.7 交易排序和抢跑

风险：

- 用户在价格更新前后抢先开仓或平仓。
- Keeper 或交易 relay 排序影响权益。
- 同一区块内多笔交易顺序仍然会影响状态。

必须接受或缓解：

- Solana 交易有确定顺序，后交易基于前交易后的状态。
- 必须明确“交易顺序即结算顺序”是协议规则。
- 可用私有 relay、滑点/价格保护、交易过期 slot 降低风险。

### 2.8 舍入和 dust 攻击

风险：

- 大量小额仓位利用 floor 舍入累计 dust。
- 拆仓可能比合仓更有利。
- shares 铸造/销毁舍入可能被套利。

必须测试：

```text
settle(one large position) ~= settle(many split positions)
```

缓解：

- 最小本金。
- 统一向协议有利舍入。
- dust 留在 Vault。
- 定期审计 dust 规模。

### 2.9 费用和本金定义

风险：

- 用户理解的“最大亏损”是否包含手续费。
- shares 基于扣费前还是扣费后本金铸造。

必须锁定：

```text
position_principal = user_amount - open_fee
max_trading_loss = position_principal
total_user_outflow = user_amount
```

### 2.10 关闭 0 价值仓位

风险：

- 用户关闭 0 价值仓位后，是否永久放弃未来恢复权。
- 恶意前端可能诱导用户关闭 0 价值仓位。

必须设计：

- 关闭 0 价值仓位需要强提示。
- 一旦 `Closed`，不再参与恢复。
- 可考虑不允许关闭 0 价值 dormant 仓位，或要求二次确认。

### 2.11 账户存储成本

风险：

- 亿级 open 仓位意味着海量 Solana accounts。
- rent、索引、RPC 查询都会成为瓶颈。

必须设计：

- 仓位账户压缩。
- 批量仓位账户或压缩状态树。
- 用户历史仓位归档策略。
- 关闭仓位账户回收 rent。

### 2.12 升级与治理风险

风险：

- 合约一旦真实资金上线，核心状态迁移极难。
- 可升级权限本身是安全风险。

必须设计：

- 升级权限多签。
- 时间锁。
- 参数变更延迟。
- 状态版本化。
- 紧急暂停边界。

## 3. 生命周期推演缺口

### 3.1 开仓

必须确保：

- 先同步池价格，再铸造 shares。
- 开仓使用扣费后本金。
- 新仓从当前交易之后参与后续价格变动。
- 池权益为 0 时，新仓不稀释 dormant 恢复权。

### 3.2 持仓

必须确保：

- 权益不会为负。
- 休眠仓位不参与不利区间统计。
- 价格回到 zero_price 才重新激活。
- 未提取盈利余额后续可被亏损消耗。

### 3.3 平仓

必须确保：

- 先同步池价格。
- 再计算 shares 可领取金额。
- 先更新状态，再转账。
- 关闭后不再参与恢复。

### 3.4 极端行情

必须确保：

- 价格跳变不会造成溢出。
- 池权益不会变负。
- 大量 dormant bucket 激活不会超过账户限制。
- 交易排序规则清晰。

## 4. 建议立即新增的验证

1. shares 铸造/销毁不稀释测试。
2. active/recovery shares 公平分配测试。
3. dormant bucket 激活边界测试。
4. 价格跳变 10x、-90% 测试。
5. 拆仓/合仓等价测试。
6. 子池路由套利测试。
7. 同一区块交易排序模拟。
8. 0 价值仓位关闭风险测试。
9. 亿级账户索引和存储成本评估。
10. 预言机 stale/confidence 攻击测试。

## 5. 第二轮审计新发现的高危问题

第一轮审计聚焦于模型语义和单点机制公平性。第二轮审计沿“合约工程 + 黑客攻击面 + 治理 + 合规”四个维度补强，列出仍需在合约和文档中固化的高危问题。每条都必须在主网上线前完成修复或显式接受。

### 5.1 Position 账户的双套账本风险

风险：

- 旧设计的 Position 同时含 `locked_loss` 与 `realized_profit_balance`（逐仓模型字段），同时又引入 shares 字段（聚合模型字段）。如果两套字段同时实写，会出现“两套清算账本”，导致升级期或 bug 期资金路径冲突。

锁定方案：

- 生产合约 Position 字段以 shares 为唯一资金路径来源（见 `18-shares模型实现细则与边界条件.md`）。
- `locked_loss` 与 `realized_profit_balance` 不进入 Position 账户，仅由索引服务事件回放生成展示与审计字段。
- `07-智能合约设计.md` 必须同步删除生产合约 Position 中的 `locked_loss / realized_profit_balance` 字段，留作离线 oracle 与索引服务专用。

### 5.2 反向稀释与“近零池估值”陷阱

风险：

- 当 `pool_equity` 接近但未到 0，比率 `pool_equity / total_active_shares` 可能极小。新用户开仓 `shares_minted = principal * total_shares / pool_equity` 会导致新仓 shares 数量级远高于现有 active shares，老仓权重被反向稀释。

修复：

- 引入 `dilution_safety_bps`，当 `pool_equity * dilution_safety_bps < total_active_shares * 10_000` 时强制走“归零迁移”路径：
  - 把残余 `pool_equity` 计入 dust。
  - 把 `active_shares` 全量迁移为 `recovery_shares`，并写入 dormant bucket。
  - 从 0 池状态开始铸造新仓 shares。
- 详见 `18-shares模型实现细则与边界条件.md` §5.5。

### 5.3 shares_minted 为 0 的资金黑洞

风险：

- 极小 `principal` + 极大池估值 → `shares_minted = 0`（floor 舍入）。如果合约不校验，用户本金被吞但 shares 为 0，相当于资助协议。

修复：

- 合约强制 `shares_minted > 0`，否则交易失败 `SharesMintedTooSmall`。
- 配合 `min_margin` 限制最小本金。

### 5.4 sync_pool 路径相关性与抢跑

风险：

- 协议接受 `transfer = min(pnl, counterparty_equity)` 的路径相关性。攻击者可在大行情前抢先调用 `sync_pool` 把对手池抽干，使后来的合法用户在对手归零后无法获得 transfer。
- 长期未被同步的子池可被一次性收割大量 transfer。

修复：

- Keeper 必须周期性调用 `sync_pool`，最大空闲时间 `max_idle_slots`。
- 用户交易必须传入 `expected_price_min/max` 与 `expected_price_max_age_slots`，防止三明治抢跑。
- Solana 没有公开 mempool，但 leader / Jito relay 仍可以构造 MEV，必须接受这是协议内生风险并在风险揭示中告知。

### 5.5 子池路由不能用户选择

风险：

- 子池 pool_equity 比率不一致，自由选择子池则用户可套利。

修复：

- `sub_pool_id = hash(market || owner)[..] % sub_pool_count`，固定路由。
- 攻击者使用多钱包绕过路由仍合规，但每个钱包只能进入单一子池。
- 合并子池仅由治理 Level 5 流程执行，且必须经历存量自然清空期。

### 5.6 同钱包多仓位的“内部对冲刷量”

风险：

- 用户在同一 SubPool 同时开多和开空，方向中性但消耗预言机更新与 sync 资源，可能利用 dust 累积。

缓解：

- 协议层不禁止内部对冲（合法行为），但限制同钱包同 SubPool 仓位数量上限（建议 `max_positions_per_wallet_per_sub_pool = 8`）。
- 索引服务对“高频对冲刷量”进行风控告警。

### 5.7 dormant bucket 数量爆炸

风险：

- 攻击者制造大量微小仓位让其归零，注册大量 `zero_price_tick` bucket，把 segment tree 撑爆，使 `sync_pool` OOG。

缓解：

- `min_margin` 防止微小仓位。
- `max_dormant_bucket_count_per_direction` 限制单方向 bucket 数。
- 同 tick 的归零仓位必须合并入同一 bucket。
- 大跳变激活时退化为 lazy activation：sync 时只更新 `pending_activation_total`，用户领取时再 apply。

### 5.8 vault 余额与子池/dormant 的对账风险

风险：

- 多子池架构下，所有子池 pool_equity 共享同一 vault。Bug 可能导致多个子池同时认为自己持有同一笔资金，造成双重支付。

缓解：

- 强制不变量：
  ```text
  vault_balance >= sum(all sub_pool.long_pool_equity + short_pool_equity)
                + sum(market.long_dust + short_dust)
                + sum(pending_close_request_reserve)
  ```
- 每个状态修改指令结尾自动检查；失败立即 `InvariantViolation` 暂停。
- 索引服务对账作为冗余监控。

### 5.9 升级与治理的紧急响应路径

风险：

- 真实资金合约升级失败可能导致全协议失效，且 Solana 字节码本身不可“回滚”，只能通过部署旧版本字节码替代。
- 多签密钥失窃 = 资金路径风险。

缓解：

- 见 `16-合约升级与治理紧急响应.md`：权限分级、时间锁分级、版本化、迁移、回滚、演练。
- 在主网部署当天将 `ProgramUpgradeAuthority` 转给多签，禁止保留单签。

### 5.10 合规与监管缺口

风险：

- 即使合约去中心化，前端、域名、运营公司、关键开发者团队仍可能被监管追究。
- 被制裁地址访问公开前端是协议团队的合规事故。

缓解：

- 见 `17-合规与地理屏蔽.md`：地理屏蔽、钱包风控、风险揭示、KYC 入口、营销规范。
- 公开前端必须接入制裁名单与链上分析数据库。

### 5.11 PDA Frontrunning 与 position_id 抢占

风险：

- Position PDA seeds 含 `Market.next_position_id`。攻击者抢先开仓让 `next_position_id` 跳变，受害者交易因 PDA 期望不一致失败。
- 这是 DoS，不直接造成资金损失，但影响用户体验。

缓解：

- Position PDA 改用 `seeds = ["position", market, owner, owner_nonce]`，由用户提交 `owner_nonce` 决定，不依赖全局递增 id。
- 若仍使用全局递增，前端必须在交易内取最新 `next_position_id` 并接受失败重试。

### 5.12 Pyth confidence / publish_time 操纵

风险：

- Pyth 在网络拥堵时可能更新延迟。攻击者使用接近过期但仍有效的价格在快速行情中开仓。
- Pyth confidence 过窄时被错误信任。

缓解：

- 用户必须在交易里提交最新可得的 Pyth update。
- 合约对 confidence 同时检查“相对”与“绝对”值。
- 极端情况下进入 `Restricted` 模式禁止开仓。

### 5.13 fee_vault 与 Vault 的隔离

风险：

- 如果 fee_vault 与 Vault 共用 PDA 或权限可被治理任意切换，治理失误可能挪用用户资金。

缓解：

- `fee_vault` 与 `Vault` 必须是独立 PDA，独立 authority。
- 没有任何指令可以把 Vault 余额转入 fee_vault。
- fee_vault 提款指令限制接收方只能是 `fee_authority`，且金额不超过协议明确收入。

### 5.14 时间戳 vs slot 的依赖

风险：

- Solana `Clock::unix_timestamp` 可能被 leader 微调。任何资金敏感判断必须使用 slot 而非 timestamp。

修复：

- `last_sync_slot`, `closed_slot`, `open_request_slot` 等关键字段使用 slot。
- `opened_at`, `updated_at` 时间戳仅用于展示。

### 5.15 自动暂停的恢复路径滥用

风险：

- 自动暂停只能由特定条件触发，但恢复路径如果被治理过度授权，可能被滥用快速恢复。

修复：

- 恢复需要 `EmergencyAuthority` 与 `GlobalConfigAuthority` 双签。
- 恢复前合约必须自动检查不变量。
- 任何恢复事件必须公开。

### 5.16 索引服务被攻击造成展示失真

风险：

- 后端索引被入侵或 RPC 错误导致展示金额与链上不一致。用户根据展示决策可能造成损失。

修复：

- 用户最终领取金额必须由链上 `claim_*` 指令决定，不依赖前端展示。
- 索引展示标记为“估算”并显示链上对账状态。
- 多 RPC 冗余 + 自动 reconcile。

### 5.17 attacker 利用 dust 累积

风险：

- shares 铸造、销毁、recovery accrual 中所有 floor 舍入产生 dust。攻击者通过反复开平仓累积 dust。

修复：

- 所有 floor 都向协议方向舍入。
- dust 不进入 pool_equity，只入 `long_dust / short_dust` 字段。
- `harvest_dust` 指令把 dust 转给 fee_vault 或保险基金，不分配给用户。
- 测试覆盖 `settle(M) == settle(M/n + ... + M/n)` 在 dust 范围内一致。

## 6. 总设计师最终方案

最终方案：

```text
隔离 Market
  -> SubPool 多空权益池
  -> active_shares / recovery_shares
  -> zero_price_tick dormant bucket (前缀/后缀聚合)
  -> 子池分片提升并发, PDA 化路由
  -> 每笔交易 sync_pool 后即时开仓/平仓 (带价格保护)
  -> dilution_safety 自动归零迁移
  -> ProgramUpgradeAuthority 多签 + 时间锁
  -> 链下合规层 + 链上协议层分离
```

实现团队下一步不应再讨论是否采用 shares 模型，而应证明和实现以下内容：

1. shares 铸造/销毁不稀释，且无 0 shares 资金黑洞。
2. recovery/dormant bucket 激活正确，OOG 时退化为 lazy 路径。
3. 子池路由不可套利，PDA 固定。
4. 极端价格跳变下池权益不为负，超过阈值自动暂停。
5. 当前区块内每笔被纳入交易都能 O(1)/O(log n) 完成。
6. Vault 与 fee_vault 严格隔离，跨子池资金对账不变量持续成立。
7. ProgramUpgradeAuthority 多签 + 时间锁部署当天上线。
8. 合规层屏蔽与公开前端 ToS / 风险揭示完成。
9. 价格保护与 oracle 抢跑防御覆盖所有资金敏感指令。
10. 完整边界测试套件覆盖第 14 节攻击场景。
