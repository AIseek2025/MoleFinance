# MoleOption 前后端高并发架构

## 1. 目标

前端和后端必须支撑以下场景：

- 数百万到上亿用户访问交易页面。
- 数百万用户同时查看仓位权益。
- 极端行情下大量用户同时开仓、请求平仓和领取资金。
- 链上 epoch 结算存在 pending、finalized、restricted 等状态。
- 用户看到的展示值尽可能实时，但最终提款金额必须以链上 finalized 状态为准。

## 2. 核心原则

### 2.1 后端不决定资金

后端可以计算、缓存和展示估算值，但不能决定最终提款金额。最终资金转移必须由链上程序基于 finalized epoch 和仓位状态校验。

### 2.2 展示值分层

前端必须区分：

- 参考盈亏：基于当前价格和开仓参数的展示值。
- 估算权益：索引服务按最新已知状态估算。
- 已最终化权益：基于链上 finalized epoch 可验证。
- 可领取金额：链上 `claim_closed_position` 可支付金额。

### 2.3 极端行情降级

当请求量、RPC 延迟、epoch 未最终化或预言机异常时，系统应降级展示和限流，而不是返回错误的确定性金额。

## 3. 总体架构

```text
用户浏览器 / 钱包
  |
全球 CDN / WAF / DDoS 防护
  |
前端静态资源
  |
API Gateway
  |
读服务集群 ----- Redis / 内存缓存
  |
索引服务集群 ---- PostgreSQL / 分区表 / 时序库
  |
链上事件消费者 ---- RPC / Geyser / Yellowstone gRPC
  |
Solana Program
```

交易写入路径：

```text
前端 -> 钱包签名 -> RPC / Transaction Relay -> Solana
```

后端不代签用户交易。

## 4. 前端架构

### 4.1 静态化

交易前端应尽可能静态部署：

- CDN 缓存静态资源。
- 多区域部署。
- 前端版本 hash 固定。
- 配置通过只读 API 或链上配置加载。

### 4.2 钱包交易构造

前端负责构造：

- open_position。
- request_close_position。
- claim_closed_position。
- apply_position_epochs。

复杂交易可由后端提供 transaction template，但用户必须本地钱包签名。

### 4.3 实时状态

前端状态分层：

- WebSocket/SSE 获取市场聚合状态。
- REST 获取仓位列表和历史。
- 直接链上读取用于关键确认。
- 交易提交后订阅 signature 状态。

### 4.4 Pending 状态展示

当前 epoch 未 finalized 时：

- 显示“结算中”。
- 展示估算权益，但标记为非最终。
- 禁止把估算值称为可领取金额。
- 平仓按钮变为“提交平仓请求”。

## 5. 后端读服务

### 5.1 API 类型

推荐 API：

```text
GET /markets
GET /markets/:id
GET /markets/:id/epochs
GET /users/:wallet/positions
GET /positions/:id
GET /positions/:id/estimate
GET /positions/:id/claimable
GET /health/oracle
GET /health/settlement
```

### 5.2 缓存策略

缓存分级：

- 市场静态参数：长缓存。
- 市场聚合状态：1-3 秒缓存。
- 价格和预言机状态：亚秒或 1 秒缓存。
- 用户仓位：短缓存 + 主动失效。
- claimable：必须标记对应 finalized epoch。

### 5.3 热点保护

极端行情下：

- 热门市场页走 CDN edge cache。
- 仓位查询按 wallet 分片。
- API Gateway 做限流和排队。
- 大户和做市策略可用专用 read replica。

## 6. 索引服务

### 6.1 数据来源

索引服务消费：

- Program events。
- 账户变更。
- 交易日志。
- Pyth 价格更新。
- Keeper 状态。

推荐使用：

- Geyser plugin。
- Yellowstone gRPC。
- 多 RPC 提供商冗余。

### 6.2 数据表

核心表：

```text
markets
positions
settlement_epochs
settlement_shards
close_requests
position_epoch_applications
oracle_prices
program_events
```

大表必须按：

- market。
- epoch。
- wallet hash。
- time。

进行分区或分片。

### 6.3 一致性

索引服务必须记录：

```text
last_processed_slot
finalized_slot
reorg_status
source_rpc
```

前端必须知道数据延迟。

## 7. 实时推送

### 7.1 WebSocket / SSE

推送：

- 市场价格。
- epoch 状态。
- shard 进度。
- 用户交易确认。
- 仓位估算变化。

### 7.2 降级

WebSocket 压力过大时：

- 降级为轮询。
- 降低推送频率。
- 只推市场级状态，不推每个仓位。
- 用户仓位由客户端按需查询。

## 8. 后端计算边界

后端可以计算：

- 参考盈亏。
- 估算权益。
- epoch 进度。
- 风险提示。

后端不能决定：

- 最终可领取金额。
- 用户是否一定盈利。
- 绕过链上校验的权益。

## 9. 极端行情处理

当价格短时剧烈波动且用户大量请求：

### 9.1 前端

- 显示高波动提示。
- 显示预言机更新时间。
- 显示 epoch 是否 finalized。
- 禁止展示未最终化金额为“确定收益”。

### 9.2 后端

- 启用只读保护。
- 提高缓存命中。
- 限流非关键接口。
- 保留交易构造和状态查询接口优先级。

### 9.3 合约

- 使用 epoch 边界冻结结算输入。
- 未最终化 epoch 不允许确定性提款。
- 允许用户提交平仓请求，最终化后领取。

## 10. 容量规划

### 10.1 百万用户级

- 多区域 CDN。
- API 水平扩展。
- PostgreSQL read replica。
- Redis cluster。
- WebSocket 分区。

### 10.2 千万用户级

- 按 market 和 wallet hash 拆服务。
- Kafka / Redpanda 事件总线。
- ClickHouse 或时序库做分析。
- 用户仓位读模型独立存储。

### 10.3 亿级用户级

- 多区域主动主动。
- 索引服务按 market + epoch + shard 分区。
- 冷热数据分层。
- 专用交易 relay。
- 多 RPC 和自建 validator / geyser 基础设施。
- 降级模式常态化设计。

## 11. 监控指标

必须监控：

- API QPS。
- P95/P99 延迟。
- WebSocket 连接数。
- RPC 错误率。
- 索引延迟 slot。
- epoch finalize 延迟。
- shard backlog。
- close request backlog。
- 缓存命中率。
- 价格更新时间。
- 前端交易失败率。

## 12. 数据正确性验证

持续运行：

- 索引结果与链上账户抽样对账。
- 后端估算与链上 claimable 对比。
- epoch 聚合结果重放。
- shard 统计重算。
- Vault 余额与可提款总额检查。

## 13. 安全与滥用防护

- WAF。
- DDoS 防护。
- IP、钱包、设备指纹多维限流。
- API key 给机构用户。
- 读写接口分离。
- 防止恶意刷仓位查询。
- 防止 WebSocket 连接耗尽。

## 14. 结论

前后端不能用普通 DeFi 小项目架构支撑 MoleOption。真实资金上线前，索引、缓存、实时推送、降级、监控和链上最终校验必须作为同等重要的基础设施建设。用户看到的实时数据可以是估算，但领取资金必须只依赖链上 finalized 状态。
