# MoleOption 杠杆交易场与风控设计

## 1. 杠杆交易场定义

MoleOption 中，每个杠杆交易场是一个独立 Market：

```text
Market = symbol + collateral_mint + leverage
```

不同杠杆不共享资金池、不共享重置价格、不共享盈亏清算。

## 2. 为什么必须隔离杠杆

### 2.1 风险偏好不同

5x 用户选择的是较低波动放大，100x 用户选择的是极高波动放大。混合清算会让低杠杆用户被动承接高杠杆用户的盈亏需求。

### 2.2 名义价值差异

```text
notional = principal * leverage
```

同样 100 USDC 本金：

- 5x 名义价值为 500 USDC。
- 100x 名义价值为 10,000 USDC。

若混池，100x 仓位的 1% 价格波动就可能需要大量低杠杆亏损方来支付。

### 2.3 防止杠杆内卷

如果混池，高杠杆可能成为理性占优策略，因为它能用更大名义价值争夺同一池流动性。长期会挤出中低杠杆用户。

## 3. 首版杠杆选择

建议首个真实资金版本只开放一个杠杆：

- 首选：10x。
- 备选：20x。

不建议首版直接开放 50x 或 100x，因为：

- 对预言机更敏感。
- 对流动性更苛刻。
- 用户更难理解权益快速耗尽。
- 测试和风控压力更大。

## 4. 市场参数

每个 Market 应配置：

```text
symbol
collateral_mint
leverage_bps
min_margin
max_margin_per_position
max_total_principal
max_total_notional
open_fee_bps
oracle_price_feed
max_oracle_age_seconds
max_confidence_bps
max_price_move_bps_per_sync
price_tick
tick_aggregation_factor
max_dormant_bucket_count_per_direction
dilution_safety_bps
max_idle_slots
sub_pool_count
paused
frozen_new_position
```

### 4.1 首版参数建议初值

下表给出首版（BTC/USD 10x 单市场）建议初值，必须经经济仿真验证后再上链：

```text
min_margin                           = 10 USDC (10_000_000 raw)
max_margin_per_position              = 100_000 USDC
max_total_principal                  = 5_000_000 USDC (限额主网阶段)
max_total_notional                   = 50_000_000 USDC (= principal * 10x)
open_fee_bps                         = 5      (0.05%)
max_oracle_age_seconds               = 5      (Pyth 高频更新)
max_confidence_bps                   = 100    (1% confidence/price)
max_confidence_absolute              = 价格的 0.5%
max_price_move_bps_per_sync          = 2000   (20%)
price_tick                           = 1e6    (按 PRICE_SCALE = 1e8 折合 0.01 USD)
tick_aggregation_factor              = 100    (即 dormant bucket 粒度 = 1 USD)
max_dormant_bucket_count_per_direction = 4096
dilution_safety_bps                  = 1      (1 bps, 即 1e-4)
max_idle_slots                       = 30     (大约 12 秒)
sub_pool_count                       = 4      (首版限额下足够)
max_emergency_pause_slots            = 36000  (4 小时按 0.4s/slot)
recovery_check_window                = 300    (2 分钟)
```

参数变更必须按 `16-合约升级与治理紧急响应.md` 时间锁分级执行。

## 5. 仓位限制

### 5.1 最小本金

设置最小本金，避免 dust 仓位：

```text
min_margin = 10 USDC 或更高
```

### 5.2 单仓上限

限制单个仓位影响：

```text
max_margin_per_position <= market_total_principal * configured_ratio
```

首版可使用固定上限。

### 5.3 市场总上限

主网初期必须设置：

```text
max_total_principal
max_total_notional
```

随着审计、监控和流动性成熟逐步提高。

## 6. 多空不平衡风控

市场严重单边时，盈利兑现率可能恶化。可展示并限制：

```text
long_notional / short_notional
```

策略：

- 只展示不限制：首版推荐。
- 动态手续费：单边拥挤方向收费更高。
- 开仓限制：极端不平衡时限制继续开同方向。
- 激励反向仓位：降低反向方向手续费。

首版不建议引入复杂动态费率，先以清晰展示为主。

## 7. 预言机风控

### 7.1 新鲜度

价格更新时间必须满足：

```text
now - publish_time <= max_oracle_age_seconds
```

### 7.2 置信区间

Pyth 价格置信区间必须满足：

```text
confidence / price <= max_confidence_bps
```

### 7.3 单次跳变

每次 `sync_pool` 中，当前价格相对该子池 `last_price` 的变化不得超过阈值：

```text
abs(P_now - sub_pool.last_price) / sub_pool.last_price <= max_price_move_bps_per_sync
```

如果超过：

- 首版：拒绝同步并自动暂停该子池。
- 后续：进入分段同步或治理复核模式。

注意：长期未同步的子池可能因攻击者抢先调用 `sync_pool` 而触发该限制。Keeper 必须在 `max_idle_slots` 内主动同步所有子池，避免长期空闲。

### 7.4 标的校验

Market 必须绑定唯一 price feed，不能由用户任意传入替代价格账户。同时合约必须校验 `oracle_account.owner == PYTH_PROGRAM_ID`，防止伪造 oracle program 注入恶意价格。

### 7.5 抢跑与价格保护

为防止 MEV 三明治、抢跑 sync_pool 套利、利用过期价格开平仓，所有资金敏感指令必须接收用户提交的价格保护参数：

```text
expected_price_min: u64
expected_price_max: u64
expected_price_max_age_slots: u64
```

合约校验：

```text
sync_pool 后的 sub_pool.last_price ∈ [expected_price_min, expected_price_max]
sub_pool.last_sync_slot - oracle.publish_slot <= expected_price_max_age_slots
```

否则交易失败 `PriceProtectionFailed`。

### 7.6 抗 idle 子池收割

Keeper 网络必须在 `max_idle_slots` 内同步每个子池。如果某子池长期未被同步，先抵达的攻击者会一次性吸收大量价格变化收益。合约层防御：

- `max_idle_slots` 默认 30。
- 超时后任何指令调用前必须先调用 `sync_pool` 公开指令。
- 监控告警：子池 idle 超过阈值立即通知 keeper 网络。

## 8. 暂停机制

### 8.1 自动暂停条件

- 预言机价格过期。
- 预言机置信区间过大。
- 价格单次跳变超过阈值。
- Vault 余额与账面权益异常。
- 程序检测到不变量失败。

### 8.2 管理员暂停

多签可暂停市场。暂停后：

- 禁止开仓。
- 不允许 `claim_closed_position`。
- 允许查询与 `request_close_position` 排队。
- 允许 `apply_lazy_settlement`。
- 不允许 `EmergencyAuthority` 在暂停中直接转移用户资金。
- 见 `16-合约升级与治理紧急响应.md` 关于暂停权限分层。

### 8.3 恢复条件

恢复前必须确认：

- 预言机连续 `recovery_check_window` slot 健康。
- 市场账面状态一致（不变量自动检查通过）。
- 没有未处理的异常交易。
- `EmergencyAuthority` 与 `GlobalConfigAuthority` 双签。

### 8.4 frozen_new_position 单向冻结

某些情况下需要禁止开仓但允许平仓（例如即将下线的市场或子池）。引入 `frozen_new_position: bool` 标志：

- 设置后 `open_position` 拒绝。
- 平仓、claim、recovery 仍可继续。
- 用户有自然退出窗口，不强制立即关闭。

`frozen_new_position` 由 Level 1 时间锁治理调整。

## 9. 费用设计

### 9.1 开仓费

首版收取固定开仓费：

```text
open_fee = amount * open_fee_bps / 10_000
```

建议初值：

```text
open_fee_bps = 5
```

即 0.05%。

### 9.2 平仓费

首版可不收平仓费，减少用户理解成本。

### 9.3 协议收入分配

后续可分配：

- 协议金库。
- 保险基金。
- 回购销毁或治理激励。
- Keeper 激励。

首版可先全部进入 fee vault。

## 10. 保险基金

### 10.1 定位

保险基金不应承诺补足用户盈利。其用途应限定为：

- 处理舍入 dust。
- 应对极端 bug 或预言机事故后的治理补偿。
- 支付 keeper 或审计费用。

### 10.2 不建议用途

不建议用保险基金常态化补贴盈利兑现率，否则会改变 MoleOption 的零和透明模型。

## 11. 高杠杆开放策略

推荐顺序：

1. 10x。
2. 20x。
3. 5x。
4. 50x。
5. 100x。

高杠杆上线前必须满足：

- 预言机稳定性验证。
- 市场流动性充足。
- 经济仿真通过。
- 单次价格跳变处理成熟。
- 前端风险教育充分。

## 12. 风险展示指标

前端每个 Market 应展示：

- 多空总本金。
- 多空总名义价值。
- 当前盈利兑现率估算。
- 上次重置价格。
- 当前预言机价格。
- 预言机更新时间。
- 市场是否暂停。
- 最大单仓限制。
- 市场剩余容量。

## 13. 用户风险提示文案要点

必须表达：

- “永不爆仓”表示仓位不会被传统强平，不表示不会亏完本金。
- 最大损失为本金和相关费用。
- 显示盈利是参考值，实际可提取金额取决于市场对手盘亏损。
- 低流动性或严重单边市场可能导致盈利无法全额兑现。
- 高杠杆市场本金耗尽速度更快。
- 当前权益为 0 不代表仓位被删除；价格反转且出现新的对手盘亏损时，仓位可重新获得可提取权益。

## 14. 风控验收标准

- 预言机异常时不能执行错误价格清算。
- 市场暂停后不能继续开仓。
- 单仓和总市场限制有效。
- 高杠杆市场不会影响低杠杆市场。
- 手续费计算不影响最大亏损承诺的解释一致性。
- 价格保护参数生效：违反 `expected_price_min/max` 必须失败。
- 子池 `idle slot` 超过 `max_idle_slots` 时强制要求 sync_pool。
- `dilution_safety_bps` 触发归零迁移路径时不破坏 vault 资金。
- 暂停期间 `claim_closed_position` 与 `force_close_zero_value_position` 被正确禁用。
- shares_minted == 0 必须拒绝。
- 离线 oracle 重建结果与索引服务展示偏差在 dust 范围内。

## 15. 结论

MoleOption 的风控不是传统保证金率风控，而是“市场隔离 + 价格有效性 + 本金上限 + 盈利兑现约束”的组合。开发时必须把这些约束写入合约、测试和前端，而不是只停留在说明文档。
