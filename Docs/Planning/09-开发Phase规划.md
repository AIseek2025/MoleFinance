# MoleOption 开发 Phase 规划

## 1. 总体路线

MoleOption 应按“模型验证 -> 生产级可扩展合约 -> 前端闭环 -> 测试网压力验证 -> 安全审计 -> 限额主网 -> 扩展市场”的顺序推进。

不建议在模型未定稿前同时开发多市场、多杠杆、治理和复杂前端。

## 2. Phase 0：项目初始化与模型定稿

### 目标

把白皮书概念转化为可实现的数学规格。

### 交付物

- 最终数学模型说明。
- 锁定亏损不可逆与权益恢复机制规格。
- 清算公式测试样例。
- 离线仿真脚本雏形。
- 首个真实资金版本范围确认。

### 关键决策

- 如何用 `locked_loss` 和 `realized_profit_balance` 表达历史亏损恢复。
- 当前权益为 0 的仓位如何继续参与后续盈利分配。
- 首个标的和杠杆。
- 手续费是否从本金中扣除。

### 验收标准

- 至少 20 个手工案例可复现。
- 关键不变量写成测试。
- 团队能用一致语言解释“永不爆仓”和“盈利兑现率”。

## 3. Phase 1：离线清算引擎与仿真

### 目标

在不上链的情况下验证核心经济模型。

### 交付物

- Rust 或 Python 清算引擎。
- 随机价格路径模拟器。
- 多用户多空仓位模拟器。
- 白皮书 Alice/July/Bob 案例复现。
- 不变量测试。

### 工作内容

- 实现参考盈亏。
- 实现结算增量盈亏。
- 实现全局重置。
- 实现平仓权益计算。
- 实现价格反转后由新亏损方支付旧亏损仓位恢复权益。
- 实现手续费模型。
- 实现市场隔离模拟。

### 验收标准

- 10,000 条随机路径不变量通过。
- 用户最大亏损不超过本金。
- 盈利支付不超过亏损方可兑现损失。
- Alice/July/Bob/C 类价格反转恢复案例通过。
- 拆仓和合仓结果在舍入误差内一致。

## 4. Phase 2：Solana/Anchor 生产级合约

### 目标

实现单市场、单抵押资产、单杠杆，但合约架构从第一版就支持亿级仓位的 epoch 分片清算闭环。

### 交付物

- Anchor 项目。
- GlobalConfig、Market、Position、SettlementEpoch、SettlementShard、CloseRequest 账户。
- Market Vault。
- Pyth 价格校验。
- open_position。
- open_epoch。
- settle_shard_batch。
- finalize_epoch。
- apply_position_epochs。
- request_close_position。
- claim_closed_position。
- 当前权益为 0 的 open 仓位继续参与后续重置。
- pause_market / resume_market。
- Anchor 集成测试。

### 范围限制

- 单市场。
- 固定杠杆。
- 固定抵押资产。
- 首版可限制 TVL、市场数量、单仓规模和开放用户规模。
- 不允许使用需要全市场遍历的链上重置。

### 验收标准

- 本地 validator 端到端交易成功。
- 非 owner 不能平仓。
- 预言机无效时无法结算。
- Vault 余额和账面权益一致。
- 分片 epoch 结算结果与离线朴素遍历 oracle 等价。
- 未最终化 epoch 下不会确定性超额提款。
- 所有核心错误码测试通过。

## 5. Phase 3：生产级前端闭环

### 目标

提供用户可操作的测试网界面。

### 交付物

- Next.js 前端。
- 钱包连接。
- 市场页。
- 开仓表单。
- 仓位列表。
- 平仓流程。
- 风险说明组件。
- Devnet 配置。

### 页面

- 首页。
- 交易页。
- 仓位页。
- 风险说明页。

### 验收标准

- 用户可通过钱包完成开仓和平仓。
- UI 清晰展示参考盈亏和当前权益。
- 所有交易前有风险确认。
- 市场暂停时 UI 禁止开仓。

## 6. Phase 4：测试网公开验证

### 目标

在 Devnet 或专用测试环境中收集真实交互数据。

### 交付物

- Devnet 部署。
- 测试代币领取或说明。
- Keeper 网络雏形。
- 基础索引服务。
- 市场统计页。
- 用户反馈表。

### 测试重点

- 高频开平仓。
- 多用户同时交易。
- 预言机延迟。
- 市场单边。
- 仓位当前权益归零后随价格反转恢复。
- 前端风险理解。

### 验收标准

- 连续运行 7 天无资金不变量错误。
- 至少完成 1,000 次开平仓测试交易。
- 收集并修复关键 UX 问题。
- 监控能发现异常预言机和失败交易。

## 7. Phase 5：高并发清算压力验证

### 目标

验证 Phase 2 已实现的 epoch 分片清算架构在百万到亿级仓位规模下仍然正确、可恢复、可运维。

### 交付物

- 高并发 epoch 分片清算验证报告。
- 与离线遍历 oracle 的结果对比测试。
- 极端仓位数量性能测试。

### 研究方向

- Settlement epoch。
- Shard 分片统计。
- 累计盈利指数。
- 累计亏损指数。
- 仓位懒结算。
- 分桶清算。
- 批处理重置。

### 验收标准

- 大量仓位下交易计算成本可控。
- 与遍历模型在测试场景下结果一致或差异可解释。
- 模拟 100 万、1,000 万、1 亿仓位时，单笔链上交易不需要遍历全市场。
- 不变量继续成立。

## 8. Phase 6：安全审计准备

### 目标

让外部审计团队能够有效审查协议。

### 交付物

- 合约冻结版本。
- 架构文档。
- 数学规格。
- 权限图。
- 账户图。
- 测试报告。
- 已知风险说明。
- 审计范围说明。

### 验收标准

- 内部安全清单全部通过。
- 经济仿真报告完成。
- 无高危已知问题。
- 管理员权限迁移到多签方案。

## 9. Phase 7：限额主网上线

### 目标

以有限风险启动真实资金市场。

### 上线策略

- 只开放 1 个市场。
- 设置低 TVL 上限。
- 设置单仓上限。
- 设置高频监控。
- 保留紧急暂停。
- 明确 Beta 风险。

### 验收标准

- 主网部署校验完成。
- 多签权限生效。
- 监控和告警运行。
- 前端指向主网配置。
- 用户文档和免责声明发布。

## 10. Phase 8：市场扩展

### 目标

扩展到多个标的和杠杆层级。

### 扩展顺序

1. 同标的新增 5x、20x。
2. 新增 ETH/USD 或 SOL/USD。
3. 新增 50x。
4. 在流动性充分后研究 100x。

### 验收标准

- 每个 Market 独立清算。
- 市场间无资金污染。
- 前端能清楚展示不同市场风险。

## 11. Phase 9：治理与生态

### 目标

逐步把参数管理、市场新增和费用分配交给治理。

### 可能交付物

- 治理合约或治理集成。
- 参数投票。
- 保险基金规则。
- 开发者 API。
- 策略 SDK。

## 12. 优先级总表

```text
P0:
  数学模型定稿
  离线仿真
  合约资金安全
  预言机校验

P1:
  前端交易闭环
  Keeper
  索引服务
  测试网验证

P2:
  可扩展清算
  多市场
  多预言机
  保险基金

P3:
  治理
  策略生态
  高级风控
```

## 13. 近期推荐下一步

下一步应执行 Phase 0 和 Phase 1：

1. 固化 `locked_loss` 不可逆、`realized_profit_balance` 恢复权益的生产级模型。
2. 编写离线清算引擎。
3. 用属性测试证明资金守恒和价格反转恢复。
4. 再启动 Anchor 合约。

这样可以避免在数学模型还未完全稳定时把复杂性固化到链上账户结构中。
