# MoleOption 测试验证与安全审计

## 1. 测试目标

MoleOption 的测试目标不是只证明代码能运行，而是证明协议在随机价格路径、极端市场、恶意输入和异常预言机下仍满足资金安全不变量。

## 2. 测试分层

```text
数学单元测试
  -> 属性测试
  -> 离线经济仿真
  -> Anchor 集成测试
  -> 前端端到端测试
  -> 测试网压力测试
  -> 外部安全审计
```

## 3. 核心不变量

所有测试围绕以下不变量展开：

```text
每个仓位:
  0 <= locked_loss <= principal
  withdrawable >= 0

每次重置:
  actual_transfer <= total_profit
  actual_transfer <= total_realizable_loss
  sum(realized_profit) <= sum(realized_loss)

每个市场:
  total_withdrawable <= vault_balance
  market leverage 不变
  market collateral 不变

每个用户:
  loss <= principal + explicit_fees
```

## 4. 数学单元测试

### 4.1 PnL 测试

覆盖：

- 多头价格上涨。
- 多头价格下跌。
- 空头价格上涨。
- 空头价格下跌。
- 价格不变。
- 高杠杆。
- 极小本金。
- 大额本金。

### 4.2 可兑现亏损测试

覆盖：

- 亏损小于剩余本金。
- 亏损等于剩余本金。
- 亏损大于剩余本金。
- 已锁定损失为 0。
- 已锁定损失接近本金。

### 4.3 配对清算测试

场景：

- 盈利等于亏损。
- 盈利大于亏损。
- 盈利小于亏损。
- 两人场景：亏损方账面亏损大于盈利方账面盈利时，实际转移金额等于盈利方账面盈利。
- 两人场景：亏损方账面亏损小于盈利方账面盈利时，实际转移金额等于亏损方可兑现亏损。
- 无盈利。
- 无亏损。
- 多个盈利方按各自账面盈利占比分配 `ACTUAL_TRANSFER`。
- 多个亏损方按各自可兑现亏损占比承担 `ACTUAL_TRANSFER`。
- 多盈利方和多亏损方同时存在时，总支付金额和总锁定损失在舍入误差内相等。

## 5. 属性测试

使用随机生成器生成：

- 用户数量。
- 仓位本金。
- 方向。
- 杠杆。
- 价格路径。
- 开仓和平仓时刻。
- 重置频率。
- epoch 切分。
- shard 分片顺序。
- 懒结算延迟。

每次随机执行后验证核心不变量。

### 5.1 示例属性

```text
For any valid market state and valid price:
  after reset_market(state, price):
    no position has locked_loss > principal
    total realized profit <= total realized loss
```

### 5.2 高并发等价属性

验证分片 epoch 清算与朴素全量遍历清算在相同输入下等价：

```text
naive_global_settlement(state, price)
  ~= sharded_epoch_settlement(state, price)
```

差异只能来自明确定义的整数舍入 dust。

### 5.3 拆仓属性

验证一个大仓位拆成多个小仓位不会产生系统性优势：

```text
settle(M) ~= settle(M/2 + M/2)
```

允许有限 dust 差异。

## 10. 经济攻击测试

### 10.1 重置时机攻击

测试用户是否能通过开仓前后操纵重置顺序获利。

预期：

- 开仓和平仓前强制 reset。
- 用户不能跳过应承担的价格区间。

### 10.2 拆仓套利

测试大量拆分仓位是否能利用舍入获利。

缓解：

- 最小本金。
- dust 留在 Vault。
- 舍入策略统一。

### 10.3 预言机延迟套利

测试用户使用旧价格开仓或平仓。

缓解：

- 最大价格年龄。
- 必要时要求最新 Pyth update。

### 10.4 零权益仓位恢复攻击

当前权益为 0 的仓位必须继续参与后续方向性分配，这是产品核心机制。测试重点不是禁止它恢复，而是确认恢复资金完全来自新的亏损方可兑现损失。

必须验证：

- `locked_loss` 不会减少。
- `realized_profit_balance` 的增长不超过本轮亏损方可兑现损失。
- 没有新亏损方时，零权益仓位不会凭空恢复权益。
- 拆仓不能放大零权益恢复收益。

### 10.5 Shares 与休眠桶攻击

方向权益池 shares 模型必须额外测试：

- 新仓 shares 铸造不会稀释老仓权益。
- 池权益为 0 时，新仓不会稀释 dormant/recovery shares 的未来恢复权。
- active/recovery shares 分配没有时间先后优先级。
- dormant bucket 在价格刚好等于、略高于、略低于 `zero_price_tick` 时激活正确。
- 大量 dormant bucket 同时跨越时，不会超过链上账户和计算限制（退化为 lazy activation 路径）。
- 子池之间权益率差异不会被用户路由套利（PDA 路由固定）。
- 同一区块内不同交易顺序下，结果符合“交易顺序即结算顺序”的协议规则。
- shares_minted == 0 必须拒绝。
- 接近零池估值时触发 `dilution_safety` 自动归零迁移，不会出现反向稀释。
- dust 不可被用户直接提取，只能通过 `harvest_dust` 进入 fee_vault。

### 10.6 sync_pool 路径相关性测试

- `P0->P1->P2` 与 `P0->P2` 两条路径下，pool_equity / shares 结果可能不同，但都必须满足核心不变量（pool_equity >= 0、active_shares 与 pool_equity 同时为 0 的语义、vault 余额覆盖）。
- 长期 idle 子池被攻击者抢先 sync 时，transfer 不会让协议产生超额支付，且 keeper 触发 sync 与攻击者触发 sync 在协议层等价。

### 10.7 抢跑与价格保护测试

- `expected_price_min/max` 区间外价格必须导致交易失败 `PriceProtectionFailed`。
- `expected_price_max_age_slots` 超过时拒绝。
- 三明治抢跑场景中，受害者交易在价格保护内仍能正常执行。
- Pyth 程序 ID 校验：传入非 Pyth program 的 oracle account 必须失败。

### 10.8 子池路由与多钱包攻击

- 同钱包多次开仓必须路由至同一子池。
- 多钱包绕过路由不会带来跨钱包的资金优势（每个子池内部仍按 LP 公平）。
- 跨子池数据隔离：一个子池数据损坏不会影响其他子池资金计算。

### 10.9 Vault 与 fee_vault 隔离测试

- 任何指令都不能把 vault 余额转入 fee_vault。
- fee_vault 提款只能由 fee_authority 触发，且金额受协议明确收入约束。
- vault_balance 不变量自动检查覆盖所有子池累计 pool_equity。

### 10.10 升级与治理测试

- 多签 propose -> queue -> wait timelock -> execute 流程在测试网完整跑通。
- 紧急暂停下 force_close 与 request_close_position 的可用性符合预期。
- Schema 版本不一致的 Position 调用必须失败 `SchemaVersionMismatch`。
- 迁移指令幂等：重复调用不破坏状态。
- Level 5 状态结构迁移在退出窗口结束后才允许执行。

### 10.11 合规与前端屏蔽测试

- 受限地区 IP 访问主前端时必须拒绝。
- 制裁地址命中风控时拒绝构造交易。
- 风险揭示书签字记录正确保存。
- 第三方前端使用协议时不影响主前端合规边界。

### 10.12 极端账户压力测试

- 攻击者制造大量 micro 仓位让其归零，dormant_bucket_count_per_direction 受限。
- 同钱包同子池仓位数受 `max_positions_per_wallet_per_sub_pool` 限制。
- 价格剧烈穿越多个 bucket 时，sync_pool 不超出 Solana CU/账户上限；超出时退化为 lazy 路径并要求用户在 claim 前 apply。

### 10.13 PDA 抢占测试

- Position PDA 使用 `owner_nonce` 而非全局 next_position_id；攻击者抢占其他用户 position_id 路径不会生效。
- nonce 冲突时返回 `NonceConflict`，前端可以使用下一个 nonce 重试。

## 11. 安全审计范围

### 11.1 合约审计

审计重点：

- 账户权限。
- PDA seeds。
- Token 转账。
- 重复提款。
- 数学溢出。
- 舍入错误。
- 预言机校验。
- 市场暂停。
- 状态不变量。

### 11.2 经济模型审计

审计重点：

- 盈利分配是否守恒。
- 用户亏损是否封顶。
- 是否存在拆仓套利。
- 是否存在大户操纵重置收益。
- 是否存在低流动性误导。

### 11.3 前端风险审计

审计重点：

- 是否误导用户理解收益。
- 是否隐藏盈利兑现风险。
- 是否清楚展示最大亏损。
- 是否正确处理失败交易。

## 16. 结论

MoleOption 的主要风险来自数学模型和清算状态，而不仅是普通智能合约漏洞。测试与审计必须覆盖“代码安全”和“经济安全”两条线，否则不能进入真实资金环境。
