# 当前区块即时清算与简化模型评估

## 1. 结论

产品要求是：

```text
用户在当前区块提交平仓，必须在当前区块得到精确可领取金额
```

因此前文的“异步 epoch 分片最终化”方案不作为生产实时清算主路径。它能保证最终化后精确，但不能保证当前区块内完成亿级仓位的全市场清结算。

要满足当前区块即时精确清算，智能合约每次平仓必须只读取少量账户并完成 O(1) 或 O(log n) 计算。任何需要遍历数百万、数千万、上亿仓位的模型都不可能在 Solana 当前区块内完成。

## 2. 零价值仓位规则

MoleOption 不设置仓位死亡状态。

仓位状态应只有：

```text
Open
CloseRequested
Closed
```

含义：

- `Open`：仓位仍然有效，保留方向、杠杆、名义价值和未来恢复资格。
- `CloseRequested`：用户已申请关闭，等待可领取金额确定或领取。
- `Closed`：用户已最终退出，仓位不再参与后续清算。

当仓位浮亏达到 -100%、-200% 甚至更多时：

```text
position_equity = max(0, computed_equity)
```

仓位价值记为 0，不记为负数，也不删除仓位。

该规则的结果是：系统可能长期保留数千万甚至上亿个 `Open` 且当前价值为 0 的仓位。因此合约设计不能依赖“删除归零仓位来降低计算量”。

## 3. 现有逐仓精确模型的瓶颈

逐仓模型需要知道每个仓位：

- 方向。
- 杠杆。
- 本金。
- 已锁定不可逆亏损。
- 未提取已实现收益。
- 最近结算价格。
- 是否当前权益为 0。
- 后续是否重新恢复权益。

在每次价格变化时，系统需要判断每个仓位本轮：

- 是盈利方还是亏损方。
- 盈利 claim 是多少。
- 可兑现亏损是多少。
- 是否触底为 0。
- 是否先扣减未提取收益。

如果市场有 1 亿个 open 仓位，当前区块内不可能逐个读取并计算这些状态。

## 4. “当前区块必须结算完”意味着什么

区块链交易的确要求当前交易读写的账户状态在当前区块内确定。用户平仓交易如果要当前区块领取资金，合约必须在该交易内完成所需计算。

因此有两条路：

1. 保持逐仓精确模型：不能保证亿级仓位当前区块即时平仓，只能异步最终化。
2. 修改为聚合数学模型：让平仓金额只依赖全局聚合指数和用户本地 checkpoint，从而当前交易 O(1) 计算。

如果产品强要求当前区块即时平仓，必须选择第二条路。

## 5. 正式主模型：聚合权益池指数

为了在 Solana 当前交易内完成精确平仓提款，MoleOption 正式采用聚合权益池指数模型：每个方向的仓位共享同一个方向权益池，通过 shares 表示用户在该池中的权益权重。

每个 Market 维护：

```text
long_pool_equity
short_pool_equity
long_total_shares
short_total_shares
long_notional
short_notional
last_price
```

同时必须维护零值休眠仓位的聚合桶：

```text
long_dormant_buckets[zero_price_tick]
short_dormant_buckets[zero_price_tick]
```

这些桶记录已经归零但未死亡的仓位权重，用于价格回到归零锚点时重新纳入统计。

关键原则：

- 每笔会改变资金状态的交易，必须先把所属池从 `last_price` 同步到当前有效预言机价格。
- 同步只更新池级状态，不遍历用户仓位。
- 同一笔交易随后再执行开仓、平仓或领取。
- 交易结束时，池状态已经是当前区块内该交易执行后的确定状态。

每个用户仓位不直接记录复杂历史逐笔亏损，而记录：

```text
direction
shares
entry_price
leverage
last_index_checkpoint
```

## 6. 区块价格变动时的 O(1) 清算

当价格从 `P_old` 到 `P_new`：

```text
long_pnl = long_notional * (P_new - P_old) / P_old
short_pnl = -short_notional * (P_new - P_old) / P_old
```

如果多头盈利、空头亏损：

```text
transfer = min(long_pnl, short_pool_equity)
long_pool_equity += transfer
short_pool_equity -= transfer
```

如果空头盈利、多头亏损：

```text
transfer = min(short_pnl, long_pool_equity)
short_pool_equity += transfer
long_pool_equity -= transfer
```

池子权益最低为 0，不会为负。

池同步函数：

```text
sync_pool(P_now):
  if P_now == last_price:
    return

  long_pnl = long_notional * (P_now - last_price) / last_price
  short_pnl = -short_notional * (P_now - last_price) / last_price

  if long_pnl > 0:
    transfer = min(long_pnl, short_pool_equity)
    long_pool_equity += transfer
    short_pool_equity -= transfer
  else if short_pnl > 0:
    transfer = min(short_pnl, long_pool_equity)
    short_pool_equity += transfer
    long_pool_equity -= transfer

  last_price = P_now
```

该函数是 O(1)，只读写 Market/SubPool 账户。

## 7. 用户权益计算

用户持有方向池 shares：

```text
user_equity = pool_equity * user_shares / total_shares
```

如果某方向池权益变为 0：

- 所有该方向用户当前权益为 0。
- 仓位仍然保留 shares 和方向。
- 系统记录每个归零仓位或归零聚合桶的 `zero_price_tick`。
- 价格继续向不利方向移动时，这些 shares 处于 dormant 状态，不再计入当前可赔付亏损或盈利 claim。
- 未来价格反转且对手方池产生可转移亏损时，该方向池权益重新增加。
- 当价格回到对应 `zero_price_tick` 时，dormant shares 重新激活，并按 claim 权重比例分享可分配利润。

这天然满足“不设置仓位死亡状态”。

## 8. 该模型的优点

- 每次价格更新只更新两个方向池，O(1)。
- 当前区块可计算出全市场池级清算结果。
- 用户平仓只需读取自己的仓位和 Market 池状态，O(1)。
- 可以保留亿级零价值仓位，因为不需要每个区块遍历它们。
- 池权益最低为 0，不会出现负值。
- 恢复机制自然成立：池权益重新增加时，所有未关闭仓位按 shares 恢复。

## 9. 与白皮书原则的关系

该模型不是背离白皮书，而是白皮书原则在大用户量和当前区块即时提款约束下的可扩展表达。

白皮书原则：

```text
盈利方按各自账面盈利占比分享亏损池
亏损方按各自可兑现亏损占比承担
```

大规模实现：

```text
同方向用户按 pool shares 分享方向池权益变化
```

需要特别验证的公平性：

- 同方向不同开仓时间、不同入场价的仓位需要通过 shares 定价设计公平性。
- 需要定义新用户进入池时如何铸造 shares，避免稀释老用户。
- 需要定义不同杠杆是否仍然必须隔离，建议仍然隔离。
- 需要重新验证用户体验是否仍符合白皮书精神。

## 10. Shares 铸造

开仓时：

```text
shares_minted = principal * total_shares / pool_equity
```

如果池子为空：

```text
shares_minted = principal
```

如果池权益为 0 但仍存在旧 shares，不能简单使用 `shares_minted = principal` 把新用户和历史零价值仓位混在同一层。否则新用户可能用很少本金获得与大量历史零价值仓位同等的未来恢复权，稀释老仓位。

必须把 shares 拆成两类：

```text
active_shares      当前有权益支撑的 shares
recovery_shares    池权益归零后仍保留未来恢复资格的历史 shares
```

当池权益归零：

```text
recovery_shares += active_shares
active_shares = 0
pool_equity = 0
move recovery_shares into dormant bucket at zero_price_tick
```

新用户在池权益为 0 时开仓：

```text
new_active_shares = principal
pool_equity += principal
active_shares += new_active_shares
pool_notional += principal * leverage
```

新用户必须始终允许开仓。不能因为某方向池权益为 0 就暂停该方向开仓，因为 MoleOption 的核心是多空双方持续自由进入、互相提供流动性，让市场自然自调节。

未来对手方亏损注入可分配资金时，不能按开仓时间、历史亏损先后或 recovery/active 身份设置优先级。MoleOption 的公平原则是：同一市场内所有当前有盈利或恢复 claim 的仓位，按各自 claim 权重比例分享本轮可分配利润。

```text
total_claim_weight = recovery_claim_weight + active_claim_weight

recovery_pool_credit = transfer * recovery_claim_weight / total_claim_weight
active_pool_credit = transfer * active_claim_weight / total_claim_weight
```

其中 claim weight 不是开仓时间权重，而是该类仓位在当前价格变动下的账面盈利/恢复需求权重：

```text
claim_weight_i = current_profit_or_recovery_claim_i
```

含义：

- `active_shares` 与 `recovery_shares` 只是账务分层，用于防止零权益时新仓稀释旧仓恢复权。
- 它们不代表分配优先级。
- `recovery_shares` 在价格未回到其 `zero_price_tick` 前处于 dormant 状态，不参与本轮 claim。
- 历史零价值仓位、新开仓仓位、反复恢复仓位，只要在当前价格区间产生正向 claim，就按 claim 权重比例分享本轮 transfer。
- 如果没有对手方亏损 transfer，recovery shares 不会凭空恢复。

该规则保持两个目标：

- 任何时候任何方向都可以新开仓。
- 新开仓不会稀释历史零价值仓位的恢复权。
- 所有盈利/恢复方在同一市场内公平分配，没有时间先后优先级。

用户本金加入对应方向池：

```text
pool_equity += principal
pool_notional += principal * leverage
total_shares += shares_minted
```

这种方式类似 LP share，避免新用户任意稀释老用户。

开仓交易顺序：

```text
sync_pool(P_now)
mint_shares_for_new_position()
add principal/equity/notional to pool
store position shares
```

因此，新开仓用户在本交易结束后已经进入池状态；从下一个价格同步开始，该用户自然参与总体盈亏。

## 11. 平仓

平仓时：

```text
withdrawable = pool_equity * user_shares / total_shares
```

然后：

```text
pool_equity -= withdrawable
total_shares -= user_shares
position.status = Closed
```

还需要同步减少该用户对应的名义敞口：

```text
pool_notional -= position_notional
```

如果池权益为 0：

```text
withdrawable = 0
```

用户可以选择关闭拿 0，也可以继续持有 shares 等待未来恢复。

## 12. 是否仍然“永不爆仓”

是。

用户最大损失为本金：

```text
0 <= user_equity <= pool_equity_share
```

池权益不会为负，用户权益不会为负。仓位不会被系统删除。

## 13. 是否支持当前区块即时平仓

支持，前提是：

- 当前交易先用最新价格更新 Market 池状态。
- 平仓基于更新后的池状态计算用户 shares 价值。
- 同一个 Market 写账户会产生 Solana 账户锁竞争，因此高并发时需要市场分片或多个流动性子池。

注意：它支持当前交易内精确计算，但不代表同一 Market 能无限并发写入。Solana 对同一 Market 账户写锁仍是瓶颈。

也就是说，0.4 秒区块内能完成的是“已经被 Solana leader 纳入该区块的交易”的链上计算和转账。链下同时涌入的几百万或几千万请求，不可能全部进入同一个区块；超出区块容量和账户写锁吞吐的请求会排队、失败或进入后续区块。

因此高并发目标应定义为：

```text
每笔被纳入区块的开仓/平仓交易，都能在该交易内 O(1) 精确完成。
系统通过子池分片提高可并行进入区块的交易数量。
```

而不是：

```text
所有互联网用户请求都必须在同一个 0.4 秒区块内完成。
```

## 14. 高并发写锁问题

即使 O(1) 数学解决了计算量，Solana 仍然存在同一账户写锁。

解决方案：

- 同一标的/杠杆拆成多个子池。
- 用户开仓时路由到某个子池。
- 平仓只影响所在子池。
- 子池之间通过前端聚合展示。
- 大额用户可路由到深度更高的子池。

这会牺牲部分全市场共享流动性，但换来并行吞吐。

## 15. 零值锚点的 O(1)/O(log n) 实现

为了避免逐个扫描几千万个零价值仓位，归零仓位必须按价格 tick 聚合。

推荐数据结构：

```text
DormantBucket {
  direction
  zero_price_tick
  total_recovery_shares
  total_recovery_notional
}
```

当前价格移动时：

- 多头只激活 `zero_price_tick <= current_price_tick` 的 dormant buckets。
- 空头只激活 `zero_price_tick >= current_price_tick` 的 dormant buckets。

链上不能一次遍历所有 bucket，因此需要：

- 固定价格 tick 分段。
- 分层 bucket。
- Fenwick tree 或 segment tree 聚合账户。
- 或限制每个市场的 tick 范围和 bucket 数量。

目标是让“当前可参与统计的恢复权重”可以在 O(log n) 或受限 O(1) 账户读取内得到。

## 16. 最终决策

由于 MoleOption 同时要求：

1. 亿级 open 仓位。
2. 当前区块即时平仓确定金额。
3. 不删除零价值仓位。

因此正式采用“隔离市场 + 方向权益池 + shares + recovery/dormant bucket + 子池分片”的链上实时结算模型。逐仓全局比例模型保留为原则解释、离线仿真和小规模对照，不作为生产链上实时结算算法。
