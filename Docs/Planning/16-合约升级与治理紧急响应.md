# MoleOption 合约升级与治理紧急响应

## 1. 背景与目标

MoleOption 是托管亿级用户真实资金的衍生品协议。一旦主网部署，合约升级、参数调整和紧急响应都是高风险操作。本文件给出：

- 升级权限分级。
- 升级流程与时间锁分级。
- 紧急暂停与恢复路径。
- 状态版本化与迁移协议。
- 升级后用户可见性与退出权。
- 治理多签操作规范。

它不取代 `08-杠杆交易场与风控设计.md` 的运行期风控，而是补充“链上代码与权限本身”的安全治理。

## 2. 权限分级

链上至少存在以下权限角色，必须互相隔离，不允许同一密钥同时具备：

```text
ProgramUpgradeAuthority    程序字节码升级权限
GlobalConfigAuthority      全局配置变更权限
MarketAuthority            单市场参数变更权限
EmergencyAuthority         紧急暂停权限
FeeAuthority               费用账户管理权限
KeeperAuthority            非资金敏感的常规调用权限
```

主网阶段必须满足：

- `ProgramUpgradeAuthority` 必须由 `5-of-7` 或更高门槛的多签持有，签名者地理与组织分散。
- `GlobalConfigAuthority` 必须为多签，并通过时间锁执行。
- `EmergencyAuthority` 可以是较低门槛多签（如 `2-of-3`），但只能调用受限指令集合（暂停、限流、冻结新开仓），不能改资金路径。
- `FeeAuthority` 不持有 Vault 权限，只能从 `fee_vault` 提款，且 `fee_vault` 与用户 `Vault` 必须是不同账户。
- `KeeperAuthority` 是常规化身份，不持有任何资金敏感权限；任何用户或合约都可以替代 keeper 触发公开指令。

## 3. 时间锁分级

升级操作按风险等级使用不同时间锁：

```text
Level 0: 紧急暂停 / 限流              立即生效  (EmergencyAuthority)
Level 1: 风险参数调整 (oracle 阈值、限额)  >=  6 小时
Level 2: 一般参数调整 (费用、tick 粒度)    >= 24 小时
Level 3: 新增或下线市场                 >= 48 小时
Level 4: 程序字节码升级                 >= 72 小时
Level 5: 状态结构迁移 / 紧急资金路径修改   >= 7 天 + 全市场暂停
```

Level 0 必须满足：

- 不能改变任何已有用户仓位的 shares、recovery_shares、dormant_bucket 余额。
- 不能减少 vault 余额。
- 不能修改 `pool_equity` 直接数值。
- 只能切换状态机标志位（paused、restricted、frozen_new_position）。

Level 1 至 Level 5 必须经历：

```text
propose -> queue -> wait timelock -> execute
```

并且 `propose` 与 `execute` 必须由不同多签签名集合签出，`queue` 期间任何一个 `EmergencyAuthority` 多签可以单方面 `cancel`。

## 4. 紧急暂停与恢复路径

### 4.1 自动暂停

合约必须包含自动暂停触发器：

- 预言机价格过期超过 `max_oracle_age_seconds`。
- 预言机置信区间相对价格超过 `max_confidence_bps`。
- 单次 `sync_pool` 价格变动超过 `max_price_move_bps_per_sync`。
- `vault_balance < required_pool_equity_total + recovery_dust_floor`。
- 任何关键不变量在 runtime 检测中被破坏。

自动暂停后：

- 禁止开仓。
- 禁止 `claim_closed_position`。
- 允许查询、`request_close_position` 排队、`apply_lazy_settlement`。
- 不允许 `EmergencyAuthority` 在自动暂停中直接转移用户资金。

### 4.2 人工暂停

`EmergencyAuthority` 可立即暂停以下范围之一：

- 单 Market。
- 单 SubPool。
- 全协议。

人工暂停的最大持续时间为 `max_emergency_pause_slots`（建议初值：`>=` 4 小时对应的 slot 数）。超时后必须由 `GlobalConfigAuthority` 走 Level 1 时间锁续期，否则自动解除。

### 4.3 恢复

恢复操作必须满足：

- 预言机连续 `recovery_check_window` slot 健康。
- 链上不变量自动检查通过。
- 没有正在处理的紧急升级提案。
- `EmergencyAuthority` 与 `GlobalConfigAuthority` 同时签字。

恢复后必须先解除 `claim_closed_position` 限流，再恢复开仓，避免在仍可能存在状态不一致时新增风险敞口。

## 5. 状态版本化与迁移协议

### 5.1 强制版本字段

所有可升级账户结构必须包含：

```rust
pub struct AccountVersionHeader {
    pub schema_version: u16,
    pub program_version: u16,
    pub migration_status: u8,   // 0 = stable, 1 = pending_migration, 2 = migrating
    pub reserved: [u8; 32],
}
```

`reserved` 用于未来字段扩展，无需重新分配账户。

### 5.2 迁移指令规范

任何破坏字段语义的升级必须提供 `migrate_<account>_v{old}_to_v{new}` 指令，并满足：

- 是幂等的：重复调用不会造成状态破坏。
- 是单方向的：迁移完成后 `schema_version` 单调递增。
- 是分页的：单笔交易处理有限数量账户，不允许全量遍历。
- 不改变资金所有权：用户的可提款总额在迁移前后必须严格相等（允许 dust 留在 Vault）。
- 在迁移期间，被迁移账户处于 `migration_status = migrating`，禁止开仓平仓。

### 5.3 用户退出权

任何状态结构迁移开始前，协议必须先提供至少 `>= 7 天` 的“退出窗口”：

- 在窗口期内禁止新开仓。
- 在窗口期内允许 `request_close_position` 与 `claim_closed_position`。
- 用户主动平仓使用旧合约逻辑，结果链上可验证。
- 窗口结束后才执行迁移。

如果迁移本身是为了修复严重 bug，导致无法等待 7 天，则必须：

- 多签级别提升至 `>= 6/7`。
- 公示原因。
- 仍允许 `request_close_position` 队列。
- 迁移指令需要外部审计的紧急 patch 报告。

## 6. 程序字节码升级流程

### 6.1 主网部署前

部署时即把 `ProgramUpgradeAuthority` 转给主签名者多签。**禁止** 在主网阶段保留单签 upgrade authority。如果协议希望未来变为不可升级，可以在足够审计后将 upgrade authority 设置为 `null`，固化合约。

### 6.2 升级提案

```text
propose_upgrade(new_program_buffer, schema_version, migration_plan_hash)
```

签名要求：`>= 5/7` 多签。

### 6.3 时间锁公示

提案进入 Level 4 / Level 5 时间锁。期间：

- 任何成员可签 `cancel_upgrade(proposal_id)` 单方面取消。
- 用户可以查看新 program buffer hash 和迁移计划。
- 如果 `EmergencyAuthority` 不同意，可单签 `veto`。

### 6.4 执行

到期后由任意 `ProgramUpgradeAuthority` 多签调用：

```text
execute_upgrade(proposal_id)
```

执行前合约自动检查：

- 没有进行中的开仓 / 平仓 / claim 高峰（可由内置 throttle metric 决定）。
- 多签签名仍然有效。
- 如果是 Level 5 升级，必须先全市场暂停。

执行后立即触发 `MigrationStarted` 事件，索引服务和前端进入只读保护模式。

### 6.5 事后验证

升级完成后必须：

- 链上自动调用 `verify_invariants` 指令。
- 抽样若干 Market 与 Position 计算 `withdrawable`，对比迁移前快照。
- 索引服务对账无误差。
- 任何不变量失败立即触发自动暂停，并启动回滚提案。

## 7. 回滚与紧急 patch

链上字节码无法真正“回滚”，但可以重新部署旧版本字节码达成等价效果。

### 7.1 回滚条件

- 升级后核心不变量失败。
- 用户 claim 异常超过阈值。
- 与离线 oracle 仿真结果偏差超过预设。

### 7.2 回滚流程

```text
emergency_pause_all_markets
propose_upgrade(rollback_program_buffer)
fast-track timelock with super-majority signatures
execute_upgrade
run_post_rollback_invariant_check
resume_markets_gradually
```

“快通道时间锁”最短 `>= 12 小时`，且必须有 `>= 6/7` 多签和外部独立审计签字。

## 8. 治理参数变更规范

### 8.1 限制

某些参数永远不允许通过治理修改：

- Market 的标的、抵押资产、杠杆。
- 任意 `Position` 的 shares、recovery_shares、locked_loss、realized_profit_balance。
- DormantBucket 内部 `total_recovery_shares`。
- Vault 余额。

### 8.2 只能通过指令调整的参数

- `open_fee_bps`。
- `max_oracle_age_seconds`。
- `max_confidence_bps`。
- `max_price_move_bps_per_sync`。
- 单仓上限。
- 市场总名义价值上限。
- 子池路由参数（增加子池数量但不缩减）。

### 8.3 永不允许

- 扣减用户余额。
- 改变 shares 计数或 dormant 注册。
- 任何形式的“管理员注资到 pool_equity”，因为这会改变 shares 估值，构成隐性稀释。

如果协议确实需要注资（例如保险基金兜底），必须有显式独立指令，且：

- 标记事件 `InsuranceFundInjection`。
- 仅在协议侧自有保险基金账户与 Vault 之间转移。
- 必须经过独立提案与时间锁。
- 公开记账，不进入 `pool_equity`，独立账户结算。

## 9. 子池数量调整

子池只能增加，不能减少。原因：

- 减少子池意味着合并 pool_equity 与 shares，不可避免造成跨子池估值偏差，破坏每个子池内部公平。
- 即使长尾子池低活跃，也不应强制并入其他子池，因为 dormant bucket 与 recovery shares 的归属要保持稳定。

如果某子池长期不活跃，可以采取：

- 路由策略减少新仓位流入，但不强制并池。
- 鼓励用户主动迁移：通过前端引导、奖励或新仓优惠（不能改变已存在仓位）。

## 10. 治理透明度

所有治理操作都必须 emit 事件并在公开仓库中追踪：

- `UpgradeProposed`
- `UpgradeCancelled`
- `UpgradeExecuted`
- `MigrationStarted`
- `MigrationCompleted`
- `EmergencyPaused`
- `MarketParamUpdated`
- `RollbackExecuted`

公开仓库要求：

- 每个提案有公示页面。
- 公示页面包含 program buffer hash、迁移计划、影响评估、签名状态。
- 用户可以订阅公示更新。

## 11. 与漏洞赏金的衔接

发现严重漏洞的合规渠道：

- 公开 bug bounty 程序，明确等级、赔付上限、披露规则。
- 严重漏洞披露后协议立即触发 EmergencyPause，不依赖治理时间锁。
- 漏洞赏金支付走独立保险基金账户，不影响 Vault。

## 12. 多签操作安全要点

签名者必须遵守：

- 硬件钱包或独立签名设备。
- 不在单一云账户、单一邮箱、单一手机存储多个签名密钥。
- 关键签名行动需要 voice / video 二次确认。
- 提案验证脚本本地运行，比对 program buffer hash 与公开仓库一致。
- 拒绝任何“紧急但未公示”的提案，除非已在协议红色应急流程范围内。

## 13. 升级演练

主网上线前必须演练：

- Level 4 程序升级流程。
- Level 5 状态迁移流程。
- 紧急暂停 + 快通道升级 + 回滚。
- 多签丢失部分密钥后的恢复路径。
- 索引服务与前端在升级期间的降级模式。

演练记录保留作为审计材料。

## 14. 与产品与合规的衔接

- 前端必须显示当前合约版本与最近 7 天升级历史。
- 前端必须在治理提案时显著提示用户。
- 风险提示必须说明：合约可升级、用户存在升级期不可控风险，并指明退出窗口机制。
- 面向受监管市场用户的版本应单独披露升级权限分布与多签构成。

## 15. 结论

合约升级与治理是“代码安全”之外、同样致命的攻击面。MoleOption 必须在主网上线第一天就把 `ProgramUpgradeAuthority`、紧急暂停、参数变更、状态迁移和回滚流程全部写进合约和治理文件，并且实战演练。否则任何技术上的精确清算优势都可能被治理失误一次性抵消。
