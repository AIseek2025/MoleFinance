# MoleOption 工程规范与开发指引

## 1. 工程原则

MoleOption 是资金协议，开发优先级应为：

1. 资金安全。
2. 数学正确。
3. 可测试。
4. 可解释。
5. 用户体验。
6. 性能优化。

任何优化不得破坏前三项。

## 3. Rust 规范

### 3.1 禁止浮点数

链上和核心仿真逻辑不得使用 `f32` 或 `f64` 表示价格、金额和比例。

使用：

```text
u64  token amount
u64  scaled price
u128 notional and multiplication result
i128 signed pnl
```

### 3.2 Checked Math

所有加减乘除必须使用 checked 方法或封装函数：

```rust
checked_add
checked_sub
checked_mul
checked_div
```

禁止裸用可能溢出的算术表达式处理资金。

### 3.3 明确舍入方向

所有除法函数必须在名字中表达舍入：

```rust
mul_div_floor
mul_div_ceil
```

默认资金支付使用 floor。

### 3.4 纯函数优先

清算核心应写成纯函数，便于测试：

```rust
calculate_delta_pnl
calculate_actual_transfer
apply_settlement
compute_withdrawable
```

指令函数只负责账户校验、状态读写和 token 转账。

## 4. Anchor 规范

### 4.1 账户约束

尽量在 `#[account(...)]` 中表达：

- seeds。
- bump。
- has_one。
- owner。
- mut。
- token mint。
- token authority。

复杂业务约束在 handler 中二次校验。

### 4.2 PDA Seeds

所有 PDA seeds 必须文档化，并保持稳定。修改 seeds 等同于破坏状态兼容。

### 4.3 错误码

使用明确错误码，不要用泛化 `InvalidInput` 覆盖所有错误。

### 4.4 事件

关键资金状态变化必须 emit 事件，方便索引和审计。

## 5. 数学实现规范

### 5.1 统一缩放

建议：

```text
PRICE_SCALE = 1e8
RATE_SCALE = 1e12
BPS_SCALE = 10_000
```

### 5.2 Token Decimals

USDC 通常为 6 decimals。不要假设所有抵押资产都是 6 decimals。Market 应记录 collateral decimals 或从 mint 读取后固定。

### 5.3 PnL 单位

所有 PnL 最终应转换为抵押资产最小单位。

### 5.4 Dust 处理

由于 floor 舍入产生的 dust 留在 Market Vault。后续可通过治理迁移到保险基金，但首版不应复杂化 dust 提取逻辑。

## 6. 测试规范

### 6.1 单元测试

每个数学函数必须覆盖：

- 正常上涨。
- 正常下跌。
- 价格不变。
- 多头盈利。
- 多头亏损。
- 空头盈利。
- 空头亏损。
- 亏损超过剩余本金。
- 盈利超过亏损池。
- 亏损池超过盈利。

### 6.2 属性测试

随机生成：

- 价格路径。
- 仓位方向。
- 本金。
- 杠杆。
- 开平仓时刻。

长期验证：

```text
locked_loss <= principal
locked_loss is monotonic
realized_profit_balance increases only from matched counterparty losses
realized_profit_balance may decrease only when the same open position later becomes a loser
sum_paid_profit <= sum_charged_loss
withdrawable <= vault_balance
```

### 6.3 集成测试

必须测试：

- 初始化配置。
- 初始化市场。
- 开仓。
- 多用户多空开仓。
- 重置。
- 平仓。
- 暂停和恢复。
- 预言机错误。
- 非法权限。

### 6.4 高并发清算测试

必须测试：

- 分片顺序变化不影响最终结果。
- epoch 未最终化时不能提现确定性权益。
- 仓位跨多个 epoch 后懒结算结果等价于逐 epoch 即时结算。
- 大规模随机仓位下，单个指令账户数和计算单元保持在 Solana 限制内。

## 7. 前端规范

### 7.1 风险展示

交易界面必须展示：

- 最大亏损。
- 参考盈亏。
- 当前权益。
- 盈利兑现风险。
- 预言机更新时间。

### 7.2 禁止误导

不得把 `display_pnl` 直接命名为“可赚金额”或“可提取盈利”。

推荐命名：

- 参考盈亏。
- 估算当前权益。
- 可提取金额。

### 7.3 交易确认

高杠杆或低流动性市场中，确认弹窗必须提示：

```text
盈利可能无法全部兑现；最大亏损为本金和费用。
```

## 10. 文档维护规范

任何影响以下内容的代码变更必须同步更新文档：

- 数学公式。
- 账户字段。
- 指令接口。
- 市场参数。
- 风险提示。
- 测试假设。

## 11. 安全开发清单

提交前检查：

- 是否有 unchecked math。
- 是否有未校验 signer。
- 是否有未绑定 market 的 position。
- 是否有可重复提款路径。
- 是否有错误 mint 的 token account。
- 是否有可伪造 oracle account。
- 是否有管理员直接转走用户资金路径。
- 是否有状态先转账后更新导致的问题。

## 12. 上线前工程门槛

Devnet 前：

- 本地测试全过。
- 核心仿真脚本可复现。
- 前端能完成端到端流程。

主网前：

- 外部审计。
- 漏洞赏金计划。
- 监控和暂停机制。
- 多签权限。
- 市场限额。
- 回滚和公告流程。
