# MoleOption 核心机制与数学模型

## 1. 模型设计目标

MoleOption 的数学模型需要同时满足：

- 用户最大亏损不超过本金。
- 协议不产生无资金来源的盈利支付。
- 盈利方收益来自亏损方可兑现损失。
- 仓位清算过程公开、确定、可复现。
- 可以在 Solana 上以有限计算成本实现。

## 2. 关键实体

### 2.1 Market

一个 Market 是独立清算场，定义为：

```text
Market = 标的资产 + 抵押资产 + 杠杆倍数
```

例如：

```text
BTC/USD + USDC + 10x
```

每个 Market 独立拥有资金池、重置价格和仓位集合。

### 2.2 Position

每个仓位包含：

```text
principal        初始本金
locked_loss      已锁定损失
entry_price      开仓价格
last_settle_price 最近结算价格或市场 reset price
leverage         杠杆倍数
direction        +1 多头，-1 空头
notional         principal * leverage
status           open / closed
```

### 2.3 Vault

Vault 是 Market 的真实资金池。所有本金进入 Vault，所有平仓提款从 Vault 支出。

## 3. 基础变量

```text
P_0      仓位开仓价格
P_r      最近市场重置价格
P_now    当前预言机价格
L        杠杆倍数
M        本金 principal
D        方向，long = +1，short = -1
N        名义价值，N = M * L
LL       已锁定损失 locked_loss
RM       剩余保证金，RM = M - LL
```

约束：

```text
M > 0
L > 0
0 <= LL <= M
0 <= RM <= M
```

## 4. 双层盈亏体系

### 4.1 参考盈亏

参考盈亏用于用户理解，类似永续合约：

```text
display_pnl = D * N * (P_now - P_0) / P_0
```

特点：

- 面向 UI。
- 不等于可提款金额。
- 可以小于 `-principal`。
- 可以大于当前市场可支付能力。

### 4.2 结算增量盈亏

清算层应基于最近重置价格计算本轮增量：

```text
delta_pnl = D * N * (P_now - P_r) / P_r
```

盈利增量：

```text
profit_increment = max(0, delta_pnl)
```

亏损增量：

```text
loss_increment = max(0, -delta_pnl)
```

可兑现亏损增量：

```text
realizable_loss_increment = min(loss_increment, RM)
```

## 5. 全局配对清算

一次重置中，市场内所有仓位分成盈利方和亏损方。

```text
TOTAL_PROFIT = sum(profit_increment_i)
TOTAL_REALIZABLE_LOSS = sum(realizable_loss_increment_i)
ACTUAL_TRANSFER = min(TOTAL_PROFIT, TOTAL_REALIZABLE_LOSS)
```

`ACTUAL_TRANSFER` 是本轮真正发生所有权转移的资金总额。它同时受两个上限约束：

- 不能超过盈利方的账面盈利总额，否则会向盈利方支付超过其应得盈利的金额。
- 不能超过亏损方的可兑现亏损总额，否则会让亏损方损失超过其当前可承受亏损。

### 5.1 两人最小对手结算

在只有一个亏损方 A 和一个盈利方 B 的情况下，清算退化为最小值规则：

```text
realized_profit_B = min(profit_increment_B, realizable_loss_A)
realized_loss_A = min(profit_increment_B, realizable_loss_A)
```

这意味着：

- 如果 A 账面亏损很大，而 B 账面盈利较小，B 只能拿走自己的盈利金额，A 不会因为 B 平仓而失去全部账面亏损。
- 如果 A 账面亏损较小，而 B 账面盈利很大，B 只能拿到 A 当前可兑现亏损的金额，B 的实际收益小于账面盈利。
- A 增加的 `locked_loss` 等于 B 实际拿到的金额；这部分资金已经支付给 B，因此不可逆。

### 5.2 多人比例清算

当盈利方和亏损方数量大于 2 时，协议先计算全局可转移金额：

```text
ACTUAL_TRANSFER = min(sum(profit_increment_i), sum(realizable_loss_j))
```

然后按两侧各自权重分配：

```text
profit_weight_i = profit_increment_i / TOTAL_PROFIT
loss_weight_j = realizable_loss_increment_j / TOTAL_REALIZABLE_LOSS

realized_profit_i = ACTUAL_TRANSFER * profit_weight_i
realized_loss_j = ACTUAL_TRANSFER * loss_weight_j
```

因此，所有账面盈利仓位不是按人数平均分配，而是按各自账面盈利金额占总账面盈利的比例分享本轮可实际锁定亏损金额。所有账面亏损仓位也不是平均承担，而是按各自可兑现亏损金额占总可兑现亏损的比例承担本轮实际转移金额。

盈利兑现率：

```text
profit_realization_rate =
  if TOTAL_PROFIT == 0:
    0
  else:
    ACTUAL_TRANSFER / TOTAL_PROFIT
```

亏损承担率：

```text
loss_charge_rate =
  if TOTAL_REALIZABLE_LOSS == 0:
    0
  else:
    ACTUAL_TRANSFER / TOTAL_REALIZABLE_LOSS
```

盈利方实际收益：

```text
realized_profit_i = profit_increment_i * profit_realization_rate
```

亏损方实际权益扣减：

```text
realized_loss_i = realizable_loss_increment_i * loss_charge_rate

profit_credit_spent_i = min(realized_profit_balance_i, realized_loss_i)
locked_loss_increase_i = realized_loss_i - profit_credit_spent_i

realized_profit_balance_i_new = realized_profit_balance_i_old - profit_credit_spent_i
locked_loss_i_new = locked_loss_i_old + locked_loss_increase_i
```

说明：如果一个仓位此前通过价格反转获得了 `realized_profit_balance`，但尚未平仓提取，那么后续再次成为亏损方时，应先扣减这部分未提取盈利余额；只有超出未提取盈利余额的亏损，才继续增加不可逆 `locked_loss`。否则零权益仓位恢复后会获得不承担后续亏损的免费期权。

## 6. 资金守恒

每一轮结算必须满足：

```text
sum(realized_profit_i) == sum(realized_loss_i) == ACTUAL_TRANSFER
```

允许存在舍入误差时，应满足：

```text
sum(realized_profit_i) <= sum(realized_loss_i)
```

舍入残差应留在 Vault 或协议 dust bucket，不得造成超额支付。

## 7. 仓位权益

对于单个仓位，链上需要维护或可推导其当前可提取权益：

```text
equity = principal - locked_loss + realized_profit_balance
```

其中：

- `locked_loss` 表示该仓位历史上已经实际支付给其他盈利平仓者的本金损失，单调递增，不可冲回。
- `realized_profit_balance` 表示该仓位在后续重置中作为盈利方，从其他亏损仓位处获得、但尚未平仓提取的已实现收益。它可增加，也可在仓位后续再次成为亏损方时被先行扣减。
- `principal - locked_loss` 是仓位尚未支付出去的原始本金余额。
- `realized_profit_balance` 可以让仓位在价格反转后恢复部分或全部历史亏损，甚至进一步获得盈利。

约束：

```text
0 <= locked_loss <= principal
realized_profit_balance >= 0
equity >= 0
equity <= vault_available_balance
```

## 8. 核心恢复机制：亏损不可逆，权益可再增长

这是 MoleOption 必须在首个真实资金版本中实现的核心机制，不是后续扩展。

### 8.1 机制定义

当用户 A 处于账面亏损，用户 B 处于账面盈利并平仓时，系统需要向 B 支付可兑现盈利。该支付来自 A 等亏损方的可兑现亏损：

```text
transfer_to_B = min(B_profit_claim, total_realizable_loss_from_losers)
```

A 被分摊到的支出会增加 A 的 `locked_loss`。这部分资金已经离开 A 对应权益并支付给 B，因此不可逆：

```text
locked_loss_A_new = locked_loss_A_old + paid_by_A
```

后续如果价格反转，A 的方向性账面亏损缩小或变为账面盈利，这并不会自动减少 `locked_loss_A`。但是如果此时用户 C 因价格反转产生账面亏损，并且 A 在该轮成为盈利方或权益恢复方，则 A 可以从 C 的可兑现亏损中获得新的已实现收益：

```text
realized_profit_balance_A_new =
  realized_profit_balance_A_old + profit_paid_by_C_to_A
```

于是 A 的可提取权益增加：

```text
equity_A = principal_A - locked_loss_A + realized_profit_balance_A
```

这就是“历史亏损恢复”的准确含义：不是冲回 `locked_loss`，而是在后续价格反转和新亏损流动性出现时，通过新增 `realized_profit_balance` 恢复仓位权益。

恢复场景同样受最小值规则约束。如果 A 以及其他历史亏损仓位合计需要兑现的恢复/盈利金额很大，但新亏损方 C 等本轮只产生较小可兑现亏损，则本轮实际恢复总额仍然被新亏损方可兑现亏损封顶：

```text
recovery_transfer =
  min(total_profit_or_recovery_claim, total_new_realizable_loss)
```

在两人恢复场景中：

```text
recovery_to_A = min(A_current_profit_claim, C_realizable_loss)
```

在千万级盈利方和千万级亏损方同时存在时，仍然先计算全局 `ACTUAL_TRANSFER`，再让所有盈利/恢复方按各自账面盈利或恢复 claim 占比分享，让所有新亏损方按各自可兑现亏损占比承担。

恢复分配不得按开仓时间、亏损发生时间或恢复身份排序。历史亏损仓位和新开仓仓位只要在当前结算区间产生正向 claim，就按同一套 claim weight 公式参与分配。

### 8.2 Alice/B/C 过程抽象

1. A 做多，价格下跌，A 账面亏损。
2. B 做空并平仓，B 的盈利从 A 的可兑现亏损中支付。
3. A 的 `locked_loss` 增加，该部分不可逆。
4. 价格随后上涨，A 的方向性表现改善。
5. C 做空或持有空头，因价格上涨产生账面亏损。
6. A 平仓或参与重置时，A 作为盈利方从 C 的可兑现亏损中获得支付。
7. A 的 `realized_profit_balance` 增加，可提取金额高于在 B 平仓时同步退出的金额。

### 8.3 为什么不是免费期权

A 的恢复资金不是协议凭空支付，也不是追回 B 已经拿走的资金，而是来自 C 在后续市场路径中产生的真实可兑现亏损。

系统仍然满足：

```text
sum(new_realized_profit) <= sum(new_realizable_loss)
```

A 只是保留了原方向仓位的持续敞口。当市场反转且存在新对手盘亏损时，A 可以再次作为盈利方参与分配。

如果 A 恢复出 `realized_profit_balance` 后没有平仓，而市场再次反向运动使 A 成为亏损方，则 A 的后续亏损先扣减未提取的 `realized_profit_balance`，再增加新的 `locked_loss`。这保证了 A 保留的是持续风险敞口，而不是只享受恢复、不承担后续亏损的单向权利。

### 8.4 仓位耗尽后的处理

即使：

```text
locked_loss == principal
```

仓位也不应在首个真实资金版本中失去方向性经济权利。它的当前基础权益为 0，但它仍保留：

- 方向。
- 杠杆。
- 名义价值。
- 开仓价格或结算参考。
- 后续参与盈利分配的资格。

此时：

```text
equity = realized_profit_balance
```

如果后续价格反转且有新的对手盘亏损，该仓位可以重新获得可提取权益。

### 8.5 关键边界

恢复机制必须满足：

- 不减少历史 `locked_loss`。
- 不向没有新亏损来源的仓位支付收益。
- 不允许仓位最大净亏损超过本金。
- 不允许盈利分配超过本轮亏损方可兑现损失。
- 不因仓位当前权益为 0 而删除其方向性敞口。

## 9. 零值锚点与休眠机制

当某仓位历史累计已锁定亏损达到 100% 本金时：

```text
locked_loss == principal
equity == 0
```

协议不关闭、不删除、不判定该仓位死亡，而是记录该仓位首次归零时的价格：

```text
zero_price_i = P_now
```

### 9.1 休眠规则

如果价格继续向该仓位不利方向移动，则该仓位进入休眠统计状态：

- 多头：`P_now < zero_price_i` 时休眠。
- 空头：`P_now > zero_price_i` 时休眠。

休眠仓位仍然存在，但：

- 当前权益记为 0。
- 不再提供亏损赔付。
- 不计入当前可兑现亏损统计。
- 不计入当前盈利 claim 统计。
- 不因为价格进一步不利移动而产生负权益。

### 9.2 重新激活规则

当价格回到该仓位归零时的锚点价格：

- 多头：`P_now >= zero_price_i`。
- 空头：`P_now <= zero_price_i`。

该仓位重新进入统计范围。重新进入后，它按协议定义的仓位权重参与当前市场的可分配利润，而不是按开仓时间优先恢复。

这可以理解为：

```text
if adverse_side_of_zero_price:
  active_weight_i = 0
else:
  active_weight_i = position_weight_i
```

### 9.3 与公平分配的关系

零值锚点机制不改变公平原则：

- 休眠仓位在不利区间不消耗系统计算和亏损赔付能力。
- 价格回到 `zero_price` 后，仓位恢复参与资格。
- 恢复后与其他同市场同方向有 claim 的仓位按权重比例分享可分配利润。
- 没有时间先后优先级。

### 9.4 亿级仓位实现要求

如果逐个检查休眠仓位是否恢复，亿级仓位不可行。生产实现必须按 `zero_price_tick` 聚合：

```text
DormantBucket(direction, zero_price_tick):
  total_recovery_weight
  total_notional
  position_count
```

池级计算时通过前缀/后缀聚合得到当前价格下已经重新激活的休眠权重：

- 多头激活权重：`sum(bucket.weight where zero_price_tick <= P_now_tick)`。
- 空头激活权重：`sum(bucket.weight where zero_price_tick >= P_now_tick)`。

该聚合必须在链上用可验证的数据结构实现，例如分层价格桶、Fenwick tree、segment tree 或固定 tick 分段账户，不能依赖链下索引决定资金结果。

## 10. 杠杆隔离数学意义

不同杠杆不应共享池子。原因：

```text
notional = principal * leverage
```

同样本金下，100x 仓位的盈亏增量是 10x 仓位的 10 倍。如果混池：

- 高杠杆盈利会抽取低杠杆亏损方资金。
- 低杠杆用户承担了未选择的高波动风险。
- 高杠杆成为理性占优策略，导致杠杆内卷。

因此：

```text
Market(symbol, collateral, leverage)
```

必须是清算隔离边界。

## 11. 费用模型

首版只建议收取开仓费：

```text
open_fee = principal * fee_bps / 10_000
net_principal = principal - open_fee
```

为了用户最大亏损清晰，建议：

- 用户输入金额为总支付金额。
- 手续费单独展示。
- 仓位本金使用扣费后的 `net_principal`。
- 最大亏损为 `net_principal`，总资产流出最大为 `principal`。

也可以定义最大亏损包含手续费，但必须在 UI 中明确。

## 12. 价格与精度

链上不得使用浮点数。

推荐：

```text
PRICE_SCALE = 1e8
AMOUNT_SCALE = token decimals
RATE_SCALE = 1e12
```

计算：

```text
delta_pnl = direction * notional * (P_now - P_r) / P_r
```

应使用 `i128` 或更高安全范围进行中间计算，并使用 checked math。

## 13. 舍入原则

资金安全优先：

- 支付给用户时向下取整。
- 扣减用户损失时也应与支付保持一致，避免凭空创造 dust。
- 汇总残差留在 Vault。
- 不允许因为舍入导致 `locked_loss > principal`。

## 14. 核心不变量

必须在测试中长期维护：

```text
For every position:
  0 <= locked_loss <= principal
  realized_profit_balance >= 0
  withdrawable >= 0
  if dormant:
    withdrawable == 0

For every reset:
  actual_transfer <= total_profit
  actual_transfer <= total_realizable_loss
  sum_paid_profit <= sum_charged_loss

For every market:
  total_withdrawable <= vault_balance
```

## 15. 推荐结论

首个真实资金版本必须实现“锁定亏损不可逆，但仓位可通过后续对手盘亏损恢复权益”的核心模型。工程上不应把 `locked_loss` 设计为可减少字段，而应通过 `realized_profit_balance` 或等价的已实现收益账户表达恢复。这样既保持历史资金流不可逆，又保留 MoleOption 产品最重要的价格反转恢复能力。
