# MoleOption 清算引擎设计

## 1. 清算引擎定位

MoleOption 没有传统强平引擎。清算引擎的职责不是删除亏损仓位，而是在价格变化后把同一市场内的盈利增量和亏损增量进行配对，更新仓位权益。

核心职责：

- 读取有效预言机价格。
- 计算本轮多空盈亏增量。
- 限制亏损方最大损失为剩余本金。
- 限制盈利方收益不超过亏损方可兑现损失。
- 更新市场重置价格和仓位状态。

## 2. 触发方式

### 2.1 用户操作触发

以下操作前必须触发市场重置或等价结算：

- 开仓。
- 平仓。
- 增加保证金，如果未来支持。
- 减少保证金，如果未来支持。

### 2.2 Keeper 触发

Keeper 可在以下条件下调用 `reset_market`：

- 固定时间间隔。
- 价格相对上次重置超过阈值。
- 市场长时间未重置。
- 风控系统要求同步状态。

### 2.3 手动触发

任何用户都可以触发公开重置，避免 keeper 中心化。

## 3. 离线参考清算算法

直接遍历所有仓位的算法只能作为离线参考实现、仿真基准和测试 oracle。真实资金智能合约不得采用“先直接遍历、后续再重构”的路径，因为合约上线后迁移成本和风险极高。

输入：

```text
market
positions[]
P_now
```

输出：

```text
updated positions
updated market.reset_price
events
```

步骤：

1. 校验市场未暂停。
2. 读取并校验 Pyth 价格。
3. 对每个 open 仓位计算 `delta_pnl`。
4. 聚合 `TOTAL_PROFIT` 和 `TOTAL_REALIZABLE_LOSS`。
5. 计算 `ACTUAL_TRANSFER`。
6. 计算盈利兑现率和亏损承担率。两人场景等价于 `min(亏损方可兑现亏损, 盈利方账面盈利)`；多人场景中，盈利方按各自账面盈利占比分享 `ACTUAL_TRANSFER`，亏损方按各自可兑现亏损占比承担 `ACTUAL_TRANSFER`。
7. 更新盈利方 `realized_profit_balance`。
8. 更新亏损方 `locked_loss`。
9. 若 `locked_loss == principal`，仓位当前基础权益为 0，但仍保持 open 并继续参与后续方向性盈利分配。
10. 更新 `market.reset_price = P_now`。
11. 发出 `MarketReset` 事件。

## 4. 伪代码

```rust
fn reset_market(market: &mut Market, positions: &mut [Position], price: Price) -> Result<()> {
    require!(!market.paused, ErrorCode::MarketPaused);
    validate_price(price, market)?;

    if price.value == market.reset_price {
        return Ok(());
    }

    let mut total_profit: u128 = 0;
    let mut total_loss: u128 = 0;
    let mut deltas: Vec<PositionDelta> = Vec::new();

    for position in positions.iter() {
        if !position.is_open() {
            deltas.push(PositionDelta::zero());
            continue;
        }

        let delta = calculate_delta_pnl(position, market.reset_price, price.value)?;

        if delta >= 0 {
            let profit = delta as u128;
            total_profit = total_profit.checked_add(profit)?;
            deltas.push(PositionDelta::profit(profit));
        } else {
            let raw_loss = (-delta) as u128;
            let remaining = position.principal.checked_sub(position.locked_loss)?;
            let realizable_loss = raw_loss.min(remaining);
            total_loss = total_loss.checked_add(realizable_loss)?;
            deltas.push(PositionDelta::loss(realizable_loss));
        }
    }

    let actual_transfer = total_profit.min(total_loss);

    if actual_transfer == 0 {
        market.reset_price = price.value;
        return Ok(());
    }

    for (position, delta) in positions.iter_mut().zip(deltas.iter()) {
        match delta.kind {
            DeltaKind::Profit => {
                // Profit takers share the actually transferable loss pool
                // pro rata by their book profit, never by headcount.
                let realized = mul_div_floor(delta.amount, actual_transfer, total_profit)?;
                position.realized_profit_balance =
                    position.realized_profit_balance.checked_add(realized)?;
            }
            DeltaKind::Loss => {
                // Losers are charged pro rata by realizable loss, capped by
                // remaining margin through delta.amount.
                let charged = mul_div_floor(delta.amount, actual_transfer, total_loss)?;
                let credit_spent = position.realized_profit_balance.min(charged);
                position.realized_profit_balance =
                    position.realized_profit_balance.checked_sub(credit_spent)?;
                let locked_loss_increase = charged.checked_sub(credit_spent)?;
                position.locked_loss = position.locked_loss.checked_add(locked_loss_increase)?;
                if position.locked_loss >= position.principal {
                    position.locked_loss = position.principal;
                }
            }
            DeltaKind::None => {}
        }
    }

    market.reset_price = price.value;
    Ok(())
}
```

## 5. 生产级链上清算算法

Solana 真实资金版本必须从第一版开始采用 epoch 化分片清算：

1. `open_epoch`：冻结上一 epoch 的价格区间，开始收集分片统计。
2. `settle_shard`：每个 Shard 分批统计本分片盈利增量和可兑现亏损。
3. `finalize_epoch`：Market 汇总所有 Shard，计算全局 `ACTUAL_TRANSFER`、盈利兑现率和亏损承担率。
4. `apply_position_epochs`：仓位在平仓、刷新或下一次交互时，按已最终化 epoch 懒结算自身权益。

这样可以保持白皮书的全局比例清算语义，同时避免单笔交易遍历上亿仓位。

### 5.1 聚合状态

Market 维护：

```text
long_active_notional
short_active_notional
long_remaining_margin
short_remaining_margin
long_profit_index
short_profit_index
long_loss_index
short_loss_index
reset_price
```

### 5.2 指数化结算

当价格上涨：

- 多头为盈利侧。
- 空头为亏损侧。

当价格下跌：

- 空头为盈利侧。
- 多头为亏损侧。

重置时更新 epoch 或 shard 级指数，而不是逐个仓位写入。

仓位在平仓时根据其入场时记录的指数差异计算个人损益。

### 5.3 难点

亏损上限、未提取盈利余额和剩余本金是仓位级变量，简单方向级聚合可能无法准确处理大量接近归零或多次恢复的仓位。需要以下方案之一：

- 分桶：按剩余本金率或入场区间分桶。
- 批处理：每次处理有限仓位，允许市场处于 partial reset 状态。
- 懒结算 + 仓位 checkpoint：仓位交互时补结算，并更新聚合状态。
- 使用离线参考算法作为测试 oracle，而不是作为链上生产算法。

建议先用离线模拟器验证经济模型；链上真实资金版本直接实现可扩展清算结构。

## 6. 平仓结算

平仓流程：

1. 读取价格。
2. 执行 market reset。
3. 对仓位应用所有未结算指数或直接状态。
4. 计算可提款：

```text
withdrawable = principal - locked_loss + realized_profit_balance
```

5. 校验：

```text
withdrawable <= vault_balance
```

6. 更新仓位为 closed。
7. 从 Vault 转账给用户。

## 7. 开仓结算

开仓流程：

1. 读取价格。
2. 执行 market reset。
3. 收取用户资金。
4. 扣除手续费。
5. 创建仓位：

```text
principal = amount - fee
locked_loss = 0
realized_profit_balance = 0
entry_price = P_now
market_entry_reset_price = P_now
```

6. 更新 Market 聚合状态。

## 8. 异常处理

### 8.1 价格无效

暂停本次重置并返回错误：

- 价格过期。
- 价格小于等于 0。
- 置信区间过大。
- 价格跳变超过阈值。

### 8.2 总盈利为 0

无盈利方，不发生资金转移，仅更新重置价格。

### 8.3 总可兑现亏损为 0

盈利方没有资金来源，不发生资金转移，仅更新重置价格。

### 8.4 舍入残差

由于整数除法，实际分配可能小于 `ACTUAL_TRANSFER`。残差留在 Vault，记录为协议 dust 或市场盈余。

### 8.5 仓位基础本金耗尽

当：

```text
locked_loss == principal
```

仓位不进入终止状态，也不从清算中移除。它的基础本金余额为 0，但仍然保留方向、杠杆和名义价值。若后续价格反转使该仓位成为盈利方，且市场中有新的亏损方提供可兑现亏损，则该仓位的 `realized_profit_balance` 增加，从而恢复可提取权益。

## 9. 防攻击考虑

### 9.1 重置抢跑

如果任何开平仓都会先重置，用户不能通过在旧价格状态下插入开平仓规避结算。

### 9.2 预言机延迟套利

必须使用价格新鲜度和置信区间限制。若 Pyth update 由用户提交，需确保使用最新可用价格。

### 9.3 拆仓攻击

拆仓不应改变总权益结果。测试中必须验证：

```text
one position with M == n positions with M/n
```

在舍入误差范围内结果一致。

### 9.4 Dust 攻击

设置最小本金，避免用户创建大量极小仓位造成计算或索引压力。

## 10. 事件字段

`MarketReset` 应包含：

```text
market
old_reset_price
new_reset_price
total_profit
total_realizable_loss
actual_transfer
profit_realization_rate
loss_charge_rate
timestamp
```

`PositionClosed` 应包含：

```text
position
owner
principal
locked_loss
realized_profit
withdrawable
close_price
```

## 11. 清算引擎验收标准

- 任意价格路径下，用户亏损不超过本金。
- 任意重置中，盈利分配不超过可兑现亏损。
- 任意平仓中，提款不超过 Vault 余额。
- 相同输入状态下，重置结果确定且可复现。
- 价格异常时不会更新市场状态。

## 12. 实施建议

先开发离线直接遍历版清算引擎作为数学 oracle，并用它验证 epoch 分片清算结果。真实资金智能合约必须直接实现 epoch、Shard、CloseRequest 和仓位懒结算，不能依赖未来升级替换核心清算模型。

由于产品要求当前区块内即时确定平仓金额，生产清算引擎采用方向权益池 shares 模型，使每次价格更新和平仓变为 O(1) 池级计算。逐仓比例模型保留为离线 oracle，用于验证资金守恒和解释白皮书原则。
