## 2. 总设计师视角发现的规划漏洞与修复

下表把本轮发现的规划漏洞按风险等级分类。所有项均已在文档侧落地，下一步由实现团队在合约与运维侧落地。

### 2.1 高危（必须主网前修复）

| ID | 漏洞 | 影响 | 修复 | 落地文档 |
| --- | --- | --- | --- | --- |
| H1 | Position 字段双套账本：同时持有 `locked_loss/realized_profit_balance` 与 `shares`。 | 资金路径模糊；升级期可能错配。 | 生产 Position 仅持有 shares 类字段；旧字段仅留事件回放生成。 | 07 §4.3, 18 §7 |
| H2 | 接近零池估值时新仓 `shares_minted = principal*total_shares/pool_equity` 反向稀释老仓。 | 新仓 shares 数量级失真，老仓权重被稀释。 | 引入 `dilution_safety_bps`，触发自动归零迁移。 | 18 §5.5, 15 §5.2 |
| H3 | `shares_minted == 0` 资金黑洞。 | 用户存款被池子吞掉，没有 shares 凭证。 | 强校验 `shares_minted > 0`，否则失败 `SharesMintedTooSmall`。 | 18 §5.2, 07 §9 |
| H4 | sync_pool 路径相关性 + 长期 idle 子池被一次性收割。 | 抢跑可大额 transfer；用户被三明治。 | Keeper `max_idle_slots`、价格保护参数 `expected_price_min/max`、idle 检测。 | 08 §7.5/7.6, 18 §4.3 |
| H5 | 子池路由如果允许用户自选可被套利。 | 跨子池套利。 | PDA 化路由 `hash(market||owner)%count` 固定。 | 18 §3 |
| H6 | dormant bucket 在大跳变下可能 OOG。 | sync_pool 失败导致市场停摆。 | bucket 数量上限、最小本金、退化为 lazy activation 路径。 | 18 §6/§10.3, 15 §5.7 |
| H7 | Vault 与 fee_vault 混用风险。 | 治理失误可挪用用户资金。 | Vault 与 fee_vault 必须独立 PDA、独立 authority、互不可转。 | 07 §8.1, 15 §5.13 |
| H8 | ProgramUpgradeAuthority 可能保留单签或时间锁缺失。 | 升级权限被盗即资金全失。 | 多签 + 时间锁分级 + 状态版本化 + 演练。 | 16 全文 |
| H9 | 合规层缺失：受限地区、制裁地址未屏蔽。 | 监管追责风险，运营公司主体责任。 | 官方前端地理 + 钱包风控、ToS、风险揭示、KYC 入口策略。 | 17 全文 |
| H10 | Position PDA 使用 `Market.next_position_id` 全局递增可能被抢占 DoS。 | 用户交易频繁失败。 | PDA seeds 改为 `["position", market, owner, owner_nonce]` 由用户提交。 | 07 §8.5, 15 §5.11 |

## 3. 黑客攻击者视角发现的攻击向量与缓解

按链上 / 链下分类。攻击向量编号 `A*` 与 `15 §5` 中条目相互引用。

### 3.1 链上攻击

| ID | 攻击 | 缓解 |
| --- | --- | --- |
| A1 | 抢先 sync_pool 收割长期 idle 子池。 | Keeper `max_idle_slots`，强制周期同步；价格保护拒绝过期价格。 |
| A2 | MEV 三明治：在大额平仓前后插入开平仓。 | 用户必须传入 `expected_price_min/max`、`expected_price_max_age_slots`，违反即拒绝。 |
| A3 | Pyth confidence 操纵：在 confidence 极小窗口套利。 | 同时检查 confidence 相对（bps）与绝对（USD）阈值；多源数据并行（首版 Pyth，后续多源）。 |
| A4 | dust 累积攻击：拆仓 + 反复开平仓收割舍入。 | 全部 floor 向协议；最小本金；dust 入独立字段不可用户提取；测试 `settle(M) ≈ settle(M/n+...)`。 |
| A5 | dormant bucket 数爆炸 OOG。 | `max_dormant_bucket_count_per_direction`；同 tick 强制合并；lazy activation。 |
| A6 | 反向稀释：在池接近 0 时大额开仓。 | `dilution_safety_bps` 强制归零迁移路径。 |
| A7 | 0 shares 资金黑洞。 | 强校验 `shares_minted > 0`。 |
| A8 | PDA 抢占 DoS。 | PDA seeds 使用 owner_nonce。 |
| A9 | 假 oracle program 注入。 | 校验 `oracle_account.owner == PYTH_PROGRAM_ID`。 |
| A10 | 同钱包内部对冲刷量。 | 限制每钱包每子池仓位数。 |
| A11 | 重入风险。 | 先写状态后转账；CPI 调用必须独立审计；无回调 hook。 |
| A12 | 跨子池数据竞争。 | 子池资金由独立 sub_pool 账户隔离；vault 不变量统一对账。 |
| A13 | 通过递增 position_id 抢占。 | 同 A8。 |
| A14 | 极端跳变下池权益变负或 transfer 错位。 | `transfer = min(pnl, counterparty_equity)`；`max_price_move_bps_per_sync` 自动暂停。 |
| A15 | 关闭 0 价值仓位放弃恢复权后再恶意申诉。 | force_close 必须 `acknowledge_forfeit_recovery=true`，事件公开记录；前端二次确认。 |
| A16 | 利用治理时间锁失误。 | 多签 + 时间锁分级 + emergency veto + 演练。 |
| A17 | 升级期间状态损坏。 | 状态版本化；schema_version 校验；迁移幂等；全协议暂停期 + 用户退出窗口。 |
| A18 | 通过 fee_vault 偷取 vault。 | 两个 PDA、不同 authority、协议侧无任何指令可让 vault->fee_vault。 |

## 8. 结论

第二轮审计的核心结论：

- 主模型（方向权益池 + shares + recovery/dormant + 子池分片）在数学上自洽，工程上可在 Solana 0.4 秒区块内 O(1)/O(log N) 完成开仓与平仓。
- 真正的风险不在“能否做”，而在“能否安全做”：边界条件（H1-H10）一旦缺漏，单一漏洞就足以导致整个协议被攻击。
- 治理（16）、合规（17）、shares 实现细则（18）共同构成了主网部署的 “第三柱” —— 它们与白皮书精神和合约清算模型同等重要。
- 协议的去中心化精神不豁免任何一方的合规责任。

实施团队不应在合约里再讨论“是否采用 shares 模型”、“是否 epoch 化”、“是否引入 dormant bucket”这类已锁定问题，而应专注于：

- 让合约严格满足 18 文档中所有不变量。
- 让治理和合规体系在主网部署当天即生效。
- 让测试与演练覆盖 19 文档中给出的全部攻击场景。

主网上线之后，协议的安全护栏才真正开始接受市场的检验。
